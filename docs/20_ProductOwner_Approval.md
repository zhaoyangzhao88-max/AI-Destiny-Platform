## 4. 批准项

### 4.1 ✅ ADR-020：Approved

| 字段 | 内容 |
|------|------|
| **决策编号** | ADR-020 |
| **决策名称** | V1 PySide6 与 Vue 集成方案 |
| **审批结果** | ✅ **Approved** |
| **审批日期** | 2026-07-10 |

**审批理由：**

1. **技术路线合理** — QWebEngineView 嵌入式集成方案是 PySide6 生态中最成熟、最自然的 Web 嵌入方式，无需引入额外的 IPC 层或独立窗口管理复杂度。
2. **用户体验统一** — 单进程、单窗口架构保证了桌面 APP 的使用体验连贯性，避免独立窗口带来的焦点管理、任务栏管理等问题。
3. **未来迁移路径清晰** — Vue 模块经 QWebEngineView 嵌入仅影响 V1 桌面壳，未来 Web 扩展时 Vue 代码可直接复用（ADR-018 §8.3.4），无锁定风险。
4. **职责分离充分** — WebChannel（宿主集成：窗口事件/生命周期）与 HTTP/WS（业务通信：AI 对话/分析）的双层通信架构职责清晰，互不耦合。
5. **成本已评估** — Chromium 捆绑体积（~100MB+）和内存开销在 ADR-018 决策时已纳入考量，作为 V1 桌面 APP 的第一性体验，该代价可接受。

**附带条件：**
- V1 初期以 PySide6 原生控件起步，Vue 按需渐进引入（呼应 ADR-018 后续影响第 1 条）
- 实现阶段需关注 WebChannel 线程安全（Python 主线程与 JS 引擎的调用）

### 4.2 ✅ docs/19：Accepted Concept

| 字段 | 内容 |
|------|------|
| **文档名称** | 分层记忆架构设计（Layered Memory Architecture） |
| **审批结果** | ✅ **Accepted Concept** |
| **审批日期** | 2026-07-10 |

**审批理由：**

1. **解决了核心产品风险** — 长期对话中的 AI"遗忘"用户信息属于不可接受的产品缺陷。三层记忆模型（工作记忆/摘要记忆/核心记忆）从根本上解决了 LLM 上下文窗口的硬限制问题。
2. **架构设计完整** — 三层定义清晰、数据契约完整、读写策略完备、与 ADR-012 数据三层体系的映射关系明确。
3. **与现有架构兼容** — 与 ADR-016（单核心 Agent）、ADR-019（双通道通信）、ADR-012（数据三层体系）天然对接，无架构冲突。
4. **压缩策略可行** — 滚动压缩 + 会话级压缩 + 主题聚合 + 去重的组合策略覆盖面广，可回溯指针保留原始回溯能力。
5. **风险透明** — §9 明确标明了 4 个开放问题（压缩细节丢失、摘要冲突、阈值调优、多 Agent 演进），体现了设计诚实度。

**附带条件：**
- 设计值（70% 水位线、20 轮默认保留）为实现调优参数，Phase 3 编码过程中需根据实际模型（如 GPT-4o、Claude 等）的上下文窗口调优
- 压缩延迟需在实现阶段评估：在线压缩（对话中实时压缩）vs 离线压缩（空闲时后台压缩）的取舍
- Accepted Concept 状态允许作为 Phase 3 编码的设计依据
