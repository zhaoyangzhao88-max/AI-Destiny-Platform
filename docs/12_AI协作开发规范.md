## 第一章：项目 AI 协作原则

### 1.1 三层协作模型

本项目采取 **「产品负责人驱动 + AI 协作执行 + 文档即契约」** 的协作模型：

```
产品负责人 ──→ 定义目标、确认决策、验收结果
    │
    ▼
AI 协作层  ──→ 执行任务、生成方案、记录过程
    │
    ▼
文档即契约 ──→ 所有关键信息写进文档，不依赖口头或隐性知识
```

### 1.2 核心原则

| 原则 | 说明 | 违反示例 |
|------|------|----------|
| **文档优先** | 任何项目关键信息必须写入文档，不允许仅存在于聊天记录中 | AI 在对话中做出设计决策但未写入 ADR |
| **不假设** | 标记【待确认】的内容不得擅自填补，必须等待产品负责人确认 | AI 自行决定技术选型并开始编码 |
| **先读后写** | 开始任何任务前必须读取相关文档，理解上下文 | 未读 HANDOVER 就开始修改代码 |
| **最小修改** | 外科手术式修改，只改必须改的行 | 顺便"优化"了无关代码 |
| **完整记录** | 每次任务执行后必须记录到协作日志 | 修改了大量文件但日志只写了"修复了问题" |
| **可追溯** | 每个决策、每次修改应能从文档追溯责任人、时间和理由 | 修改文件后未在日志中记录范围和理由 |

### 1.3 禁止行为清单

- ❌ 禁止在未读取核心文档前开始工作
- ❌ 禁止直接修改已确认的 ADR
- ❌ 禁止自行填补【待确认】内容
- ❌ 禁止删除或覆盖用户数据
- ❌ 禁止自动 git 提交（必须先展示 diff 等待确认）
- ❌ 禁止在 AI 沙箱内直接调用外部 LLM API
- ❌ 禁止重写无关代码（包括"顺手优化"）

## 第二章：AI 参与角色定义

### 2.1 角色分类

在本项目中，AI 根据任务类型扮演不同角色。同一 AI 会话在不同阶段可能切换角色。

| 角色 | 职责范围 | 权限边界 | 典型场景 |
|------|----------|----------|----------|
| **文档撰写者** | 创建、补充、优化项目文档 | 可以新增 Draft 文档，不得修改已 Approved 内容 | 写 PRD、写技术文档、补充 ADR |
| **方案分析师** | 分析需求，提出多个可选方案 | 提供建议方案，标记为"建议"，不可自行为【待确认】做决定 | 技术选型对比、架构设计讨论 |
| **代码开发者** | 编写业务代码、修复 Bug | 仅修改任务范围内的代码，不做范围外修改 | 实现功能、修复缺陷 |
| **评审者** | 对代码或文档进行 Review | 只提修改建议，不直接修改目标内容 | Code Review、文档评审 |
| **测试者** | 编写和执行测试用例 | 可以创建测试文件，不修改被测源码（除测试框架配置外） | 单元测试、集成测试 |

### 2.2 角色切换规则

```
开始任务 → 确定任务类型 → 匹配角色 → 执行 → 记录 → 切换/结束
```

- 同一轮次中如需切换角色，需明确声明角色转换
- 示例：*"文档撰写阶段结束，现在切换为代码开发者角色开始实现"*

### 2.3 AI 能力边界声明

AI 在执行所有角色时需明确以下事项：

1. **信息来源标注** — 基于项目内文档 / 基于外部知识 / 基于模型训练数据
   - 此字段与 ADR-012（数据三层体系）共用确定性分级原则，用于在协作记录中标注信息的来源可信度
   - 具体字段定义见 §8.1 报告模板的「任务信息」说明
2. **确定性分级** — 事实性陈述 / 分析性推理 / 推测性建议
   - 映射自 ADR-012 的三层体系：`fact → 事实性陈述，inference → 分析性推理，pending → 推测性建议`
   - 分级目的是帮助后续 AI 快速评估历史协作信息中的可信度层次，避免将推测性建议当作确定性事实使用
3. **不确定时处理** — 不确定时需向产品负责人提问，而非自行推断

### 2.4 多 AI 协作规范引用

多 AI 并行协作场景下的身份标识、交接协议、冲突裁决机制，不在本文档定义范围内。请参考专项文档：

| 机制 | 文档 | 核心内容 |
|------|------|---------|
| **身份标识** | `docs/17_AI Agent角色与权限规范.md` 附录 C | Agent ID 注册规则、使用方式 |
| **交接协议** | `docs/17_AI Agent角色与权限规范.md` 第 10 章 | 开始任务检查表、完成报告模板、上下文快照 |
| **冲突裁决** | `docs/17_AI Agent角色与权限规范.md` 第 9 章 | 裁决层级、标准流程、紧急路径 |
| **权限模型** | `docs/17_AI Agent角色与权限规范.md` 第 8 章 | 三级权限体系、角色权限矩阵 |

> **注意：** `docs/17` 的角色体系（按治理职能划分的 5 种 Agent）与本文档第二章的角色体系（按任务类型划分的 5 种功能角色）互为补充。同一 AI 在一次会话中可同时承担两套角色，但需分别声明。

## 第三章：标准开发流程

### 3.1 通用工作流

```
                    ┌─────────────────────────┐
                    │  收到任务指令             │
                    └──────────┬──────────────┘
                               ▼
                    ┌─────────────────────────┐
                    │  ① 读取相关文档          │ ← 先读文档再动手
                    │  - 读取 CLAUDE.md §开发前   │
                    │    必读文档 的 P0 清单（全部）│
                    │  - 读取 CLAUDE.md §开发前   │
                    │    必读文档 的 P1 清单（按需）│
                    │  - 确认相关 ADR 最新状态    │
                    │  - 读取历史协作日志了解近期活动│
                    └──────────┬──────────────┘
                               ▼
                    ┌─────────────────────────┐
                    │  ② 任务分析 & Plan       │
                    │  - 确定影响范围           │
                    │  - 识别受影响的 ADR       │
                    │  - 识别风险点             │
                    └──────────┬──────────────┘
                               ▼
                    ┌─────────────────────────┐
                    │  ③ Plan 确认             │ ← 需要产品负责人同意
                    └──────────┬──────────────┘
                               ▼
                    ┌─────────────────────────┐
                    │  ④ 执行 Plan             │
                    │  - 严格按范围修改         │
                    │  - 边做边记录             │
                    │  - 不做范围外修改         │
                    └──────────┬──────────────┘
                               ▼
                    ┌─────────────────────────┐
                    │  ⑤ 自检 & 验证           │
                    │  - 验证修改正确性         │
                    │  - 检查是否有遗漏         │
                    │  - 确认未影响其他功能     │
                    └──────────┬──────────────┘
                               ▼
                    ┌─────────────────────────┐
                    │  ⑥ 输出执行报告          │ ← 必须记录到协作日志
                    └──────────┬──────────────┘
                               ▼
                    ┌─────────────────────────┐
                    │  ⑦ 等待验收              │ ← 产品负责人确认后结束
                    └─────────────────────────┘
```

> **Plan 模式适配说明：** "Plan 模式"在 Claude Code 中原生支持。使用其他 AI 工具时，应将其映射为等效的"方案规划"步骤——即正式执行前，先输出一份结构化 Plan（目标、影响分析、受影响的 ADR、修改文件列表），等待产品负责人确认后再执行。

#### ⚠️ Step 1 完成条件：预检清单（Pre-flight Checklist）

在读取上述文档后，AI 必须在 `docs/11_AI协作日志.md` 中记录预检确认，逐项勾选以下清单：

- [ ] **P0-1：HANDOVER.md** — 已读取（项目全局概览与交接规则）
- [ ] **P0-2：01_项目简介.md** — 已读取（项目背景与核心理念）
- [ ] **P0-3：02_产品定位与愿景.md** — 已读取（产品差异化与商业模式）
- [ ] **P0-4：04_ADR_产品决策记录.md** — 已读取（所有已确认产品决策）
- [ ] **P0-5：05_知命_Persona角色设定.md** — 已读取（AI 角色「知命」的完整设定）
- [ ] **03_当前开发状态.md** — 已读取（已完成/进行中/待办事项）
- [ ] **11_AI协作日志.md** — 已读取（历史 AI 协作记录，了解近期活动和遗留问题）
- [ ] **相关 ADR 状态确认** — 如任务涉及 ADR，已确认相关 ADR 的最新状态（Approved / Accepted Concept / Draft）

> **规则：** 如果协作日志中缺少本任务的预检确认记录，视为"未完成入口读取"，后续工作视为不合规。
> **例外：** 仅执行文件搜索/内容查询等不涉及项目理解的简单任务，可跳过预检清单，但应在协作日志中注明"跳过预检——任务类型为简单查询"。

### 3.2 按任务类型的细分工流

#### 3.2.1 文档任务

```
读取文档上下文 → 明确文档定位与受众 → 撰写/修改 → 交叉引用检查
    → 格式一致性检查 → 输出报告 → 记录日志
```

- 新建文档需在文档头部标注版本、日期、状态
- 修改现有文档需在日志中说明修改范围和原因
- Draft 状态文档可自由修改，Approved 状态文档仅能通过 ADR 流程修改

#### 3.2.2 开发任务

```
读取相关文档 → 理解需求边界 → 确定技术方案 → Plan 确认
    → 编码实现（严格按范围） → 自检 → 输出报告 → 记录日志
```

- 编码前必须确认 Plan 已被产品负责人接受
- 编码过程中不得偏离 Plan 范围
- 修改现有文件时先展示受影响的行范围，再动手
- 开发任务必须完成 §3.1 预检清单后才可进入 Plan 阶段；如协作日志中无预检记录，Plan 视为未完成入口读取，需退回 Step 1

#### 3.2.3 Review 任务

```
读取目标文件 → 读取相关上下文文档 → 多维度审查
    → 输出审查报告（含风险等级） → 记录日志
```

- 区分"严重问题"、"建议优化"、"风格偏好"三个等级
- 问题必须有具体行号/段落引用

### 3.3 AI 协作时的上下文获取规范

本规范适用于 AI 需要进行架构分析、代码审查、设计决策、任务规划等需要理解项目背景的场景。

#### 3.3.1 文件优先原则

当 AI 需要进行架构分析、代码审查、设计决策、任务规划时：

- **不允许**仅根据用户提供的摘要判断项目状态或做出设计决策
- **必须优先读取项目中的真实文档**，获取第一手信息
- 读取范围包括但不限于：
  - `docs/*.md` — 项目文档体系
  - `CLAUDE.md` — 项目入口规则与必读文档索引
  - `README` — 项目概述（如有）
  - 架构设计文档（如 `08_系统架构设计.md`）
  - ADR 文档（如 `04_ADR_产品决策记录.md`）
  - API 契约文档（如 `13_API接口契约设计.md`）
  - 数据模型文档（如 `14_WebSocket实时通信协议设计.md` 4.3 节）
- 读取后应在分析中**引用具体文件路径和章节**，而非笼统描述

**典型应用场景：**

| 场景 | 必须读取的文档 | 禁止的做法 |
|------|---------------|-----------|
| 架构分析 | ADR 文档、系统架构设计、API 契约、开发状态 | 仅凭用户口头描述判断架构 |
| 代码审查 | 架构设计、开发规范、测试规范 | 仅审查代码语法不对比设计约定 |
| 设计决策 | 所有相关 ADR、现有架构文档、UX 设计 | 在未知已有决策的情况下提出替代方案 |
| 任务规划 | 当前开发状态、协作日志、HANDOVER | 重复已完成的课程/任务 |

#### 3.3.2 Markdown 读取原则

当用户无法直接上传项目文件时，AI 应通过开发代理（Claude Code、OpenCode 等 AI 执行环境）执行文件读取任务，遵循以下标准流程：

1. **AI 提供执行提示词** — 明确指定需要读取的文件路径和关注点
2. **Agent 在项目目录执行读取** — 使用实际读取工具扫描相关文件
3. **Agent 输出结构化结果**：
   - 文件结构概览
   - 相关章节的完整内容或摘要
   - 当前设计状态与约束条件
   - 已存在的架构决策
   - 潜在的冲突点

**结果质量标准：**

- 输出应足够详细，使 AI 能基于真实文档做出判断
- 不应仅输出文件名列表，必须输出具体内容
- 对于长文档，优先读取目录结构和关键章节
- 标注每个信息的来源文件路径

#### 3.3.3 架构判断原则

在提出任何架构修改建议之前，必须确认以下文档以**避免重复设计、与已有决策冲突、或修改已确定的架构原则**：

- [ ] 当前 ADR 决策（`04_ADR_产品决策记录.md`）
- [ ] 当前 API 契约（`13_API接口契约设计.md`）
- [ ] 当前架构设计文档（`08_系统架构设计.md`）
- [ ] 当前开发状态（`03_当前开发状态.md`）
- [ ] 已完成任务列表（`11_AI协作日志.md`）

**禁止行为：**

- ❌ 在未读取 ADR 的情况下提出新决策
- ❌ 在未读取 API 契约的情况下设计新接口
- ❌ 在未读取开发状态的情况下分配任务优先级
- ❌ 在未读取协作日志的情况下重复已完成的课程

**建议做法：**

- ✅ 明确声明所参考的文档："基于 ADR-019（双通道通信）和 13_API 契约 3.8 节..."
- ✅ 发现已有决策与当前需求不符时，先记录质疑再讨论修改
- ✅ 新设计与已有设计的差异部分需显式说明

#### 3.3.4 AI 交接信息标准（强制标准）

当 AI 执行文件读取和分析任务后，返回结果中必须包含以下结构化信息。以下 5 项信息为强制要求，不得遗漏：

| # | 信息项 | 说明 |
|---|--------|------|
| 1 | **修改文件列表** | 本次任务创建/修改了哪些文件 |
| 2 | **修改原因** | 每项修改的必要性与背景 |
| 3 | **影响范围** | 修改对现有架构、文档、代码的影响分析 |
| 4 | **一致性检查结果** | 修改是否与 ADR / API 契约 / 架构设计一致 |
| 5 | **下一步建议** | 基于当前完成状态，建议后续优先级 |

### 标准交接输出模板

AI 在返回文件读取和分析结果时，必须按以下模板输出：

```
修改文件列表：
- docs/xxx.md（修改：x.x节 xxx）

修改原因：
- xxx

影响范围：
- xxx

一致性检查：
- ✅/❌ 与 ADR-xxx 一致/不一致

下一步建议：
- xxx
```

> 此模板为最小强制格式，AI 可根据任务复杂度增加补充说明，但以上 5 项信息缺一不可。

> **合规说明：**
> - 上述 5 项信息为 **强制要求**，每次 AI 执行涉及文件读取、分析、设计决策或代码修改的任务后，必须在返回结果中包含此结构化信息
> - 如果任务仅为简单文件查询（如搜索关键词、统计行数），可在输出中注明"任务类型为简单查询"并选择性提交结构化信息
> - 在 `docs/11_AI协作日志.md` 中记录的每次任务条目，应至少包含"修改文件列表"和"修改原因"两项，其余项按任务复杂度决定
> - 审计时如发现交接记录缺少上述信息，将在审计报告中标注为合规问题

## 第四章：阶段性审计机制

### 4.1 审计触发条件

以下情况必须执行阶段审计：

- **Phase 阶段结束** — 一个完整的开发阶段完成时
- **架构设计完成** — 系统架构方案初步定稿时
- **API 契约完成** — 接口契约设计完成后
- **数据模型完成** — 数据库 Schema 或核心数据模型完成后
- **技术选型完成** — 关键技术方案选型确认后
- **进入代码开发前** — 从设计阶段进入编码阶段之前
- **重大架构调整后** — 对已确认的架构方向做出修改后
- **定期例行审计** — 每完成 20 条协作日志条目或每 4 周执行一次轻量审计（两个条件满足其一即触发）

### 4.2 审计目标

审计不是为了修改代码，而是验证以下维度：

| 维度 | 验证内容 |
|------|----------|
| **架构一致性** | 当前设计与已确认的 ADR 是否一致 |
| **文档一致性** | 各文档之间的引用和决策是否相互矛盾 |
| **ADR 与实现方向一致性** | 实现是否遵循了 ADR 确定的方向 |
| **API 契约完整性** | 接口设计是否覆盖了所有必要的交互场景 |
| **数据模型合理性** | 数据实体和关系是否足够支持业务需求 |
| **AI 协作流程完整性** | 协作日志是否完整、规范是否得到遵守 |
| **阶段进入条件** | 是否具备进入下一阶段的条件，存在哪些阻止因素 |

### 4.3 审计原则

#### 4.3.1 文件优先原则

审计必须基于真实项目文件。审计者在执行审计时：

- **必须**读取相关的 Markdown 文档、ADR、API 契约、架构设计等真实文件
- **禁止**仅根据聊天记录或对话上下文判断项目状态
- **禁止**仅根据用户提供的摘要或口头描述做出架构判断
- **禁止**在未读取相关 Markdown 文件的情况下做任何架构或设计层面的修改

#### 4.3.2 只审计，不随意修改

在审计阶段，审计者的职责是**发现和报告问题**，而非直接修复。

**禁止行为：**

- ❌ 未经项目负责人确认修改 ADR
- ❌ 未经确认改变已确认的架构方向
- ❌ 自动重构核心设计或 API 契约
- ❌ 在审计报告中混入代码修改

**发现问题后必须按以下流程处理：**

```
发现问题 → 分类（架构/文档/API/数据模型/流程）
        → 分级（P0/P1/P2）
        → 提出修复建议
        → 在审计报告中记录
        → 等待项目负责人确认后再执行修复
```

### 4.4 审计等级

审计发现的问题按以下三级分类：

#### P0 — 阻止级 🔴

必须解决，否则禁止进入下一阶段。

典型场景：

- 架构冲突 — 两个或多个 ADR 之间的决策相互矛盾
- API 契约冲突 — HTTP 接口与 WebSocket 协议的数据模型不一致
- 数据模型无法实现 — 当前设计在技术上有不可行的部分
- 核心技术路线未确定 — 阻碍编码开始的关键决策尚未做出
- 跨文档 P0 级矛盾 — 字段名、路径、数据格式等实现层面的直接冲突

#### P1 — 优先级 🟡

建议解决后进入下一阶段，但紧急度低于 P0。

典型场景：

- 文档遗漏 — ADR 已确定但相关设计文档未更新
- 设计细节不足 — 方案方向正确但缺少实现所需的详细规格
- 扩展性问题 — 当前设计对预期的未来扩展缺乏考虑
- 边界情况未覆盖 — 异常流程、错误处理等非主路径的设计缺失

#### P2 — 优化级 🟢

可以后续优化，不影响阶段进入。

典型场景：

- 命名优化 — 接口名、字段名、路径名的可读性改进
- 示例完善 — API 文档的 JSON 示例补充
- 文档格式调整 — Markdown 排版、表格对齐、交叉引用补充
- 注释和说明补充 — 不影响理解但可提升可维护性的补充

### 4.5 审计报告格式

每次阶段审计必须生成独立的 Markdown 审计报告，文件命名规则为：

```
docs/<编号>_Phase<X>架构审计报告.md
```

例如：

- `docs/15_Phase2架构审计报告.md`

报告至少包含以下 10 个章节：

| # | 章节 | 内容说明 |
|---|------|----------|
| 1 | 审计目的 | 本次审计的背景、触发原因和目标 |
| 2 | 审计范围 | 审计覆盖的文档、模块、决策范围 |
| 3 | 当前项目状态 | 审计时的项目阶段、文档状态、ADR 列表 |
| 4 | 架构一致性分析 | ADR 与架构设计的交叉核对结果 |
| 5 | API 与通信协议分析 | HTTP 和 WebSocket 接口的一致性审查 |
| 6 | 数据模型准备度 | 数据实体覆盖度、Schema 设计就绪度 |
| 7 | 风险清单 | 按 P0/P1/P2 分级列出所有发现的问题 |
| 8 | 开发准备度评分 | 多维度评分（总分 100 分）及评分说明 |
| 9 | 下一阶段进入建议 | YES / NO / Conditional 判断及前置条件 |
| 10 | 附录 | 审计输入文档列表、方法说明、术语对照 |

> **审计问题跟踪：** 审计发现的所有问题统一登记到 `docs/18_架构风险跟踪表.md`，按 ARC-ID 跟踪其完整生命周期（Open → In Progress → Resolved → Verified → Closed）。审计报告本身不维护问题状态，状态以 `docs/18` 为准。

### 4.6 阶段门规则

进入项目下一阶段必须经过以下正式流程：

```
阶段完成 → 执行审计 → 风险评估 → 输出报告 → 项目负责人确认 → 进入下一阶段
```

**核心规则：**

- **审计是必需环节**，不可跳过
- **风险评估结果必须有正式文档**（审计报告）
- **项目负责人确认是最终决策者**

**P0 问题阻断规则：**

> 如果审计报告中存在 **P0（阻止级）问题**，则 **禁止进入下一阶段**。
>
> 必须先解决所有 P0 问题，通过二次审计确认后，方可进入。

**Conditional 进入规则：**

> 如果审计结果为 **Conditional（有条件进入）**，项目负责人需确认前置条件清单。
>
> 前置条件可以分阶段完成，但每个条件的完成状态需在协作日志中记录。
>
> 所有前置条件完成后，执行一次快速确认审计，确认无误后进入下一阶段。

**阶段门记录：**

每次阶段门审查需在 `docs/11_AI协作日志.md` 中记录以下信息：

- 审计结论（总分 + 推荐意见）
- P0 问题数量及简述
- 前置条件清单（如有）
- 项目负责人确认时间

### 4.7 复审流程

#### 4.7.1 目的

复审是在风险/问题修复后，通过审计确认修复正确性、完整性的标准化流程。它是"发现问题 → 修复 → 确认有效"闭环的最后一步，确保每个修复都经过验证方可关闭。

#### 4.7.2 复审等级

按风险等级定义不同的复审方式：

| 等级 | 复审方式 | 执行者 | 执行时机 | 深度要求 |
|------|---------|--------|---------|---------|
| **🔴 P0** | 独立快速确认审计 | Audit Agent（不可与修复者为同一 Agent） | 修复完成后 24h 内 | 完整复审修复方案 + 修复结果；逐条验证关闭条件 |
| **🟡 P1** | 纳入下次阶段审计 / 产品负责人直接确认 | Audit Agent 或 产品负责人 | 下次阶段审计时，或产品负责人指定时间 | 确认修复满足关闭条件即可；无需完整二次审计 |
| **🟢 P2** | 由下次定期审计验证 | Audit Agent | 下次例行审计或阶段审计时 | 简略确认状态更新；可在审计报告中合并验证 |

#### 4.7.3 P0 独立快速确认审计

P0 问题修复后必须执行独立快速确认审计，流程如下：

```
P0 修复完成（Resolved）
    │
    ▼
通知 Audit Agent（不可为原修复者）
    │
    ▼
执行快速确认审计：
  ① 读取修复方案 → 确认修复逻辑正确
  ② 读取修改文件 → 确认修改完整（无遗漏/无过度修改）
  ③ 逐条验证关闭条件 → 确认每条条件均已满足
  ④ 回归检查 → 确认修复未引入新问题
    │
    ▼
输出复审记录 → 更新跟踪表状态（Resolved → Verified）
    │
    ▼
产品负责人最终确认 → 关闭（Verified → Closed）
```

**核心要求：**
- **独立性：** 执行复审的 Audit Agent 不得与执行修复的 Agent 为同一 AI
- **完整性：** 覆盖全部关闭条件，不遗漏
- **记录：** 复审结果需在 `11_AI协作日志.md` 中记录

#### 4.7.4 P1/P2 复审

- **P1：** 已修复的 P1 问题由产品负责人直接确认，或在下次阶段审计中批量验证
- **P2：** 已修复的 P2 问题纳入下次定期审计或阶段审计的范围，由 Audit Agent 统一验证
- 产品负责人可在任何时候要求对特定 P1/P2 问题执行独立复审

#### 4.7.5 复审记录格式

每次复审后，需在 `11_AI协作日志.md` 中追加以下记录：

```
--- 复审记录 ---
风险 ID:       ARC-XX-YYY-NNN
风险标题:      [原问题描述摘要]
原修复方案:    [简述]
复审方式:      [独立快速确认审计 / 阶段审计 / 产品负责人确认]
复审结论:      [通过 / 有条件通过 / 未通过]
复审日期:      YYYY-MM-DD
执行复审:      [Audit Agent / 产品负责人]
未通过原因:    [如未通过，说明]
```

#### 4.7.6 与风险跟踪表的关系

复审流程与 `18_架构风险跟踪表.md` 的风险状态流转直接对应：

```
风险跟踪表状态      复审流程对应
────────────────────────────────────
Resolved  ──→  修复完成，等待复审
    │
    ▼
复审执行   ──→  4.7.2 按等级执行对应复审
    │
    ├── 通过 → Verified（跟踪表同步更新）
    │
    └── 未通过 → 退回 In Progress（跟踪表同步更新 + 记录原因）
    │
    ▼
产品负责人确认 → Closed（跟踪表同步更新 + 11_日志记录）
```

#### 4.7.7 复审触发条件

复审在以下条件下自动触发：

- **P0 问题：** 修复后自动触发，Audit Agent 应在 24h 内响应
- **P1 问题：** 在下次阶段审计时自动纳入复审范围
- **P2 问题：** 在下次定期审计时统一验证
- **产品负责人指定：** 产品负责人可随时指定某问题需要立即复审

> **超时处理：** 若 P0 问题修复后在 48h 内未触发复审，Audit Agent 应在协作日志中备注 `【复审超时】ARC-XXX 修复后 48h 未复审`，并主动安排复审。

#### 4.7.8 定期例行审计范围与记录

定期例行审计（§4.1 第 8 项触发条件）与阶段审计（事件驱动）定位不同，以"轻量检查"为主：

| 维度 | 阶段审计（事件驱动） | 定期例行审计（时间驱动） |
|------|-------------------|----------------------|
| **审计深度** | 完整审计 | 轻量审计 |
| **审计范围** | 全部 7 维 | 3 维核心检查 |
| **报告要求** | 10 章节完整报告 | 简略报告（追加至协作日志） |
| **执行时长** | 2-4 小时 | 30-60 分钟 |
| **审查重点** | 架构一致性、完整性、准备度 | 一致性、规范遵守、风险更新 |

轻量审计每次聚焦以下 3 个核心维度：

| # | 维度 | 检查内容 | 检查方式 |
|---|------|---------|---------|
| 1 | **风险跟踪状态** | 确认 `docs/18_架构风险跟踪表.md` 是否更新，Open/In Progress 风险是否有合理进展 | 读取跟踪表 Dashboard |
| 2 | **协作日志质量** | 检查新增协作日志是否完整（§8.1 模板）、AI 类型/角色是否标注、修改清单是否清晰 | 读取 `docs/11` 最近 N 条记录 |
| 3 | **文档一致性** | 随机抽查 2-3 篇文档，验证交叉引用有效、版本号合理、无明显的格式破坏 | 随机选取 + 目视 |

> **审计结果记录：** 轻量审计结果无需生成独立审计报告文件。在 `docs/11_AI协作日志.md` 中追加一条"定期例行审计记录"即可（格式见下）。定期例行审计按版本号递增：R1, R2, R3...（从 Phase 3 启动后重新编号）。

```
## 定期例行审计 — YYYY-MM-DD

### 检查结果
- **版本：** 第 N 次例行审计
- **触发条件：** [已满 20 条协作日志 / 已满 4 周 / 其他]

### 风险跟踪状态
- 跟踪表总数：XX（Open: X / In Progress: X / Resolved: X / Verified: X / Closed: X）
- 新增风险：X 条（自上次审计以来）
- 到期未处理风险：X 条

### 协作日志质量
- 检查范围：第 N 条 ~ 第 N 条
- 合规率：X%
- 异常记录：X 条（说明）

### 文档一致性抽查
- 抽查文件：[文件 1]、[文件 2]
- 发现项：X 项

### 结论

```

```

├── 02_产品定位与愿景.md         # 产品方向与差异化

├── 06_用户体验流程设计.md        # UX 设计
├── 07_AI动态分析流程.md         # 八字分析流程
├── 08_系统架构设计.md           # 架构框架
├── 09_开发规范.md               # 开发规范（框架性说明）
├── 10_测试规范.md               # 测试规范（框架性说明）
├── 11_AI协作日志.md             # 每次 AI 执行的记录
├── 12_AI协作开发规范.md          # ⬅️ 当前文件（完整协作规范）
├── 13_API接口契约设计.md         # HTTP + WebSocket API 接口契约
│   └── 14_WebSocket实时通信协议设计.md # WebSocket 实时通信协议设计
└── HANDOVER.md                  # 项目交接手册
```

### 5.2 文档状态定义

| 状态 | 含义 | 谁可以修改 |
|------|------|------------|
| ✅ Approved | 已确认，不可直接修改 | 需通过 ADR 变更流程 |
| ✅ Accepted Concept | 概念已确认，细节待定 | 产品负责人 + AI（讨论层面） |
| ✅ Draft | 草稿，可自由修改 | AI 和产品负责人均可 |
| 🟡 Draft（待审核） | 等待产品负责人确认 | 产品负责人 |
| 🔴 Deprecated | 已废弃 | 产品负责人 |

> **Accepted Concept 状态使用指南：**
>
> ✅ **可开始的设计/开发活动：**
> - 技术方案的设计文档编写
> - 原型开发（Prototype）和可行性验证
> - 依赖于该决策的辅助模块设计（在决策被确认前保持模块独立）
> - 接口契约的前期设计（标注"依赖 ADR-XXX 确认后定稿"）
>
> ❌ **不可开始的开发活动：**
> - 基于该 ADR 的生产级代码开发（不可进入测试/发布流程）
> - 对外部的技术承诺（如"V1 将支持 X 特性"的公开声明）
> - 对其他已确认 ADR 的修改
>
> **从 Accepted Concept 到 Approved 的条件：**
> 原型验证通过 + 产品负责人书面确认 = 升级为 Approved

### 5.2.1 普通文档状态变更流程

本节定义**普通文档**（非 ADR、非冻结基线文档）的 Draft → Approved 状态变更流程。ADR 的批准流程见 §5.3.4；冻结基线文档的变更见 docs/22 §8.1（RFC → ADR 流程）。

#### 5.2.1.1 升阶条件（基本就绪标准）

产品负责人将普通文档从 🟡 Draft 提升为 ✅ Approved 时，应确认以下 4 项条件（比 ADR 的 5 项更轻）：

| # | 条件 | 说明 |
|---|------|------|
| 1 | **内容完整** | 文档覆盖其定位应有的核心内容，非占位符空壳 |
| 2 | **一致性** | 不与已 ✅ Approved / ✅ Accepted Concept 的 ADR 冲突，与所引用的其他文档交叉一致 |
| 3 | **定位清晰** | 文档的用途、受众、适用范围已在开头说明 |
| 4 | **元信息规范** | 版本号、日期、状态标签齐全，符合 §5.2 状态定义 |

> **与 ADR 批准条件（§5.3.4.1）的差异：** 普通文档不要求"影响分析（Consequences）"和"版本兼容性"两项——普通文档影响面局限于自身，无需 Consequence 论证；但保留"一致性检查"以防与已确认决策矛盾。

#### 5.2.1.2 升阶权限

| 角色 | 权限 |
|------|------|
| **产品负责人** | 唯一有权将文档标记为 ✅ Approved |
| **AI Agent** | 可新建 🟡 Draft 文档；可将 Draft 标记为 🟡 Draft（待审核）提请评审；**不得**自行标记 ✅ Approved |

> **例外：** 已被冻结的基线文档（`docs/08` / `docs/13` / `docs/14` / `docs/19` / `docs/21` 及 ADR 本身）不走本流程，受 Architecture Freeze 保护，任何改动须走 RFC → ADR 流程（见 docs/22 §8.1）。

#### 5.2.1.3 时限与超时（比 ADR 宽松）

| 维度 | 普通文档 | 对比 ADR（§5.3.4.2/3） |
|------|---------|----------------------|
| **强制时限** | ❌ 不设。不绑定 P0/P1/P2 审批时限 | ✅ 有（12h / 48h / 5 工作日） |
| **超时自动升阶** | ❌ 不设。普通文档不因超时自动变 Approved | ✅ 有（超时自动 → Accepted Concept） |
| **软性节奏（建议非强制）** | 文档提交后，产品负责人在下次"文档评审窗口"（Phase 门 / 阶段审计）统一处理待审核文档 | — |

> **设计理由：** 不设超时自动升阶，是为了规避"无人审批即默认批准"——普通文档宁可长期停留在 Draft（可用但未定稿），也不应在无人确认时被自动认定为权威版本。Audit Agent 可在定期审计（见 ARC-16-P2-005）中提示 Draft 文档积压情况。

#### 5.2.1.4 降阶与废弃

| 场景 | 规则 |
|------|------|
| **Approved → Draft 回退** | 仅产品负责人有权执行；须在 `11_AI协作日志.md` 记录回退理由（复用 §5.3.2 变更纪律精神） |
| **废弃（→ 🔴 Deprecated）** | 按 ARC-16-P2-002（废弃文档流程，待落地）执行，处理引用关系 |

### 5.2.2 文档版本号规则

所有项目文档采用语义化版本号 **`v<major>.<minor>`** 格式。版本号与文档状态独立——同一份文档在不同状态下可能有不同的版本号节奏。

#### 版本号范围定义

| 版本范围 | 适用阶段 | 含义 | 示例 |
|---------|---------|------|------|
| **v0.x** | Draft 阶段 | 草稿迭代，内容持续修改中。x 从 1 开始递增 | v0.1 → v0.2 → v0.3 |
| **v1.0** | 首次确认 | 文档完成首次审阅/批准/报告完成，内容已稳定 | Draft → Approved 时设为 v1.0 |
| **v1.x** | 确认后修订 | 对已确认文档的局部修改（非结构性变更） | v1.0 → v1.1 → v1.2 |
| **v2.0+** | 结构性变更 | 文档结构、核心决策或范围发生重大变化 | v1.x → v2.0（需配合 ADR 变更流程） |

#### 版本号递增条件

**v0.x → v0.x+1（minor 递增，Draft 阶段）：** 满足以下任一条件即应递增 minor 版本号：实质性修改（非错别字/格式调整）、新增或删除完整章节、核心数据变化、图表/架构图变更。简单格式调整、错别字修正、标点统一 **不递增** minor。

**v0.x → v1.0（首次确认）：** 满足**全部**以下条件：① 文档完成了目的定义的全部内容（无占位符）；② 通过产品负责人或指定评审者正式审阅；③ 状态从 🟡 Draft 变更为 ✅ Accepted Concept 或 ✅ Approved；④ 审计报告类文档执行完成、报告定稿。

**v1.x → v1.x+1（确认后修订）：** 已确认文档的局部章节修改、新增示例/备注/引用、根据反馈修正事实性错误、同步反映关联文档变化。递增前应在 `docs/11_AI协作日志.md` 记录修改摘要和影响范围。

**v1.x → v2.0+（结构性变更）：** 核心决策变更、文档结构重排、适用范围内重大变化、超过 30% 内容重写，或多个连续 minor 累积（建议 v1.9 后评估）。通常需配合 ADR 变更流程并在协作日志记录缘由。

#### 版本号与文档状态对应

| 文档状态 | 典型版本范围 | 说明 |
|---------|------------|------|
| 🟡 Draft | v0.1 ~ v0.x | 草稿阶段，持续递增 minor |
| 🟡 Draft（待审核） | v0.x | 等待产品负责人确认 |
| ✅ Accepted Concept | v0.x 或 v1.0 | 概念已确认，可标记 v1.0 或保持高于 v0.5 的 Draft 版本 |
| ✅ Approved | v1.0+ | 已确认后至少 v1.0 |
| 🔴 Deprecated | 保留原版本号 + Deprecated 标记 | 版本号不随废弃变化 |

#### 版本号声明位置

每个文档头部 meta 块必须包含版本号字段：`> **版本:** v<major>.<minor> | **日期:** YYYY-MM-DD | **状态:** <Draft/Approved/Deprecated>`，每次变更时同步更新日期。

#### 例外情况

- **ADR 记录（04_ADR_产品决策记录.md）：** 有独立决策编号体系，文档级版本号仅反映 ADR 集合的编辑状态
- **协作日志（11_AI协作日志.md）：** 按时间追加，使用条目编号而非版本号
- **风险跟踪表（18_架构风险跟踪表.md）：** 版本号反映编辑迭代，不反映其中风险条目的状态

### 5.2.3 废弃文档流程

#### 5.2.3.1 废弃前置条件

文档可被标记为 🔴 Deprecated 当满足以下**任一**条件：

| # | 条件 | 说明 | 示例 |
|---|------|------|------|
| 1 | **替代文档已存在** | 新正式文档取代该文档内容定位 | V2 架构文档取代 V1 架构文档 |
| 2 | **内容已过时** | 决策/描述/数据因方向调整不再适用 | 某项技术选型被正式否决 |
| 3 | **已被合并** | 内容被吸收到其他文档，独立存在不再必要 | 专项设计内容合并入主架构文档 |
| 4 | **已被拆分** | 内容拆分到多个新文档，原编号废止 | 通用规范拆分为前端/后端规范 |
| 5 | **范围被否决** | 所定义模块/功能/阶段被正式取消 | 某计划功能模块被取消，其设计文档 |

> 废弃判定由产品负责人做出。AI Agent 可提废弃建议，但不得自行标记 Deprecated。

#### 5.2.3.2 废弃操作步骤

```
① 废弃判定（产品负责人确认满足前置条件）
    ↓
② 文档头部更新（添加 Deprecated 标记 + 替代指引）
    ↓
③ 添加废弃横幅（标题下方、目录之前，醒目提示）
    ↓
④ 交叉引用修复（更新所有引用该文档的位置）
    ↓
⑤ 通知记录（在 11_协作日志 中记录废弃决定）
    ↓
⑥ 状态同步（更新文档地图如有）
    ↓
⑦ 清理倒计时（进入保留期，到期后评估物理删除）
```

步骤②头部格式：`> **状态:** 🔴 Deprecated` + `> **废弃日期:**` + `> **替代文档:**` + `> **废弃理由:**`。步骤③横幅置于标题下方、目录之前，注明"自 YYYY-MM-DD 起不再维护"及替代文档链接。步骤④按引用场景分别处理：可找到替代来源的更新为替代文档对应章节；无直接替代的改为引用替代文档并标注"相关内容已迁移"；引用已废弃决策且无替代的保留引用并加 `（📌 该文档已废弃，所引用决策可能不再有效）` 标注。

#### 5.2.3.3 清理时间线

| 阶段 | 时间 | 状态 | 可操作内容 |
|------|------|------|-----------|
| **保留期** | Deprecated 之日起 3 个 Sprint | 🔴 Deprecated | 可读取不可修改；交叉引用修复期 |
| **评估点** | 3 个 Sprint 后 | — | 产品负责人评估是否仍需要保留 |
| **可删除** | 评估通过后 | — | 物理删除文件，或转为归档存储 |

> **重要：** 物理删除文件前，必须确保所有交叉引用已修复完成，并在 `11_协作日志.md` 中记录删除操作。禁止在未确认引用关系的情况下删除任何文件。评估后如需恢复，产品负责人可取消 Deprecated 标记，恢复为 Draft 或对应状态。

#### 5.2.3.4 特殊情况

- **ADR 废弃：** 单个 ADR 废弃不废弃整份 04_ADR 文档——在 ADR 条目头部改状态为 🔴 Deprecated、同步附录索引、上方加替代说明。
- **协作日志不废弃：** `11_AI协作日志.md` 作为历史记录保留，不标记 Deprecated。

### 5.3 ADR 管理规则

#### 5.3.1 新增 ADR

1. 在 `docs/04_ADR_产品决策记录.md` 末尾按编号递增添加
2. 模板格式（保持与现有 ADR 一致）：
   ```markdown
   ## ADR-XXX：决策标题

   | 字段 | 内容 |
   |------|------|
   | **决策编号** | ADR-XXX |
   | **决策日期** | YYYY-MM-DD |
   | **状态** | ✅ Draft |
   | **决策者** | 产品负责人 |

   **Context（背景）：**
   ...

   **Decision（决策）：**
   ...

   **Consequences（选择理由）：**
   ...
   ```
3. 新增后同步更新附录的「ADR 状态索引」表格
4. 在 `11_AI协作日志.md` 中记录

#### 5.3.2 修改 ADR

- 已 ✅ Approved 的 ADR **不得直接修改**
- 如果发现已确认决策存在重大问题，按以下流程：
  1. 在 `04_ADR_产品决策记录.md` 附录的「质疑记录」区域记录质疑内容
  2. 在 `11_AI协作日志.md` 中记录建议
  3. 等待产品负责人确认是否修改
- 产品负责人同意修改后，由产品负责人本人修改
- ✅ Draft 状态的 ADR 可在讨论后直接更新

#### 5.3.3 ADR 与代码的关系

- 代码实现必须遵循已确认的 ADR
- 开发过程中如果发现 ADR 与实际情况冲突，先提出 ADR 修改建议，再调整代码
- 不允许"先改代码，再补 ADR"的逆向操作

#### 5.3.4 ADR 批准流程

##### 5.3.4.1 批准条件

产品负责人批准 ADR（Draft → Approved）时，应确认以下条件：

| # | 条件 | 说明 |
|---|------|------|
| 1 | **问题陈述清晰** | ADR 的 Context 部分完整描述待解决的问题背景和驱动力 |
| 2 | **方案明确** | Decision 部分给出具体、可执行的技术/产品决策，而非模糊方向 |
| 3 | **影响分析完整** | Consequences 部分列出优点、代价、后续影响，至少各 2 条 |
| 4 | **不与其他 Approved ADR 冲突** | 快速交叉对比已有 Approved ADR，无逻辑矛盾 |
| 5 | **版本兼容** | 如果是技术选型 ADR，确认不破坏已确定的架构方向（ADR-014/015/016 等） |

> 满足全部 5 项条件的产品负责人可直接批准。
> 满足条件 1~3 但不满足 4~5 的，应标记为 **Accepted Concept** 并记录待解决的问题。

##### 5.3.4.2 批准时限

| ADR 等级 | 时限 | 计时起点 |
|----------|------|---------|
| **🔴 P0（阻止级）** | **12 小时内** | 产品负责人收到通知之时起 |
| **🟡 P1（优先级）** | **48 小时内** | 产品负责人收到通知之时起 |
| **🟢 P2（优化级）** | **5 个工作日内** | 产品负责人收到通知之时起 |

> **通知方式：** ADR 提交人（AI Agent 或人类）在协作日志 `docs/11_AI协作日志.md` 中追加记录，
> 标注 `【ADR审批请求】ADR-XXX（等级）`，同时通过项目约定的即时通信渠道通知产品负责人。
>
> **时限定义：** "12 小时内"指自然时间的 12 小时，含非工作日；
> "5 个工作日"指排除周末和法定节假日的 5 天。

##### 5.3.4.3 超时替代机制

若产品负责人在上述时限内未做出响应，按以下规则自动处理：

| 场景 | 超时处理 | 后续操作 |
|------|---------|---------|
| **P0 ADR** 超时 | 自动升为 ✅ **Accepted Concept**，允许开始设计/原型验证 | 产品负责人返回后需在 24h 内复审并确认（Approved）或驳回（Back to Draft） |
| **P1 ADR** 超时 | 自动升为 ✅ **Accepted Concept**，允许基于此方案开始设计 | 产品负责人返回后需在 5 个工作日内复审 |
| **P2 ADR** 超时 | 保留 🟡 **Draft（待审核）** 状态，不自动升级 | AI 可在协作日志中记录"超时：P2 ADR 待审批"，待产品负责人下次工作时处理 |

> **重要：** 超时自动升级为 Accepted Concept **不等于** Approved。Accepted Concept 的定位是"概念已确认，细节待定"——允许开始设计工作，但不应视为最终决策。
> 产品负责人返回后仍需对超时升级的 ADR 进行确认或驳回。

##### 5.3.4.4 催办机制

| 角色 | 催办条件 | 催办方式 |
|------|---------|---------|
| **提交 ADR 的 AI** | 超过时限 50% 未响应 | 在协作日志中追加 `【催办】ADR-XXX 审批，已超过时限 XX%，请产品负责人确认` |
| **Audit Agent** | 任意 ADR 超过时限未处理 | 在下次审计中标注"ADR 审批延迟"，作为治理风险记录 |
| **其他 AI Agent** | 依赖该 ADR 决策的工作受阻 | 可在协作日志中追加 `【受阻】ADR-XXX 未审批，当前工作 YYY 阻塞` 说明影响 |

### 5.4 文档交叉引用规范

- 文档之间引用使用相对路径：`[项目简介](./01_项目简介.md)`
- 引用 ADR 使用编号：`ADR-009（八字分析6步流程）`
- 引用代码使用 `文件路径:行号` 格式

## 第六章：代码修改规则

### 6.1 修改前的准备

- [ ] 已读取项目相关文档（P0 文档必须全覆盖）
- [ ] 已了解项目文件结构
- [ ] 已确认修改范围不会影响未涉及的模块
- [ ] 已确认修改不违反任何已确认的 ADR
- [ ] 已确认修改不涉及【待确认】内容（如涉及，需先确认）

### 6.2 修改中的纪律

| 规则 | 说明 |
|------|------|
| **最小范围** | 只修改任务指定的文件，不修改无关文件 |
| **最小改动** | 在一个文件内只改必须改的行，不做"顺手优化" |
| **单一职责** | 一次执行只做一件事，不混入其他修改 |
| **保持风格** | 新代码的注释密度、命名风格、格式与周围代码一致 |
| **边改边记** | 记录修改的文件列表、行范围、修改理由 |
| **不破坏** | 不删除未涉及的函数/变量/文件，哪怕看起来"没用" |

### 6.3 修改后的自检清单

- [ ] 所有修改是否在计划范围内？
- [ ] 语法无错误（可以编译/解释通过）？
- [ ] 是否破坏了任何现有功能？
- [ ] 是否有硬编码值需要改为配置？
- [ ] 是否有敏感信息（API Key、密钥）泄露风险？
- [ ] 是否有遗漏的边界情况？

### 6.4 删除文件的规则

- **禁止在未确认的情况下删除任何文件**
- 如需删除文件，必须：
  1. 在协作日志中明确列出待删除文件及理由
  2. 等待产品负责人明确确认
  3. 确认后先备份再删除

### 6.5 Git 提交规则

- **禁止自动提交 git**（严格遵守本条款）
- 必须先将 diff 展示给产品负责人确认
- 确认后方可执行提交
- 提交信息格式：
  ```
  <type>: <简要描述>

  - 修改文件：列出文件列表
  - 关联 ADR：ADR-XXX
  - 修改理由：一句话说明

  Co-Authored-By: <Agent 名称/ID> <agent-email>
  ```
  > 替换 `<Agent 名称/ID>` 为当前 AI Agent 的身份标识（如 `Agent-20260716-01`），替换 `<agent-email>` 为项目约定的 Agent 邮箱（如 `agent@zhiming-ai.dev`）。

## 第七章：测试与验收规则

### 7.1 测试级别定义

| 级别 | 说明 | 适用阶段 |
|------|------|----------|
| 单元测试 | 测试单个函数/模块的正确性 | 开发中 |
| 集成测试 | 测试多模块交互的正确性 | 模块完成后 |
| 人工验收 | 产品负责人手动验证功能 | 功能完成后 |
| 边界测试 | 测试异常/边界输入的处理 | 开发中 |

### 7.2 自检要求

每次修改后（无论是代码还是文档），执行以下自检：

```
1. 完整性检查：所有计划中的修改是否已完成？
2. 正确性检查：修改的内容是否正确？
3. 一致性检查：修改是否与现有风格/规范一致？
4. 关联性检查：是否更新了相关的文档、日志？
5. 回归检查：修改是否破坏了未涉及的功能？
```

### 7.3 验收标准

产品负责人验收时关注以下维度：

| 维度 | 说明 | 通过标准 |
|------|------|----------|
| **功能正确性** | 实现是否符合预期 | 核心功能运行无误 |
| **代码质量** | 代码可读性、维护性 | 无明显坏味道、遵循项目风格 |
| **文档完整性** | 是否更新了必要文档 | ADR 引用正确、日志完整 |
| **修改范围** | 是否超出计划 | 没有计划外修改 |
| **可测试性** | 是否便于测试 | 核心逻辑可单元测试 |

### 7.4 测试环境与限制

- 本项目中所有 AI 调用的外部 LLM API（如 SiliconFlow、DeepSeek）**必须在 AI 沙箱外执行**
- AI 可以在沙箱内编写测试代码和测试计划
- AI 可以在沙箱内执行单元测试（不涉及外部 API）
- 涉及外部服务的集成测试需在沙箱外运行

## 第八章：执行报告规范

### 8.1 报告结构

每次 AI 执行任务后，必须在 `docs/11_AI协作日志.md` 中追加记录。报告格式如下：

```markdown
## 第 N 次记录 — YYYY-MM-DD

### 任务信息

- **任务来源：** [产品负责人 / 用户指令]
- **任务描述：** [一句话描述本次任务]
- **AI 角色：** [文档撰写者 / 方案分析师 / 代码开发者 / 评审者 / 测试者]
- **使用的认知框架：** [CoT / 系统思维 / 第一性原理等]
- **信息来源：** [项目内文档 / 外部知识 / 模型训练数据 / 混合]
- **确定性等级：** [事实性陈述 / 分析性推理 / 推测性建议]

### 预检确认

- **预检清单：** [✅ 已完成 / ⬜ 跳过（简单查询）]
- **已读 P0 文档数：** 5/5
- **已读协作日志最后记录：** [日期/记录编号]

#### 字段填写规则

| 字段 | 可选值 | 说明 |
|------|--------|------|
| **信息来源** | `项目内文档` | 基于项目文档体系（docs/*.md）、代码库中已有信息 |
| | `外部知识` | 基于外部知识库、网络搜索、第三方资料、命理经典典籍 |
| | `模型训练数据` | 基于 LLM 预训练知识（模型自身训练数据涵盖的通用知识） |
| | `混合` | 综合多种来源 |
| **确定性等级** | `事实性陈述` | 直接来自项目文档或 ADR 的确定性信息、可验证的排盘计算结果 |
| | `分析性推理` | 基于已有信息做出的推理分析（建议方案、设计决策、影响分析） |
| | `推测性建议` | 基于经验和模式的推测性建议 |

> **与 ADR-012 的关系：** 两字段直接派生自 ADR-012（数据三层体系），是其在治理规范执行层面的映射——`fact → 事实性陈述`，`inference → 分析性推理`，`pending → 推测性建议`；信息来源标注对应 ADR-012 的来源可信度原则。规范记录中的确定性等级用于帮助后续 AI 快速评估历史协作信息的可信度层次。

### 修改清单

| 文件 | 操作 | 行范围 | 说明 |
|------|------|--------|------|
| `xxx.md` | 新增/修改/删除 | L1-L50 | 做了什么 |
| `xxx.py` | 修改 | L10-L20 | 做了什么 |

### 关键决策

- **决策 1：** [描述] — 理由：[理由] — 状态：[待确认/已确认/建议]
- **决策 2：** [描述] — 理由：[理由] — 状态：[待确认/已确认/建议]

### 遇到的问题

- **问题 1：** [描述] — [解决状态：已解决/待处理/需产品负责人决策]

### 引用 ADR

- **参考 ADR：** ADR-XXX, ADR-YYY
- **ADR 变更：** [无 / 建议新增 / 建议修改]

### 快速建议

- [对下一步工作的建议]
- [对项目管理的建议]

### 遗留问题

- [未解决事项]
- [下次 AI 需要注意的坑]
```

### 8.2 报告输出时机

| 场景 | 输出要求 |
|------|----------|
| 完成一个独立任务 | ✅ 必须输出 |
| 仅读取文件、未做修改 | ✅ 建议在日志中记录"已读取"加上关键词摘要 |
| 多轮对话中途暂停 | ✅ 输出阶段性记录 |
| 发现严重问题但未解决 | ✅ 必须记录问题详情 |
| 日常咨询性对话 | ⬜ 可选 |

### 8.3 报告质量要求

- **完整性：** 修改了什么、为什么改、有什么影响，三者缺一不可
- **可复现性：** 描述应足够清晰，使下一位 AI 能理解上下文并继续工作
- **诚实性：** 遇到困难或发现错误应如实记录，不掩盖问题
- **决策标注：** 所有决策需标注状态（待确认/已确认/建议），方便产品负责人快速判断哪些需要关注

### 8.4 紧急问题报告

遇到以下紧急问题，应立即记录并在日志开头加 **【紧急】** 标记：

- 安全性问题（数据泄露、权限漏洞）
- 已确认的 ADR 被发现有重大错误
- 当前开发方向与产品核心定位产生严重偏差
