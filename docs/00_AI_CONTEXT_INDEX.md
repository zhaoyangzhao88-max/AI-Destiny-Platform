## 1. 文档目的

### 1.1 为什么需要这份文档

本项目在 `docs/` 目录下已有 19 份 Markdown 文档（含本文档），覆盖产品、架构、API、AI 治理、测试五大领域。但这些文档的入口索引长期存在以下问题：

| 问题 | 来源 | 影响 |
|------|------|------|
| **多入口不一致** | CLAUDE.md P0/P1 列表、HANDOVER.md Sec 6.1 与 Sec 7 Step 1-2、docs/12 App B 各列出不同读取集 | AI Agent 首次进入时不知道优先读哪个列表 |
| **无读取等级** | 所有文档被同等对待，没有区分"必须读"和"按需读" | 每次加载全部文档造成上下文浪费 |
| **无角色差异化** | 没有按 Agent 角色（架构/开发/审计/产品/知识）区分需要读取的文档 | Audit Agent 读了所有 UX 文档但不需要 |
| **无确认机制** | 没有要求 Agent 在读取后做"已读确认" | 无法追踪 Agent 是否真的读完了核心文档 |

### 1.2 本文档统一/替换的旧入口

本文档旨在成为**唯一的 AI 第一入口**，逐步替代以下旧入口中的读取列表推荐：

| 旧入口 | 位置 | 关系 |
|--------|------|------|
| `CLAUDE.md` P0/P1 必读列表 | 项目根目录 | 建议后续更新指向 00 L0/L1 |
| `docs/HANDOVER.md` Sec 6.1 + Sec 7 | docs/HANDOVER.md | 建议后续 Step 1 简化为"先读 00" |
| `docs/12_AI协作开发规范.md` 附录 B | docs/12 附录 B | 00 第 4 章文档地图已全覆盖 |

### 1.3 本文档不做什么

- ❌ **不替代任何现有文档的内容** — 本文档仅提供索引和读取规则，不重复其他文档的内容
- ❌ **不修改已有 ADR** — ADR 的创建、修改、废弃流程以 `docs/12_AI协作开发规范.md` 第 5 章为准
- ❌ **不定义协作流程细节** — 完整的协作开发流程以 `docs/12_AI协作开发规范.md` 为准
- ❌ **不定义角色权限细节** — 完整的 Agent 角色与权限模型以 `docs/17_AI Agent角色与权限规范.md` 为准

## 2. AI 启动协议

所有 AI Agent（Claude Code、ChatGPT、其他大模型）进入本项目后，必须按以下标准流程操作：

```
┌─────────────────────────────────────────────────────────────┐
│              AI 启动协议 — 标准 5 步流程                       │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 1 ───→ 读取 AI 上下文索引 (docs/00_AI_CONTEXT_INDEX.md) │
│                  ↓ 确定自己的 Agent 角色                       │
│  Step 2 ───→ 读取角色对应文档（按第 5 章角色矩阵）              │
│                  ↓ 确定当前任务类型                            │
│  Step 3 ───→ 读取任务相关文档（按第 3 章等级确认）              │
│                  ↓ 确认涉及领域的 ADR 状态                     │
│  Step 4 ───→ 确认 ADR 状态（检查 docs/04_ADR 中相关 ADR）      │
│                  ↓ 通过第 7 章检查清单                         │
│  Step 5 ───→ 执行前检查 → 执行 → 输出完成报告                  │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### Step 1：读取上下文索引

- 读取本文档（`docs/00_AI_CONTEXT_INDEX.md`）
- 确定自己的 Agent 角色（见第 5 章）
- 如果同时承担多个角色，取**读取范围最大的角色**

### Step 2：读取角色对应文档

- 按第 5 章「Agent 读取规则」中对应角色的「必须读取」列表完整阅读
- Level 0 文档需逐份注明"已读"
- 所有 Level 0 + Level 1 文档为首次进入必须读取

### Step 3：读取任务相关文档

- 按任务领域从第 4 章文档地图中筛选 Level 2 / Level 3 文档
- 任务开始前确保相关文档的**最新版本**已被读取

### Step 4：确认 ADR 状态

- 在 `docs/04_ADR_产品决策记录.md` 中以 ADR 附录索引确认相关 ADR 状态
- **关键区分：**
  - ✅ Approved ADR（ADR-001~012）：**架构红线**，不可直接修改
  - ✅ Accepted Concept ADR（ADR-013）：概念已确认，技术方案待设计，可参与讨论
  - ✅ Draft ADR（ADR-014~020）：处于草稿阶段，可按 docs/12 第 5 章流程参与设计讨论

### Step 5：执行 → 输出完成报告

- 通过第 7 章「AI 执行前检查清单」检查
- 执行任务
- 按第 8 章「AI 执行完成报告要求」输出报告
- 在 `docs/11_AI协作日志.md` 追加记录

## 3. 文档读取等级

文档按读取必要性和频次分为四级：

| 等级 | 名称 | 读取频率 | 说明 |
|------|------|----------|------|
| **Level 0** | 必须读取 | 每次任务 | 理解项目"是什么、为什么、红线在哪"的最小知识集 |
| **Level 1** | 项目规则 | 首次进入 / 规则变更时 | 定义"如何协作、如何决策"的规则体系 |
| **Level 2** | 核心设计 | 按任务类型选择性读取 | "具体怎么做"的核心设计文档 |
| **Level 3** | 任务专项 | 仅涉及相关领域时 | 特定场景下才需要的参考文档 |

### Level 0：必须读取（7 份）

理解项目本质的最小知识集。任何 Agent、任何任务开始时都必须读取。

- `docs/00_AI_CONTEXT_INDEX.md` — ⬅️ 本文档，所有 AI 的第一入口
- `docs/01_项目宪章(Project Charter).md` — **项目最高治理文档（宪章）**，定义使命、北极星、治理原则、AI 宪法
- `docs/HANDOVER.md` — 项目全局概览、交接规则、当前阶段
- `docs/01_项目简介.md` — 项目背景、核心理念、目标用户、参考项目
- `docs/02_产品定位与愿景.md` — 产品差异化定位、商业模式、竞争分析
- `docs/04_ADR_产品决策记录.md` — 所有已确认和待确认的产品决策（架构红线）
- `docs/05_知命_Persona角色设定.md` — AI 角色「知命」的完整人格设定

> **读取顺序提示：** 00 入口 → 01 宪章（最高纲领） → HANDOVER（快速上手） → 01 简介/02 定位/04 ADR/05 Persona（领域理解）。宪章在 HANDOVER 之前，因为它定义所有其他文档的存在理由。

### Level 1：项目规则（3 份）

定义协作规则和治理机制。首次进入项目后应完整阅读一次，后续按需引用具体章节。

- `docs/12_AI协作开发规范.md` — AI 协作开发完整规范（流程、审计、报告、ADR 管理）
- `docs/17_AI Agent角色与权限规范.md` — 多 Agent 治理、权限模型、冲突裁决、交接协议
- `CLAUDE.md` — Claude Code 项目入口规则与 Plan 模式要求（项目根目录）

### Level 2：核心设计（7 份）

按任务领域选择性读取的核心设计文档。

| 文档 | 适用角色 | 主要阅读场景 |
|------|---------|-------------|
| `docs/03_当前开发状态.md` | 所有角色 | 了解项目全局进度 |
| `docs/06_用户体验流程设计.md` | Product / Architect | 用户旅程与 UX 设计 |
| `docs/07_AI动态分析流程.md` | Product / Knowledge / Developer | 八字分析 6 步流程 |
| `docs/08_系统架构设计.md` | Architect / Developer | 系统架构框架与技术方案 |
| `docs/11_AI协作日志.md` | **所有角色** | **最后 3 条执行记录必读**，了解最近工作 |
| `docs/13_API接口契约设计.md` | Developer / Architect | HTTP + WebSocket API 接口契约 |
| `docs/14_WebSocket实时通信协议设计.md` | Developer / Architect | WebSocket 协议完整设计 |

### Level 3：任务专项（3 份）

特定场景下才需要的参考文档。仅在任务明确涉及相关领域时读取。

| 文档 | 读取条件 |
|------|---------|
| `docs/09_开发规范.md` | 当前为占位符，待填充。仅编码任务涉及具体规范时 |
| `docs/10_测试规范.md` | 当前为占位符，待填充。仅测试任务涉及具体规范时 |
| `docs/15_Phase2架构审计报告.md` | 仅架构审计任务或需了解 Phase 2 审计结论时 |
| `docs/16_AI协作治理体系审计报告.md` | 仅治理体系改进任务或需了解治理审计结论时 |

## 6. 文档状态说明

### 6.1 统一状态体系

本文档定义项目的**统一文档状态体系**，适用于 `docs/` 目录下的所有文件：

| 状态 | 含义 | 可操作性 |
|------|------|---------|
| **Draft** | 草稿阶段，AI + 产品负责人可自由编辑 | AI 可建议修改，可自编辑 |
| **Proposed** | 已提交建议，等待产品负责人审阅 | AI 可建议修改，需确认后生效 |
| **Accepted** | 概念已被接受，技术方案/实现细节待定 | AI 可参与讨论，不可做最终决策 |
| **Approved** | 已正式批准，内容冻结 | 不可直接修改（按 docs/12 第 5 章流程提交质疑） |
| **Implemented** | 已实施完成，内容冻结 | 不可修改（保留仅用于追溯） |
| **Deprecated** | 已废弃，保留仅用于历史追溯 | 不可用于新任务，阅读时注意时效性 |

### 6.2 与 docs/12 第 5.2 节的关系

`docs/12_AI协作开发规范.md` 第 5.2 节定义了项目的**原始文档状态体系**。本文档第 6.1 节定义了**统一后的标准化状态体系**，用于文档地图和 AI 上下文索引。两者关系如下：

| 00 统一状态 | docs/12 原始状态 | 差异说明 |
|------------|-----------------|---------|
| Draft | Draft | 保持不变 |
| **Proposed** | Draft（pending review） | 命名统一，行为一致 |
| Accepted | Accepted Concept | 简化命名，含义不变 |
| Approved | Approved | 保持不变 |
| **Implemented** | —（未定义） | **新增状态**：覆盖审计报告(15/16)这类一次性冻结产出 |
| Deprecated | Deprecated | 保持不变 |

> **重要：** 本文档不替代 docs/12 第 5.2 节的定义。文档状态的**变更流程和审批机制**仍以 `docs/12_AI协作开发规范.md` 第 5 章为准。本文档第 6.1 节的状态体系用于文档地图**标注和索引**，为 AI Agent 提供统一的文档状态语言。

### 6.3 新增「Implemented」状态的理由

`docs/12_AI协作开发规范.md` 第 5.2 节未定义审计报告这类"一次性产出、完成后冻结"的文档状态。`docs/15_Phase2架构审计报告.md` 和 `docs/16_AI协作治理体系审计报告.md` 标注为 Implemented（v1.0）：

- 既不是 Draft（不可再修改）
- 也不是 Approved（不构成约束性产品决策）
- 后续审计应创建新报告而非更新旧报告

## 7. AI 执行前检查清单

每个 AI Agent 在开始执行任务前，必须逐项通过以下 10 项检查。任何一项不通过则不得开始执行。

| # | 检查项 | 通过标准 | 验证方式 |
|---|--------|---------|---------|
| 1 | **已读上下文索引** | 已完整阅读 `docs/00_AI_CONTEXT_INDEX.md` 全文 | 自检 |
| 2 | **已读必读文档** | 按第 5 章角色矩阵已读完「必须读取」范围内的所有文档 | 逐份确认 |
| 3 | **已读最近日志** | 已阅读 `docs/11_AI协作日志.md` 最后 3 条执行记录 | 列出最后 3 条日志时间+任务名称 |
| 4 | **任务边界清晰** | 明确知道当前任务是什么、不做什么、输出是什么 | 用一句话重述任务 |
| 5 | **ADR 红线确认** | 已确认任务涉及领域的 ADR 状态，没有计划修改 Approved ADR | 列出涉及 ADR 编号+状态 |
| 6 | **文档状态确认** | 已确认要修改/创建的文档当前状态，在允许范围内 | 列出文档+修改类型(新建/追加/修改) |
| 7 | **角色声明** | 已声明本次的 AI 类型、治理角色、功能角色 | 按 `Agent-YYYYMMDD-NN` 格式注册 |
| 8 | **关联文档检查** | 修改文档前已检查该文档的交叉引用关系 | 列出可能受影响的关联文档 |
| 9 | **执行报告准备** | 已了解完成后的报告输出要求（见第 8 章） | 自检 |
| 10 | **权限确认** | 确认本次修改在本人角色（docs/17 第 8 章）的写权限范围内 | 自检 |

> **注意：** 检查清单不替代 `docs/17_AI Agent角色与权限规范.md` 第 10 章的预检流程，两者互补。短任务（如单文件追记）可跳过第 2 项中 L2 文件的详细阅读，但 L0+L1 不可跳过。
