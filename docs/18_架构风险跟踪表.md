## 2. 风险状态体系

每个风险条目从发现到关闭经历以下 5 个状态：

```
┌──────────┐     ┌──────────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
│  Open    │ ──→ │  In Progress │ ──→ │ Resolved │ ──→ │ Verified │ ──→ │  Closed  │
│ (发现)   │     │  (处理中)    │     │ (已修复)  │     │ (已验证)  │     │ (已关闭)  │
└──────────┘     └──────────────┘     └──────────┘     └──────────┘     └──────────┘
```

| 状态 | 含义 | 进入条件 | 典型动作 |
|------|------|---------|---------|
| **🟢 Open** | 问题已发现，已登记，未开始处理 | 审计完成 / AI 发现报告 | 登记到跟踪表，分配负责人 |
| **🟡 In Progress** | 正在处理中 | 负责人确认接收，开始处理 | 制定修复方案，执行修复 |
| **🔵 Resolved** | 已修复，待验证 | 修复已提交，等待复审 | 通知 Audit Agent / 产品负责人验证 |
| **🟣 Verified** | 已验证通过，待关闭 | 复审确认修复有效 | 执行快速确认审计，记录验证结果 |
| **⚫ Closed** | 已正式关闭 | 产品负责人确认 | 在协作日志中记录关闭决定 |

### 状态流转规则

1. **不可跳转** — 必须按顺序流转（Open → In Progress → Resolved → Verified → Closed）
2. **可驳回** — Verified 未通过时退回 In Progress
3. **可暂缓** — P2 风险可由产品负责人直接从 Open 标记为 Deferred（暂缓），但不计入 Closed
4. **记录要求** — 每次状态变更需在 `docs/11_AI协作日志.md` 中记录

## 3. 风险等级

按 `docs/12_AI协作开发规范.md` 4.4 节定义的审计分级体系：

| 等级 | 名称 | 颜色 | 含义 | 响应时限 |
|------|------|:----:|------|---------|
| **P0** | 阻止级 | 🔴 | 必须解决，否则禁止进入下一阶段 | 立即处理 |
| **P1** | 优先级 | 🟡 | 建议解决后进入下一阶段，紧急度低于 P0 | 24h 内响应 |
| **P2** | 优化级 | 🟢 | 可后续优化，不影响阶段进入 | 无时限 |

### P0 典型场景

- 架构冲突（ADR 之间或 ADR 与架构设计矛盾）
- API/WS 契约跨文档矛盾（字段名、路径、数据格式直接冲突）
- 核心技术路线未确定（阻碍编码开始的关键决策缺失）
- 数据模型在技术上有不可行的部分

### P1 典型场景

- 文档遗漏（ADR 已确定但相关设计文档未更新）
- 设计细节不足（方案方向正确但缺少实现细节）
- 扩展性问题 / 边界情况未覆盖
- 流程/机制缺失但非阻止级

### P2 典型场景

- 命名优化、示例完善、文档格式调整
- 注释和说明补充
- 非功能性的可维护性改进

## 7. 风险关闭流程

### 7.1 标准 5 步流程

```
┌──────────────────────────────────────────────────────────────────┐
│                   风险关闭标准流程                                  │
├──────────────────────────────────────────────────────────────────┤
│                                                                   │
│  ① 发现 ───→ 审计发现 / AI 识别 / 产品负责人提出                  │
│        ↓       在跟踪表中登记（ID + 描述 + 等级 + 来源）            │
│                                                                   │
│  ② 登记 ───→ 填写完整风险条目（影响 + 建议 + 负责人 + 关闭条件）   │
│        ↓       分配负责人角色                                      │
│                                                                   │
│  ③ 处理 ───→ 负责人制定修复方案 → 执行修复                          │
│        ↓       状态 → In Progress                                  │
│                                                                   │
│  ④ 验证 ───→ Audit Agent 或产品负责人执行复审                      │
│        ↓       P0：独立快速确认审计                                 │
│                P1/P2：可纳入下次阶段审计或由产品负责人直接确认        │
│                                                                   │
│  ⑤ 关闭 ───→ 产品负责人确认 → 状态 → Closed                       │
│               在 docs/11_AI协作日志.md 记录关闭决定                 │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

### 7.2 各等级验证标准

| 等级 | 验证方式 | 验证人 | 验证深度 |
|------|---------|--------|---------|
| **P0** | 独立快速确认审计 | Audit Agent（不可与修复者为同一 AI） | 完整复审修复方案和修复结果 |
| **P1** | 纳入下次阶段审计 / 产品负责人直接确认 | Audit Agent / 产品负责人 | 确认修复满足关闭条件 |
| **P2** | 产品负责人确认 | 产品负责人 | 简略确认即可 |

### 7.3 跟踪表更新纪律

| 动作 | 频率 | 负责人 |
|------|------|--------|
| 新风险登记 | 发现后立即 | 发现者（任何 Agent） |
| 状态更新 | 每次状态变更 | 当前负责人 |
| Dashboard 刷新 | 每次状态变更 | Audit Agent |
| 完全关闭记录 | 关闭时 | 产品负责人 |

### 7.4 风险关闭记录模板

每次关闭风险时，必须在 `docs/11_AI协作日志.md` 中追加记录：

```
--- 风险关闭记录 ---
风险 ID:    ARC-XX-YYY-NNN
风险标题:   [原问题描述摘要]
关闭日期:   YYYY-MM-DD
关闭理由:   [修复概述 + 验证结果]
处理人:     [角色 / Agent ID]
验证人:     [Audit Agent / 产品负责人]
关联文档:   [修复涉及的文件列表]
```
