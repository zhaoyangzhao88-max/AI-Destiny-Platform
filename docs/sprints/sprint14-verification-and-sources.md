## 第一章：项目 AI 协作原则

### 1.1 三层协作模型

本项目采取 **「产品负责人驱动 + AI 协作执行 + 文档即契约」** 的协作模型：

```
产品负责人 ──→ 定义目标、确认决策、验收结果
    │
    ▼
AI 协作层  ──→ 执行任务、生成方案、记录过程
    │
    ▼
文档即契约 ──→ 所有关键信息写进文档，不依赖口头或隐性知识
```

### 1.2 核心原则

| 原则 | 说明 | 违反示例 |
|------|------|----------|
| **文档优先** | 任何项目关键信息必须写入文档，不允许仅存在于聊天记录中 | AI 在对话中做出设计决策但未写入 ADR |
| **不假设** | 标记【待确认】的内容不得擅自填补，必须等待产品负责人确认 | AI 自行决定技术选型并开始编码 |
| **先读后写** | 开始任何任务前必须读取相关文档，理解上下文 | 未读 HANDOVER 就开始修改代码 |
| **最小修改** | 外科手术式修改，只改必须改的行 | 顺便"优化"了无关代码 |
| **完整记录** | 每次任务执行后必须记录到协作日志 | 修改了大量文件但日志只写了"修复了问题" |
| **可追溯** | 每个决策、每次修改应能从文档追溯责任人、时间和理由 | 修改文件后未在日志中记录范围和理由 |

### 1.3 禁止行为清单

- ❌ 禁止在未读取核心文档前开始工作
- ❌ 禁止直接修改已确认的 ADR
- ❌ 禁止自行填补【待确认】内容
- ❌ 禁止删除或覆盖用户数据
- ❌ 禁止自动 git 提交（必须先展示 diff 等待确认）
- ❌ 禁止在 AI 沙箱内直接调用外部 LLM API
- ❌ 禁止重写无关代码（包括"顺手优化"）

## 第二章：AI 参与角色定义

### 2.1 角色分类

在本项目中，AI 根据任务类型扮演不同角色。同一 AI 会话在不同阶段可能切换角色。

| 角色 | 职责范围 | 权限边界 | 典型场景 |
|------|----------|----------|----------|
| **文档撰写者** | 创建、补充、优化项目文档 | 可以新增 Draft 文档，不得修改已 Approved 内容 | 写 PRD、写技术文档、补充 ADR |
| **方案分析师** | 分析需求，提出多个可选方案 | 提供建议方案，标记为"建议"，不可自行为【待确认】做决定 | 技术选型对比、架构设计讨论 |
| **代码开发者** | 编写业务代码、修复 Bug | 仅修改任务范围内的代码，不做范围外修改 | 实现功能、修复缺陷 |
| **评审者** | 对代码或文档进行 Review | 只提修改建议，不直接修改目标内容 | Code Review、文档评审 |
| **测试者** | 编写和执行测试用例 | 可以创建测试文件，不修改被测源码（除测试框架配置外） | 单元测试、集成测试 |

### 2.2 角色切换规则

```
开始任务 → 确定任务类型 → 匹配角色 → 执行 → 记录 → 切换/结束
```

- 同一轮次中如需切换角色，需明确声明角色转换
- 示例：*"文档撰写阶段结束，现在切换为代码开发者角色开始实现"*

### 2.3 AI 能力边界声明

AI 在执行所有角色时需明确以下事项：

1. **信息来源标注** — 基于项目内文档 / 基于外部知识 / 基于模型训练数据
2. **确定性分级** — 事实性陈述 / 分析性推理 / 推测性建议
3. **不确定时处理** — 不确定时需向产品负责人提问，而非自行推断

### 3.2 按任务类型的细分工流

#### 3.2.1 文档任务

```
读取文档上下文 → 明确文档定位与受众 → 撰写/修改 → 交叉引用检查
    → 格式一致性检查 → 输出报告 → 记录日志
```

- 新建文档需在文档头部标注版本、日期、状态
- 修改现有文档需在日志中说明修改范围和原因
- Draft 状态文档可自由修改，Approved 状态文档仅能通过 ADR 流程修改

#### 3.2.2 开发任务

```
读取相关文档 → 理解需求边界 → 确定技术方案 → Plan 确认
    → 编码实现（严格按范围） → 自检 → 输出报告 → 记录日志
```

- 编码前必须确认 Plan 已被产品负责人接受
- 编码过程中不得偏离 Plan 范围
- 修改现有文件时先展示受影响的行范围，再动手

#### 3.2.3 Review 任务

```
读取目标文件 → 读取相关上下文文档 → 多维度审查
    → 输出审查报告（含风险等级） → 记录日志
```

- 区分"严重问题"、"建议优化"、"风格偏好"三个等级
- 问题必须有具体行号/段落引用

### 3.3 AI 协作时的上下文获取规范

本规范适用于 AI 需要进行架构分析、代码审查、设计决策、任务规划等需要理解项目背景的场景。

#### 3.3.1 文件优先原则

当 AI 需要进行架构分析、代码审查、设计决策、任务规划时：

- **不允许**仅根据用户提供的摘要判断项目状态或做出设计决策
- **必须优先读取项目中的真实文档**，获取第一手信息
- 读取范围包括但不限于：
  - `docs/*.md` — 项目文档体系
  - `CLAUDE.md` — 项目入口规则与必读文档索引
  - `README` — 项目概述（如有）
  - 架构设计文档（如 `08_系统架构设计.md`）
  - ADR 文档（如 `04_ADR_产品决策记录.md`）
  - API 契约文档（如 `13_API接口契约设计.md`）
  - 数据模型文档（如 `14_WebSocket实时通信协议设计.md` 4.3 节）
- 读取后应在分析中**引用具体文件路径和章节**，而非笼统描述

**典型应用场景：**

| 场景 | 必须读取的文档 | 禁止的做法 |
|------|---------------|-----------|
| 架构分析 | ADR 文档、系统架构设计、API 契约、开发状态 | 仅凭用户口头描述判断架构 |
| 代码审查 | 架构设计、开发规范、测试规范 | 仅审查代码语法不对比设计约定 |
| 设计决策 | 所有相关 ADR、现有架构文档、UX 设计 | 在未知已有决策的情况下提出替代方案 |
| 任务规划 | 当前开发状态、协作日志、HANDOVER | 重复已完成的课程/任务 |

#### 3.3.2 Markdown 读取原则

当用户无法直接上传项目文件时，AI 应通过开发代理（Claude Code / OpenCode 等）执行文件读取任务，遵循以下标准流程：

1. **AI 提供执行提示词** — 明确指定需要读取的文件路径和关注点
2. **Agent 在项目目录执行读取** — 使用实际读取工具扫描相关文件
3. **Agent 输出结构化结果**：
   - 文件结构概览
   - 相关章节的完整内容或摘要
   - 当前设计状态与约束条件
   - 已存在的架构决策
   - 潜在的冲突点

**结果质量标准：**

- 输出应足够详细，使 AI 能基于真实文档做出判断
- 不应仅输出文件名列表，必须输出具体内容
- 对于长文档，优先读取目录结构和关键章节
- 标注每个信息的来源文件路径

#### 3.3.3 架构判断原则

在提出任何架构修改建议之前，必须确认以下文档以**避免重复设计、与已有决策冲突、或修改已确定的架构原则**：

- [ ] 当前 ADR 决策（`04_ADR_产品决策记录.md`）
- [ ] 当前 API 契约（`13_API接口契约设计.md`）
- [ ] 当前架构设计文档（`08_系统架构设计.md`）
- [ ] 当前开发状态（`03_当前开发状态.md`）
- [ ] 已完成任务列表（`11_AI协作日志.md`）

**禁止行为：**

- ❌ 在未读取 ADR 的情况下提出新决策
- ❌ 在未读取 API 契约的情况下设计新接口
- ❌ 在未读取开发状态的情况下分配任务优先级
- ❌ 在未读取协作日志的情况下重复已完成的课程

**建议做法：**

- ✅ 明确声明所参考的文档："基于 ADR-019（双通道通信）和 13_API 契约 3.8 节..."
- ✅ 发现已有决策与当前需求不符时，先记录质疑再讨论修改
- ✅ 新设计与已有设计的差异部分需显式说明

#### 3.3.4 AI 交接信息标准

当 AI 执行文件读取和分析任务后，返回结果时应包含以下结构化信息：

| # | 信息项 | 说明 |
|---|--------|------|
| 1 | **修改文件列表** | 本次任务创建/修改了哪些文件 |
| 2 | **修改原因** | 每项修改的必要性与背景 |
| 3 | **影响范围** | 修改对现有架构、文档、代码的影响分析 |
| 4 | **一致性检查结果** | 修改是否与 ADR / API 契约 / 架构设计一致 |
| 5 | **下一步建议** | 基于当前完成状态，建议后续优先级 |

**输出示例：**

```
修改文件列表：
- docs/14_WebSocket实时通信协议设计.md（修改：3.1节路径定义）

修改原因：
- 审计发现 3 路径方案对 V1 桌面 APP 过度设计，合并为单路径简化客户端实现

影响范围：
- 客户端 WS 连接管理器从 3 路并发改为 1 路
- 服务端路由层保持不变（按 type 分发）

一致性检查：
- ✅ 与 ADR-019 双通道架构一致
- ✅ 与 13_API 契约 3.8 节一致

下一步建议：
- 可进入 Phase 2 第72课
```

## 第四章：阶段性审计机制

### 4.1 审计触发条件

以下情况必须执行阶段审计：

- **Phase 阶段结束** — 一个完整的开发阶段完成时
- **架构设计完成** — 系统架构方案初步定稿时
- **API 契约完成** — 接口契约设计完成后
- **数据模型完成** — 数据库 Schema 或核心数据模型完成后
- **技术选型完成** — 关键技术方案选型确认后
- **进入代码开发前** — 从设计阶段进入编码阶段之前
- **重大架构调整后** — 对已确认的架构方向做出修改后

### 4.2 审计目标

审计不是为了修改代码，而是验证以下维度：

| 维度 | 验证内容 |
|------|----------|
| **架构一致性** | 当前设计与已确认的 ADR 是否一致 |
| **文档一致性** | 各文档之间的引用和决策是否相互矛盾 |
| **ADR 与实现方向一致性** | 实现是否遵循了 ADR 确定的方向 |
| **API 契约完整性** | 接口设计是否覆盖了所有必要的交互场景 |
| **数据模型合理性** | 数据实体和关系是否足够支持业务需求 |
| **AI 协作流程完整性** | 协作日志是否完整、规范是否得到遵守 |
| **阶段进入条件** | 是否具备进入下一阶段的条件，存在哪些阻止因素 |

### 4.3 审计原则

#### 4.3.1 文件优先原则

审计必须基于真实项目文件。审计者在执行审计时：

- **必须**读取相关的 Markdown 文档、ADR、API 契约、架构设计等真实文件
- **禁止**仅根据聊天记录或对话上下文判断项目状态
- **禁止**仅根据用户提供的摘要或口头描述做出架构判断
- **禁止**在未读取相关 Markdown 文件的情况下做任何架构或设计层面的修改

#### 4.3.2 只审计，不随意修改

在审计阶段，审计者的职责是**发现和报告问题**，而非直接修复。

**禁止行为：**

- ❌ 未经项目负责人确认修改 ADR
- ❌ 未经确认改变已确认的架构方向
- ❌ 自动重构核心设计或 API 契约
- ❌ 在审计报告中混入代码修改

**发现问题后必须按以下流程处理：**

```
发现问题 → 分类（架构/文档/API/数据模型/流程）
        → 分级（P0/P1/P2）
        → 提出修复建议
        → 在审计报告中记录
        → 等待项目负责人确认后再执行修复
```

### 4.4 审计等级

审计发现的问题按以下三级分类：

#### P0 — 阻止级 🔴

必须解决，否则禁止进入下一阶段。

典型场景：

- 架构冲突 — 两个或多个 ADR 之间的决策相互矛盾
- API 契约冲突 — HTTP 接口与 WebSocket 协议的数据模型不一致
- 数据模型无法实现 — 当前设计在技术上有不可行的部分
- 核心技术路线未确定 — 阻碍编码开始的关键决策尚未做出
- 跨文档 P0 级矛盾 — 字段名、路径、数据格式等实现层面的直接冲突

#### P1 — 优先级 🟡

建议解决后进入下一阶段，但紧急度低于 P0。

典型场景：

- 文档遗漏 — ADR 已确定但相关设计文档未更新
- 设计细节不足 — 方案方向正确但缺少实现所需的详细规格
- 扩展性问题 — 当前设计对预期的未来扩展缺乏考虑
- 边界情况未覆盖 — 异常流程、错误处理等非主路径的设计缺失

#### P2 — 优化级 🟢

可以后续优化，不影响阶段进入。

典型场景：

- 命名优化 — 接口名、字段名、路径名的可读性改进
- 示例完善 — API 文档的 JSON 示例补充
- 文档格式调整 — Markdown 排版、表格对齐、交叉引用补充
- 注释和说明补充 — 不影响理解但可提升可维护性的补充

### 4.5 审计报告格式

每次阶段审计必须生成独立的 Markdown 审计报告，文件命名规则为：

```
docs/<编号>_Phase<X>架构审计报告.md
```

例如：

- `docs/15_Phase2架构审计报告.md`

报告至少包含以下 10 个章节：

| # | 章节 | 内容说明 |
|---|------|----------|
| 1 | 审计目的 | 本次审计的背景、触发原因和目标 |
| 2 | 审计范围 | 审计覆盖的文档、模块、决策范围 |
| 3 | 当前项目状态 | 审计时的项目阶段、文档状态、ADR 列表 |
| 4 | 架构一致性分析 | ADR 与架构设计的交叉核对结果 |
| 5 | API 与通信协议分析 | HTTP 和 WebSocket 接口的一致性审查 |
| 6 | 数据模型准备度 | 数据实体覆盖度、Schema 设计就绪度 |
| 7 | 风险清单 | 按 P0/P1/P2 分级列出所有发现的问题 |
| 8 | 开发准备度评分 | 多维度评分（总分 100 分）及评分说明 |
| 9 | 下一阶段进入建议 | YES / NO / Conditional 判断及前置条件 |
| 10 | 附录 | 审计输入文档列表、方法说明、术语对照 |

### 4.6 阶段门规则

进入项目下一阶段必须经过以下正式流程：

```
阶段完成 → 执行审计 → 风险评估 → 输出报告 → 项目负责人确认 → 进入下一阶段
```

**核心规则：**

- **审计是必需环节**，不可跳过
- **风险评估结果必须有正式文档**（审计报告）
- **项目负责人确认是最终决策者**

**P0 问题阻断规则：**

> 如果审计报告中存在 **P0（阻止级）问题**，则 **禁止进入下一阶段**。
>
> 必须先解决所有 P0 问题，通过二次审计确认后，方可进入。

**Conditional 进入规则：**

> 如果审计结果为 **Conditional（有条件进入）**，项目负责人需确认前置条件清单。
>
> 前置条件可以分阶段完成，但每个条件的完成状态需在协作日志中记录。
>
> 所有前置条件完成后，执行一次快速确认审计，确认无误后进入下一阶段。

**阶段门记录：**

每次阶段门审查需在 `docs/11_AI协作日志.md` 中记录以下信息：

- 审计结论（总分 + 推荐意见）
- P0 问题数量及简述
- 前置条件清单（如有）
- 项目负责人确认时间

### 4.2 文档状态定义

| 状态 | 含义 | 谁可以修改 |
|------|------|------------|
| ✅ Approved | 已确认，不可直接修改 | 需通过 ADR 变更流程 |

| ✅ Draft | 草稿，可自由修改 | AI 和产品负责人均可 |
| 🟡 Draft（待审核） | 等待产品负责人确认 | 产品负责人 |
| 🔴 Deprecated | 已废弃 | 产品负责人 |

### 4.3 ADR 管理规则

#### 4.3.1 新增 ADR

1. 在 `docs/04_ADR_产品决策记录.md` 末尾按编号递增添加
2. 模板格式（保持与现有 ADR 一致）：
   ```markdown
   ## ADR-XXX：决策标题

   | 字段 | 内容 |
   |------|------|
   | **决策编号** | ADR-XXX |
   | **决策日期** | YYYY-MM-DD |
   | **状态** | ✅ Draft |
   | **决策者** | 产品负责人 |

   **Context（背景）：**
   ...

   **Decision（决策）：**
   ...

   **Consequences（选择理由）：**
   ...
   ```
3. 新增后同步更新附录的「ADR 状态索引」表格
4. 在 `11_AI协作日志.md` 中记录

#### 4.3.2 修改 ADR

- 已 ✅ Approved 的 ADR **不得直接修改**
- 如果发现已确认决策存在重大问题，按以下流程：
  1. 在 `04_ADR_产品决策记录.md` 附录的「质疑记录」区域记录质疑内容
  2. 在 `11_AI协作日志.md` 中记录建议
  3. 等待产品负责人确认是否修改
- 产品负责人同意修改后，由产品负责人本人修改
- ✅ Draft 状态的 ADR 可在讨论后直接更新

#### 4.3.3 ADR 与代码的关系

- 代码实现必须遵循已确认的 ADR
- 开发过程中如果发现 ADR 与实际情况冲突，先提出 ADR 修改建议，再调整代码
- 不允许"先改代码，再补 ADR"的逆向操作

### 4.4 文档交叉引用规范

- 文档之间引用使用相对路径：`[项目简介](./01_项目简介.md)`
- 引用 ADR 使用编号：`ADR-009（八字分析6步流程）`
- 引用代码使用 `文件路径:行号` 格式

## 第六章：代码修改规则

### 6.1 修改前的准备

- [ ] 已读取项目相关文档（P0 文档必须全覆盖）
- [ ] 已了解项目文件结构
- [ ] 已确认修改范围不会影响未涉及的模块
- [ ] 已确认修改不违反任何已确认的 ADR
- [ ] 已确认修改不涉及【待确认】内容（如涉及，需先确认）

### 6.2 修改中的纪律

| 规则 | 说明 |
|------|------|
| **最小范围** | 只修改任务指定的文件，不修改无关文件 |
| **最小改动** | 在一个文件内只改必须改的行，不做"顺手优化" |
| **单一职责** | 一次执行只做一件事，不混入其他修改 |
| **保持风格** | 新代码的注释密度、命名风格、格式与周围代码一致 |
| **边改边记** | 记录修改的文件列表、行范围、修改理由 |
| **不破坏** | 不删除未涉及的函数/变量/文件，哪怕看起来"没用" |

### 6.3 修改后的自检清单

- [ ] 所有修改是否在计划范围内？
- [ ] 语法无错误（可以编译/解释通过）？
- [ ] 是否破坏了任何现有功能？
- [ ] 是否有硬编码值需要改为配置？
- [ ] 是否有敏感信息（API Key、密钥）泄露风险？
- [ ] 是否有遗漏的边界情况？

### 6.4 删除文件的规则

- **禁止在未确认的情况下删除任何文件**
- 如需删除文件，必须：
  1. 在协作日志中明确列出待删除文件及理由
  2. 等待产品负责人明确确认
  3. 确认后先备份再删除

### 6.5 Git 提交规则

- **禁止自动提交 git**（严格遵守本条款）
- 必须先将 diff 展示给产品负责人确认
- 确认后方可执行提交
- 提交信息格式：
  ```
  <type>: <简要描述>

  - 修改文件：列出文件列表
  - 关联 ADR：ADR-XXX
  - 修改理由：一句话说明

  Co-Authored-By: Claude <noreply@anthropic.com>
  ```

## 第七章：测试与验收规则

### 7.1 测试级别定义

| 级别 | 说明 | 适用阶段 |
|------|------|----------|
| 单元测试 | 测试单个函数/模块的正确性 | 开发中 |
| 集成测试 | 测试多模块交互的正确性 | 模块完成后 |
| 人工验收 | 产品负责人手动验证功能 | 功能完成后 |
| 边界测试 | 测试异常/边界输入的处理 | 开发中 |

### 7.2 自检要求

每次修改后（无论是代码还是文档），执行以下自检：

```
1. 完整性检查：所有计划中的修改是否已完成？
2. 正确性检查：修改的内容是否正确？
3. 一致性检查：修改是否与现有风格/规范一致？
4. 关联性检查：是否更新了相关的文档、日志？
5. 回归检查：修改是否破坏了未涉及的功能？
```

### 7.3 验收标准

产品负责人验收时关注以下维度：

| 维度 | 说明 | 通过标准 |

### 4.1 ✅ ADR-020：Approved

| 字段 | 内容 |
|------|------|
| **决策编号** | ADR-020 |
| **决策名称** | V1 PySide6 与 Vue 集成方案 |
| **审批结果** | ✅ **Approved** |
| **审批日期** | 2026-07-10 |

**审批理由：**

1. **技术路线合理** — QWebEngineView 嵌入式集成方案是 PySide6 生态中最成熟、最自然的 Web 嵌入方式，无需引入额外的 IPC 层或独立窗口管理复杂度。
2. **用户体验统一** — 单进程、单窗口架构保证了桌面 APP 的使用体验连贯性，避免独立窗口带来的焦点管理、任务栏管理等问题。
3. **未来迁移路径清晰** — Vue 模块经 QWebEngineView 嵌入仅影响 V1 桌面壳，未来 Web 扩展时 Vue 代码可直接复用（ADR-018 §8.3.4），无锁定风险。
4. **职责分离充分** — WebChannel（宿主集成：窗口事件/生命周期）与 HTTP/WS（业务通信：AI 对话/分析）的双层通信架构职责清晰，互不耦合。
5. **成本已评估** — Chromium 捆绑体积（~100MB+）和内存开销在 ADR-018 决策时已纳入考量，作为 V1 桌面 APP 的第一性体验，该代价可接受。

**附带条件：**
- V1 初期以 PySide6 原生控件起步，Vue 按需渐进引入（呼应 ADR-018 后续影响第 1 条）
- 实现阶段需关注 WebChannel 线程安全（Python 主线程与 JS 引擎的调用）
