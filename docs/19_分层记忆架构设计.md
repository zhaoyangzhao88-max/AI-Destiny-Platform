# 19 — 分层记忆架构设计（Layered Memory Architecture）

> **版本:** v0.1 | **日期:** 2026-07-08 | **状态:** Draft  
> **关联 ADR：** ADR-012（数据三层体系）、ADR-016（单核心知命 AI Agent 架构）、ADR-019（双通道通信）  
> **关联风险：** docs/18 ARC-15-P0-002（AI 上下文窗口管理完全缺失）  
> **用途：** 定义知命 AI 在长期对话中的上下文管理架构，解决 LLM 上下文窗口（32K~128K tokens）限制下的历史保留与质量保障问题。本文档为**设计契约**，不进入编码阶段。

## 1. 问题背景

知命 AI 采用单核心 Agent 架构（ADR-016），与用户进行长期、有温度的多轮对话。但当前 LLM 存在上下文窗口硬限制（32K~128K tokens）：

| 风险 | 后果 |
|------|------|
| 历史对话超出窗口 | 早期信息被截断，AI"遗忘"用户已陈述的关键事实（出生信息、人生事件） |
| 无压缩机制 | 为保留历史而无限堆积 token，触发截断或质量严重下降 |
| 无持久化用户画像 | 每次会话从零开始，无法累积"认识用户"的长期价值 |

> **产品影响：** 长期 AI 对话中的历史丢失属于**不可接受的产品缺陷**（见 docs/15 §7.1 P0#2）。本设计通过分层记忆架构解决。

---

## 2. 设计目标与约束

| 目标 | 说明 | 衡量 |
|------|------|------|
| **历史不丢失** | 用户确认的事实跨会话长期可用 | 核心记忆命中率 100% |
| **窗口可控** | 注入模型的上下文稳定在窗口安全水位内 | 工作记忆 ≤ 窗口 70% |
| **质量不退化** | 长对话中 AI 角色语气、事实一致性保持 | 不出现"忘记之前说过的话" |
| **本地优先** | 记忆存储遵循 ADR-014 本地优先（隐私、离线） | 核心记忆本地持久化 |
| **可演进** | 未来多 Agent 拆分时记忆层可独立服务 | 记忆接口与 Agent 解耦 |

**安全水位约定（设计值，待实现调优）：**

- 上下文窗口使用上限：**70%**（留出推理余量）
- 工作记忆默认保留：最近 **20 轮**对话 + 当前任务状态
- 摘要记忆粒度：按**会话**与按**主题**双维度

## 3. 三层记忆模型

```
┌─────────────────────────────────────────────────────────────┐
│                  知命 AI — 分层记忆架构                       │
│                                                             │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ ① 工作记忆 Working Memory（常驻上下文窗口）            │  │
│  │  · 当前会话最近 N 轮对话                              │  │
│  │  · 当前任务状态 / 进行中分析                          │  │
│  │  · 用户短期意图（本轮想知道什么）                     │  │
│  │  来源：实时会话 + 摘要记忆注入                         │  │
│  └───────────────────────┬──────────────────────────────┘  │
│                          │ 超阈值时旧轮次下沉                  │
│                          ▼                                   │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ ② 摘要记忆 Summary Memory（压缩的中间层）              │  │
│  │  · 历史会话的周期压缩摘要                              │  │
│  │  · 按会话 / 按主题的阶段性结论                         │  │
│  │  · 关键决策点、已确认事实的摘要标记                    │  │
│  │  来源：工作记忆压缩产生；持久化到本地档案              │  │
│  └───────────────────────┬──────────────────────────────┘  │
│                          │ 稳定事实沉淀                      │
│                          ▼                                   │
│  ┌──────────────────────────────────────────────────────┐  │
│  │ ③ 核心记忆 Core Memory（跨会话长期画像）               │  │
│  │  · 用户基础档案（出生信息、人生事件）— ADR-012 事实层  │  │
│  │  · 用户偏好 / 角色关系（与知命的相处模式）             │  │
│  │  · 已确认的命理结论要点                               │  │
│  │  来源：ADR-012 三层数据体系；本地档案持久化            │  │
│  └──────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

**三层关系：** 工作记忆是"当前在想什么"，摘要记忆是"我们聊过什么（压缩版）"，核心记忆是"这个人是谁"。每次对话从核心记忆 + 摘要记忆中按需注入工作记忆。

## 4. 各层定义与数据契约

### 4.1 工作记忆（Working Memory）

| 字段 | 类型 | 说明 |
|------|------|------|
| `recent_turns` | array | 最近 N 轮 `{role, content, turn_id, timestamp}` |
| `active_task` | object/null | 当前进行中的分析任务状态（引用 Task 对象，见 14_WS 4.3） |
| `short_term_intent` | string | 本轮用户短期意图摘要 |
| `window_usage` | float | 当前 token 占用率（用于触发压缩） |

### 4.2 摘要记忆（Summary Memory）

| 字段 | 类型 | 说明 |
|------|------|------|
| `summary_id` | string | 摘要标识 |
| `scope` | enum | `session` / `topic` |
| `ref_session_id` | string/null | 关联会话（scope=session） |
| `topic` | string/null | 主题（scope=topic） |
| `summary` | string | 压缩后的自然语言摘要 |
| `key_facts` | array | 摘要中提取的关键事实（指向核心记忆） |
| `created_at` | string | ISO 8601 |

### 4.3 核心记忆（Core Memory）

直接复用 ADR-012 数据三层体系中的**事实（Fact）层**与用户偏好：

| 来源 | 说明 |
|------|------|
| `users.me.profile.basic_info` | 出生信息、性别、出生地（ADR-012 Fact） |
| `users.me.profile.life_events[data_tier=fact]` | 用户确认的人生事件 |
| `users.me.profile.personality` | 性格画像、数据层级 |
| 用户偏好（config/preferences） | 语言、分析深度等 |

## 5. 写入与读取策略

### 5.1 写入

| 事件 | 动作 |
|------|------|
| 用户发送消息 / AI 回复 | 追加到工作记忆 `recent_turns` |
| 用户确认事实（ADR-012 流转） | 同步写入核心记忆（本地档案，永不覆盖，版本历史见 ADR-013） |
| 工作记忆超阈值 | 触发压缩：旧轮次 → 摘要记忆（§8） |
| 会话结束 | 全量压缩为一条会话摘要；稳定事实沉淀核心记忆 |

### 5.2 读取（上下文注入顺序）

每次构造发给 LLM 的上下文时，按以下优先级拼接（均在窗口安全水位内）：

```
[系统提示词]
  + [核心记忆：用户画像 + 已确认事实]   ← 始终注入
  + [摘要记忆：相关主题/历史会话摘要]   ← 按相关性注入
  + [工作记忆：最近 N 轮]              ← 始终注入
  + [当前用户消息]
```

### 5.3 读取优先级

1. 核心记忆命中用户明确事实 → 优先于任何摘要/推测
2. 摘要记忆提供历史连贯性 → 避免重复提问
3. 工作记忆保证本轮连续性 → 当前对话不脱节

## 6. 与 ADR-012 数据三层体系的映射

| 记忆层 | 对应 ADR-012 层级 | 说明 |
|--------|------------------|------|
| 核心记忆 | **事实（Fact）** | 用户确认的真实信息，最高可信度 |
| 摘要记忆 | **推测（Inference）的中间态** | 压缩结论可能含 AI 推论，需标注 |
| 工作记忆 | 不属于持久化三层 | 临时会话态，不持久化到档案 |

**一致性规则：** 任何写入核心记忆的内容必须经过 ADR-012 数据层级流转（用户确认 → Fact），禁止将未确认的 AI 摘要直接写入核心记忆的事实层。

---

## 7. 与 WebSocket / 上下文注入的关系

- 实时对话经 ADR-019 双通道：`/ws/v1/session` 承载 `chat.message` / `ai.token`（14_WS 4.2）。
- 记忆层对 WebSocket **透明**：上下文注入发生在核心服务（Python 知命 Agent 编排，08_架构 8.2.3）内部，客户端仅收发消息，不感知记忆分层。
- `sequence` 机制（14_WS 4.1）保障断线重连后消息有序，记忆层据此重建工作记忆的 `recent_turns`。
- 断线恢复（`task.resume` / `task.list`，14_WS 6.2）复用同一会话上下文，记忆层不丢失。

## 8. 压缩与摘要生成策略

| 策略 | 触发 | 方法 |
|------|------|------|
| **滚动压缩** | 工作记忆 `window_usage` ≥ 70% | 将最旧的一批轮次（如最早 5 轮）交由 LLM 压缩为一段摘要，移入摘要记忆 |
| **会话级压缩** | 会话结束 / 用户主动"总结一下" | 全量压缩为会话摘要，提取 `key_facts` 指向核心记忆 |
| **主题聚合** | 多会话出现同一主题 | 合并为 `scope=topic` 摘要，避免重复 |
| **去重** | 摘要与核心记忆重叠 | 事实类以核心记忆为准，摘要仅保留叙述上下文 |

**质量保障：** 压缩摘要必须保留可回溯指针（原始 `turn_id` / `session_id`），用户或 AI 可回溯原始对话（呼应 ADR-013 版本历史"可回溯"原则）。

## 9. 风险与开放问题

| # | 问题 | 状态 | 说明 |
|---|------|------|------|
| 1 | 压缩导致的细节丢失 | 待实现验证 | 滚动压缩可能丢失微妙语气，需调优保留策略 |
| 2 | 摘要与事实冲突 | 由 ADR-012 层级规则约束 | 事实层优先，摘要不得覆盖 |
| 3 | 窗口水位阈值 | 设计值待调 | 70% / 20 轮为初值，实现阶段按模型调优 |
| 4 | 多 Agent 演进 | 未来（ADR-016） | 记忆层需作为独立服务可被多 Agent 共享 |
