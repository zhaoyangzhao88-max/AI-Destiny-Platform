## 8.1 V1 总体系统架构设计

> **来源：** ADR-014、ADR-015、ADR-016

### 8.1.1 产品整体架构思想

知命 AI 的 V1 架构设计围绕三个核心原则展开：

1. **隐私优先（Privacy First）：** 用户数据归属用户，核心档案本地存储，云端仅作算力扩展（ADR-014）
2. **平台聚焦（Platform Focus）：** V1 集中资源做好 Windows 桌面端体验（ADR-015）
3. **简单演进（Simple Evolution）：** 从单 Agent 起步，预留扩展路径，不早期过度设计（ADR-016）

```
┌─────────────────────────────────────────────────────────┐
```

### 8.1.2 本地优先原则

**核心思想：** 能本地做的，就不上云。

| 能力 | 处理位置 | 原因 |
|------|----------|------|
| 四柱排盘 | ✅ 本地 | 纯计算，无网络依赖 |
| 用户档案存储 | ✅ 本地 | 数据隐私，离线可用 |
| 基础命理分析 | ✅ 本地 | 规则可固化，无需大模型 |
| 对话界面渲染 | ✅ 本地 | 原生体验 |
| AI 深度分析 | ☁️ 云端按需 | 需要 LLM 大算力 |
| 真太阳时校准 | ☁️ 云端按需 | 需要地理位置数据库 |

**离线分级策略：**

| 网络状态 | 可用功能 |
|----------|----------|
| 🟢 在线 | 全部功能（含 AI 深度分析） |
| 🟡 有限连接 | 排盘、档案浏览、基础分析 |
| 🔴 离线 | 排盘、档案浏览 |

**离线策略 5 维度细化（决策结论：方案 A — 3 级基础细化，V1 采用，ARC-15-P1-004 已确认）：**

保持现有 3 级分层（Online / Limited / Offline），为每一级明确定义五个维度：

| 维度 | Online | Limited | Offline |
|:----:|--------|---------|---------|
| **队列策略** | 无需排队，请求直达 | 写操作自动进入内存队列（最大 50 条）；队列内容持久化至本地 SQLite；读操作优先尝试网络，失败回退缓存 | 所有写操作进入本地队列（内存 + SQLite 持久化）；读操作全部走本地缓存 |
| **重试策略** | 不适用 | 网络恢复后自动触发 FIFO 回放，每条队列间隔 500ms；单条失败最多重试 3 次（间隔 2s / 5s / 10s），全部失败后标记为待处理 | 同 Limited 重试策略，网络恢复后统一回放 |
| **冲突解决规则** | 不适用（始终同步最新数据） | **本地优先**：本地修改时间戳晚于云端则覆盖云端，反之忽略；同步冲突提示写入操作日志不影响当前操作 | 同 Limited |
| **API 端点可用性** | 全部 API 端点可用 | 排盘、档案读写、本地规则分析 → 可用；AI 深度分析（LLM 调用）→ 排队，网络恢复后自动执行；云端资源查询 → 降级为本地缓存 | 仅排盘和档案浏览（本地缓存） → 可用；所有远端 API → 不可用，自动排队 |
| **UI 指示器** | 顶部状态栏绿色圆点 + "已连接" | 顶部状态栏黄色圆点 + "有限连接"，队列中有待发送项时显示小数字角标；首次进入 Limited 时 toast 提示"部分功能暂不可用，网络恢复后自动同步" | 顶部状态栏灰色圆点 + "离线模式"，底部工具栏提示"离线模式 - 仅可浏览已有档案" |

> 与认证模型的协同：本地操作（排盘、档案浏览、本地分析）在任意网络状态下均无需云端 Token 即可使用，详见 `13_API接口契约设计.md` §4.5.1 设备绑定简化认证模型。

> **后续升级方向（V2 及以上）：** 若用户反馈显示"有限连接"状态体验不足，可参照方案 B（5 级）将 Limited 拆分出 Degraded / Minimal 子级，或在 API 可用性矩阵中增加端点级别的降级策略。

### 8.1.3 云端扩展原则

云端在 V1 中的角色被严格限定：

1. **AI 推理扩展：** 仅当用户发起深度分析时，将必要的上下文（脱敏）发送至 LLM API
2. **外部数据服务：** 如真太阳时校准需要地理位置数据库
3. **未来扩展预留：** 云备份、多设备同步为后期功能，V1 不做

**数据上传最小化规则：**
- 只上传当前分析所需的最小上下文
- 不上传完整人生档案
- 所有上传数据标注临时性，分析完成后不持久化在云端

### 8.1.4 Windows APP 作为 V1 入口

V1 以 **Windows 桌面应用** 为首发平台（ADR-015）：

**选型考量因素：**

| 方案 | 跨平台 | 本地能力 | 学习成本 | 决策状态 |
|------|--------|----------|----------|---------|
| PySide6 + Vue（混合架构） | Windows V1（Web V2）| ✅ 强（文件系统/SQLite/系统集成） | 中（需同时掌握 PySide6 + Vue）| ✅ 已决策（ADR-018）|

> ✅ **技术框架已决策**：PySide6 + Vue 渐进式混合架构（ADR-018，2026-07-08）  
> 详情见 [8.3 客户端技术架构选择](#83-v1客户端技术架构选择)  
> 候选方案（Tauri/Electron/Flutter/C#）已移入附录以供参考

**Windows APP 作为首发平台的优势：**
- 本地资源访问无限制（文件系统、SQLite 等）
- 离线运行能力自然支持
- 与 Windows 生态集成（通知、任务栏、系统托盘）
- 适合"本地优先"架构落地

### 8.1.5 AI Agent 总体设计

V1 采用 **单核心知命 AI Agent 架构**（ADR-016）：

```
┌──────────────────────────────┐
│       知命 AI Agent          │
│                              │
│  ┌────────────────────────┐  │
│  │   核心对话引擎          │  │
│  │   - 上下文管理          │  │
│  │   - 对话状态跟踪        │  │
│  └────────┬───────────────┘  │
│           │                  │
│  ┌────────┴───────────────┐  │
│  │   能力路由              │  │
│  │   ┌─────┐ ┌─────┐ ┌──┐ │  │
│  │   │命理  │ │档案  │ │… │ │  │
│  │   │分析  │ │管理  │ │  │ │  │
│  │   └─────┘ └─────┘ └──┘ │  │
│  └────────────────────────┘  │
│                              │
│  ┌────────────────────────┐  │
│  │   工具/插件接口         │  │
│  │   (预留多 Agent 扩展)   │  │
│  └────────────────────────┘  │
└──────────────────────────────┘
```

**Agent 内部能力划分：**
- **对话管理：** 维护与用户的长期对话上下文，管理知命角色的语气与风格（长期上下文采用分层记忆架构，详见 `docs/19_分层记忆架构设计.md`）
- **命理分析：** 调用排盘引擎结果，结合 LLM 进行八字分析
- **档案管理：** 读取/更新用户人生档案，处理三层数据体系
- **工具调用：** 根据需要调用外部工具（排盘库、LLM API 等）

**未来向多 Agent 演进的预留设计：**
- Agent 内部按模块组织 Prompt，方便未来拆分为独立 Agent
- 工具 / 插件接口标准化
- 日志和监控按能力维度分开记录

### 8.1.6 模块化架构思想

所有模块遵循以下设计原则：

1. **高内聚低耦合：** 模块间通过定义良好的接口通信，不直接依赖内部实现
2. **本地替换优先：** 每个模块优先考虑本地实现，云端仅为增强选项
3. **可测试：** 模块边界清晰，支持独立单元测试
4. **渐进增强：** 基础功能可离线运行，联网自动启用增强能力

### 8.2.2 Windows APP 职责

Windows APP 专注于**表示层**职责，不直接处理业务逻辑：

| 职责 | 说明 | 是否涉及业务逻辑 |
|------|------|-----------------|
| **UI 渲染** | 所有用户界面展示（对话、排盘、档案） | ❌ 纯展示 |
| **用户交互** | 输入采集、点击响应、手势控制 | ❌ 纯事件 |
| **本地 IPC 调用** | 将用户请求转发至 Python 核心服务 | ❌ 纯转发 |
| **数据缓存** | 本地 UI 状态缓存（不缓存核心数据） | ❌ 纯缓存 |
| **离线状态管理** | 监控连接状态，展示离线/在线指示器 | ❌ 纯状态 |
| **安装/更新管理** | APP 更新、核心服务版本管理 | ⚠️ 仅版本检查 |
| **系统集成** | 通知、任务栏、系统托盘 | ❌ 纯集成 |

**Windows APP 不直接处理：**
- 排盘计算（转发至核心服务）
- AI 分析（转发至核心服务）
- 档案数据持久化（转发至核心服务）
- 命理规则推理（转发至核心服务）

### 8.2.3 Python 核心服务职责

Python 核心服务作为**独立的本地进程**，承载所有业务逻辑：

| 职责 | 说明 | 依赖 |
|------|------|------|
| **排盘引擎** | 四柱排盘、真太阳时校准、大运流年计算 | 排盘库（本地） |
| **AI 分析引擎** | 八字分析编排、LLM 提示词管理、分析结果结构化 | LLM API（云端） |
| **知命 Agent 编排** | 对话上下文管理、能力路由、多轮交互状态跟踪 | 内部模块 |
| **档案管理系统** | 三层数据体系存储、版本历史管理、数据层级流转 | 本地数据库 |
| **规则推理引擎** | 命理规则库查询、逻辑推理、组合规则匹配 | 规则定义（本地） |
| **外部服务网关** | LLM API 调用管理、超时处理、重试策略、降级方案 | 云端 API |
| **配置管理** | 服务配置、用户偏好、模型选择 | 配置文件 |

**Python 核心服务不处理：**
- UI 渲染（由 APP 负责）
- 用户事件处理（由 APP 负责）

### 8.2.4 客户端与业务逻辑边界

**API 契约定义：**

```
┌─────────────────────┐        IPC 请求        ┌──────────────────────┐
│  Windows APP        │ ──────────────────────→ │  Python 核心服务     │
│  (表示层)            │                        │  (业务逻辑层)         │
│                     │ ←────────────────────── │                      │
│  - 仅处理 UI 状态    │        IPC 响应        │  - 所有业务计算       │
│  - 不处理业务逻辑    │                        │  - 数据持久化         │
│  - 不直接访问数据库  │                        │  - 外部 API 调用      │
└─────────────────────┘                        └──────────────────────┘
```

**边界规则：**

1. **APP 不直接访问数据库** — 所有数据操作通过核心服务的 API 进行
2. **APP 不包含命理分析逻辑** — 排盘、推理、分析全部委托给核心服务
3. **APP 不直接调用 LLM** — AI 分析经由核心服务统一调度
4. **核心服务不包含 UI 代码** — 不产生任何用户界面输出
5. **通信协议标准化** — 所有 API 定义在契约文档中（OpenAPI / protobuf）

**典型请求流程（以"八字分析"为例）：**

```
用户输入出生信息
    │
    ▼
Windows APP 采集输入 → 序列化请求 → 通过 IPC 发送至核心服务
    │
    ▼
Python 核心服务接收请求 → 调用排盘引擎 → 生成四柱八字
    │
    ▼
核心服务调用 AI 分析引擎 → 构造 Prompt → 调用 LLM API
    │
    ▼
LLM API 返回结果 → 核心服务结构化分析结果 → 存入档案数据库
    │
    ▼
核心服务将结果序列化为 IPC 响应 → 返回至 APP
    │
    ▼
Windows APP 接收响应 → 解析 → UI 渲染展示
```

### 8.3.5 PySide6 × Vue 集成机制

> **来源：** ADR-020

基于 ADR-018（PySide6 + Vue 混合架构）与 ADR-017（客户端解耦），本小节明确 PySide6 与 Vue 的**集成方案**（ADR-018 后续影响第 3 条遗留的专项决策）：

**V1 采用 QWebEngineView 嵌入式集成：**

1. **渲染方式** — Vue 应用作为 PySide6 `QWebEngineView` 的内嵌网页渲染，Vue 构建产物以本地静态资源或本地 HTTP 服务（localhost）提供。
2. **宿主集成通道** — PySide6 与内嵌 Vue 之间经 **Qt WebChannel** 双向调用（窗口事件、通知、进程生命周期）。
3. **业务通信通道** — Vue 业务交互遵循 ADR-019，通过 **HTTP API + WebSocket（`/ws/v1/session`）** 调用本地 Python 核心服务。
4. **双层通信职责** — WebChannel = 宿主集成（窗口级），HTTP/WS = 业务通信（服务级），互不耦合。

```
Windows APP（PySide6 主窗口）
  ├── 原生控件区（通知 / 任务栏 / 进程管理）
  └── QWebEngineView（内嵌 Vue 模块，按需引入）
        │  Qt WebChannel（宿主集成）
        └── Vue 业务页（AI 聊天 / 报告 / 可视化）
              │  HTTP API + WebSocket /ws/v1/session（ADR-019）
              ▼
        Python 核心服务（本地）
```

**渐进引入原则：** V1 初期以 PySide6 原生控件起步，Vue 仅在复杂交互场景按需经 `QWebEngineView` 嵌入（ADR-018 后续影响第 1 条）。V1 Windows APP 技术框架已由 ADR-018 决策为 PySide6 + Vue 渐进式混合架构，不再开放选型讨论。

## 8.5 数据三层体系

**来源：** ADR-012

### 层级定义

| 层级 | 英文 | 定义 | 存储标记 | 可信度 |
|------|------|------|----------|--------|
| **事实** | Fact | 用户明确确认的真实信息 | `confidence: fact` | ⭐⭐⭐ |
| **推测** | Inference | AI 基于命理规则的分析推论 | `confidence: inference` | ⭐⭐ |
| **待验证** | Pending | AI 提出的假设，需用户反馈 | `confidence: pending` | ⭐ |

### 层级流转规则

```
待验证 ←→ 推测 ←→ 事实
  │          │        │
  ▼          ▼        ▼
未确认    部分确认   用户确认
```

- **待验证 → 推测：** AI 反复确认后仍无法获得用户反馈，但命理分析支持
- **推测 → 事实：** 用户明确确认
- **推测 → 待验证：** 用户否认，但 AI 认为有其他可能性
- **事实不可逆：** 用户确认的事实不能自动降级，修改需通过版本历史

## 8.6 数据版本历史机制

**来源：** ADR-013

### 核心原则

1. **永不覆盖：** 修改用户资料时保留完整历史副本
2. **可回溯：** 用户可以查看任意时间点的档案状态
3. **可对比：** 支持任意两个版本之间的 diff 对比

### 版本记录结构（建议）

```json
{
  "versionId": "v20260708_001",
  "userId": "u_xxx",
  "modifiedAt": "2026-07-08T12:00:00Z",
  "modificationType": "user_edit | ai_update | system",
  "changedFields": ["birthTime", "gender"],
  "previousVersion": "v20260707_002",
  "snapshot": {
    // 修改后的完整档案快照
  },
  "diff": {
    "birthTime": { "old": "unknown", "new": "14:30" }
  }
}
```

**决策结论：V1 全快照方案（Snapshots Only，ARC-15-P1-003 已确认）**

每次修改用户档案时，存储一份完整的档案 JSON 快照。两个快照之间的 diff 由查询时动态计算生成，**不在存储层做 diff 持久化**。

| 维度 | 说明 |
|------|------|
| 实现复杂度最低 | 每条版本记录直接存完整 `snapshot` JSON，无 patch 构建/apply 逻辑，无需 checkpoint 管理，SQLite 单表 + JSON 字段即可 |
| 回溯性能最优 | 回溯任意版本直接读一条记录即可，无需沿 patch 链逐个 apply，对「历史查看」核心 UX 最友好 |
| V1 存储成本可忽略 | 单用户档案 JSON < 100KB，本地 SQLite 存储，即使每年 60 个版本单用户年存储量 < 6MB，远非瓶颈 |
| 无累积误差风险 | 全快照方案不存在 patch 链累积浮点/精度误差问题，数据可靠性最高 |
| 技术独立性 | 无需引入外部库（jsonpatch / json-diff），减少依赖链 |

**方案约束：**
- 历史版本记录为仅追加写（append-only），不支持物理删除（GDPR 合规删除需全表扫描 + 重建）
- 大面积 AI 批量更新时快照体积可能突变，需在服务端做合理分段
- diff 对比由服务端在查询时动态计算（逐字段对比两个快照），不在存储层做

**数据库 Schema 约束：** 版本历史表（`version_history`）的 `snapshot` 字段定义为 `TEXT` / `JSON` 类型，每条记录独立存储完整 JSON。

> 曾评估的替代方案（未采纳，V2+ 参考）：增量混合方案（JSON Patch + 每 30 次快照 checkpoint）、纯增量方案（JSON Patch Only）。V1 场景下复杂度代价远超存储收益，建议 V2 用户量大时重新评估。

### 8.8.1 排盘引擎抽象接口（BaziEngine）

> **来源：** ARC-15-P1-006 | bazi-skill-dist 评估：复用可行性 2.5/5，推荐路径 C（6-10 周）

排盘引擎通过 Python ABC 抽象接口接入，避免与具体实现耦合。

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional

@dataclass
class BaZiRequest:
    birth_year: int
    birth_month: int
    birth_day: int
    birth_hour: int
    birth_minute: int
    calendar_type: str           # "solar" | "lunar"
    gender: str                  # "male" | "female"
    longitude: float             # 经度（真太阳时校准，必填）
    latitude: float              # 纬度
    timezone: str                # 时区标识（必填）
    birth_time_unknown: bool = False

@dataclass
class Pillar:
    heavenly_stem: str
    earthly_branch: str
    hidden_stems: list[str]
    na_yin: str

@dataclass
class BaZiResult:
    year_pillar: Pillar
    month_pillar: Pillar
    day_pillar: Pillar
    hour_pillar: Pillar
    day_master: str
    gender: str
    birth_time_unknown: bool
    great_fortune_start_age: int
    great_fortune_cycle: list[dict]
    yong_shen: Optional[str] = None      # 用神
    xi_shen: Optional[str] = None        # 喜神
    ji_shen: Optional[str] = None        # 忌神
    engine_name: str = ""
    engine_version: str = ""
    calculated_at: str = ""

class BaziEngine(ABC):
    """排盘引擎抽象基类"""

    @abstractmethod
    def paipan(self, request: BaZiRequest) -> BaZiResult:
        pass

    @abstractmethod
    def validate(self, request: BaZiRequest) -> list[str]:
        pass

    @property
    @abstractmethod
    def engine_info(self) -> dict:
        """返回引擎能力声明，至少包含：
        handles_jieqi / handles_lichun / handles_true_solar_time / supports_yongshen
        """
        pass

    def self_check(self) -> list[str]:
        """自检：用边界案例验证引擎准确性。
        至少覆盖立春（年柱）和惊蛰（月柱）临界点。
        返回失败项列表，空列表表示全部通过。
        """
        return []
```

**self_check() 机制：** 任何新引擎接入前必须通过 self_check，强制验证节气换年和节气换月的正确性。已知 bazi-skill-dist 在这两个案例上会失败。

**已知候选引擎评估摘要（bazi-skill-dist）：**

| 类别 | 可复用 | 必须重写 | 新增设计 |
|------|--------|---------|---------|
| **内容** | 藏干表、十神计算、大运顺逆规则、BaziChart/Pillar 数据模型思路 | 四柱计算核心（无立春/节气判断）、起运年龄计算（占位实现） | 用神/喜神/忌神推导引擎、engine_info 能力声明、self_check 校验 |
| **工作量** | — | — | 6-10 周（路径 C，前端改动为零） |

## 8.9 客户端通信架构

> **来源：** ADR-019（客户端与核心服务通信协议设计）

### 8.9.1 通信设计原则

客户端与核心服务之间的通信遵循以下设计原则：

1. **客户端不直接访问业务模块** — 所有业务请求必须通过 API 通道，客户端不直接导入或调用业务模块内部实现
2. **所有业务请求经过 API 层** — API 层作为统一入口，负责请求路由、参数校验、响应格式化
3. **通信协议面向未来 Web 扩展设计** — V1 的 localhost 通信协议应天然兼容未来远程 Web API，避免后期协议迁移成本

### 8.9.2 双通道通信模型

V1 采用 **HTTP API + WebSocket 双通道通信架构**（ADR-019）：

**HTTP API 通道：**

负责请求/响应模式的业务操作，适用于有明确开始和结束的同步/异步请求。

| 类型 | 示例 | 协议 |
|------|------|------|
| 用户数据 | 用户资料 CRUD、档案查询 | HTTP REST |
| 业务请求 | 八字排盘、深度分析、报告生成 | HTTP REST |
| 系统管理 | 配置同步、版本检查、健康检查 | HTTP REST |

**WebSocket 通道：**

负责实时通信和流式输出的场景，适用于需要持续连接或推送数据的交互模式。

| 类型 | 示例 | 协议 |
|------|------|------|
| 实时聊天 | AI 角色对话、多轮交互 | WebSocket |
| 流式输出 | AI 回复逐 Token 推送、报告分段生成 | WebSocket |
| 状态通知 | 长任务进度推送、分析过程状态更新 | WebSocket |

### 8.9.4 未来 Web 扩展

双通道架构天然支持从本地通信向远程 Web API 的平滑过渡：

| 阶段 | 架构模式 | 说明 |
|------|----------|------|
| **V1（当前）** | PySide6 Client + localhost API | 桌面应用通过 localhost HTTP/WebSocket 调用本地核心服务 |
| **V2（Web 扩展）** | Vue Web Client + HTTP API | Web 前端通过远程 HTTP/WebSocket 调用云端核心服务 |
| **核心服务** | Python Core Service | 不变 —— 同一套业务逻辑，仅部署位置从本地变为云端 |

**过渡路径：**

```
V1 ──── 本地桌面模式
  PySide6 APP → localhost HTTP/WebSocket → Python 核心服务（本地进程）

V2 ──── Web 扩展模式（新增，不取代 V1）
  Vue Web APP → 远程 HTTP/WebSocket → Python 核心服务（云端部署）
  PySide6 APP → localhost HTTP/WebSocket → Python 核心服务（本地进程，向后兼容）

V3 ──── 全平台模式
  Web / Mobile APP → 远程 HTTP/WebSocket → Python 核心服务（云端部署）
  Windows APP → localhost HTTP/WebSocket → Python 核心服务（可选本地或云端）
```

**核心复用点：**
- API 接口契约（路由、请求/响应格式）在 V1 和 V2 之间保持一致
- Python 核心服务的业务模块无需修改，仅增加远程部署方式
- WebSocket 的实时通信机制在本地和远程模式下均可工作

## 8.10 核心服务部署与打包

> **来源：** ADR-017（核心业务服务与客户端解耦）、ARC-15-P1-007（已确认方案）

### 8.10.1 部署方案：PyInstaller Sidecar（V1 已确认）

Python 核心服务使用 PyInstaller 打包为独立的 `core-service.exe`（单文件或单目录模式），随 APP 安装包一同分发。PySide6 UI 进程在启动时以子进程方式启动 `core-service.exe`，通过 localhost IPC 通信（详见 §8.9）。

**决策结论（ARC-15-P1-007，产品负责人已确认）：** V1 采用 PyInstaller Sidecar 方案。

| 维度 | 评价 |
|------|------|
| 技术可行性 | PyInstaller 是 Python 打包最成熟的工具之一，对 fastapi/uvicorn 等依赖兼容性好 |
| 用户体验 | 用户零配置——无需安装 Python 运行时，安装包解压即用 |
| 工程复杂度 | 中等——需配置 hidden imports 和子进程生命周期管理，但无 C/C++ 胶水代码 |
| 版本管理灵活性 | 高——核心服务与 APP 可独立升级，Major 版本锁定确保兼容 |
| 未来演进 | 架构与 Web V2 一致——侧车进程模式可直接迁移为远程服务 |

**优点：** 用户零配置；版本独立管理；子进程管理灵活（崩溃自动重启）；开发模式可直接运行 .py。
**缺点：** 打包体积较大（Python 解释器 + 依赖 ≈ 50-100MB）；hidden imports 需手动配置；多进程 IPC 与生命周期管理复杂度增加；可执行文件有反编译风险。

### 8.10.2 版本兼容策略（SemVer）

核心服务与 APP 各自采用 SemVer（MAJOR.MINOR.PATCH），相互独立但受兼容矩阵约束：

```
核心服务版本: MAJOR.MINOR.PATCH (SemVer)
APP 版本:     MAJOR.MINOR.PATCH (SemVer)

兼容矩阵:
  APP v1.x.x → Core v1.y.z (y >= y_min, z >= z_min)
  APP v2.x.x → Core v2.y.z (y >= y_min, z >= z_min)

规则:
  - Major 版本降级：需确认兼容性表
  - Minor 版本：APP 与 Core 各自独立升级，Major 不变即可
  - Patch 版本：完全独立，向前兼容
  - Core Major 升级视为架构变更，需经过 ADR 流程
```

> 与 `13_API接口契约设计.md` §6 版本策略对齐：API 层采用 URL 路径版本化（`/api/v1/`、`/api/v2/`），核心服务独立 SemVer 与之正交——API 版本反映契约破坏性变更，Core SemVer 反映服务实现版本。

### 8.10.3 安装 / 更新 / 卸载流程

**安装过程：**
1. 解压 APP.exe + core-service.exe + 依赖资源
2. 注册 APP 快捷方式
3. 首次启动时校验 core-service.exe 完整性

**启动流程：**
1. PySide6 启动
2. PySide6 以子进程方式启动 core-service.exe
3. 等待核心服务 health check 通过（轮询 `localhost:port/health`）
4. 启动完成，APP 进入就绪状态

**更新流程：**
1. 新版本安装包同时更新 APP.exe 和 core-service.exe
2. 核心服务 Major 版本升级时，APP 强制升级
3. 核心服务 Minor/Patch 升级时，APP 可选升级

**曾评估的替代方案（未采纳，V2+ 参考）：**
- 方案 B：嵌入式 Python DLL——体积最小但复杂度最高、隔离性差、与 Web V2 矛盾
- 方案 C：预安装 Python 运行时——打包最简单但用户体验最差，不适合商业产品
