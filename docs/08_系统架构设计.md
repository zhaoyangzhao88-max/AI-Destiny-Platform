## 8.1 V1 总体系统架构设计

> **来源：** ADR-014、ADR-015、ADR-016

### 8.1.1 产品整体架构思想

知命 AI 的 V1 架构设计围绕三个核心原则展开：

1. **隐私优先（Privacy First）：** 用户数据归属用户，核心档案本地存储，云端仅作算力扩展（ADR-014）
2. **平台聚焦（Platform Focus）：** V1 集中资源做好 Windows 桌面端体验（ADR-015）
3. **简单演进（Simple Evolution）：** 从单 Agent 起步，预留扩展路径，不早期过度设计（ADR-016）

```
┌─────────────────────────────────────────────────────────┐
│                  知命 AI — V1 系统架构                    │
│                                                         │
│  ┌──────────────────────────────────────────────────┐   │
│  │                 表示层 (Windows APP)              │   │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────────┐   │   │
│  │  │ 首页/对话 │  │ 命盘展示  │  │ 人生档案/报告 │   │   │
│  │  └──────────┘  └──────────┘  └──────────────┘   │   │
│  └──────────────────────────────────────────────────┘   │
│                          │                              │
│  ┌───────────────────────┴──────────────────────────┐   │
│  │              业务逻辑层 (本地优先)                 │   │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────────┐   │   │
│  │  │ 排盘引擎  │  │ 单Agent  │  │ 档案管理系统  │   │   │
│  │  │ (本地)    │  │ 知命AI   │  │ (本地存储)   │   │   │
│  │  └──────────┘  └────┬─────┘  └──────────────┘   │   │
│  └──────────────────────┼──────────────────────────┘   │
│                         │                              │
│  ┌──────────────────────┴──────────────────────────┐   │

│  │  │ 本地数据库   │  │ 本地文件   │                 │   │

### 8.1.2 本地优先原则

**核心思想：** 能本地做的，就不上云。

| 能力 | 处理位置 | 原因 |
|------|----------|------|
| 四柱排盘 | ✅ 本地 | 纯计算，无网络依赖 |
| 用户档案存储 | ✅ 本地 | 数据隐私，离线可用 |
| 基础命理分析 | ✅ 本地 | 规则可固化，无需大模型 |
| 对话界面渲染 | ✅ 本地 | 原生体验 |
| AI 深度分析 | ☁️ 云端按需 | 需要 LLM 大算力 |
| 真太阳时校准 | ☁️ 云端按需 | 需要地理位置数据库 |

**离线分级策略：**

| 网络状态 | 可用功能 |
|----------|----------|
| 🟢 在线 | 全部功能（含 AI 深度分析） |
| 🟡 有限连接 | 排盘、档案浏览、基础分析 |
| 🔴 离线 | 排盘、档案浏览 |

> **离线策略细节** ——【待确认】

### 8.1.3 云端扩展原则

云端在 V1 中的角色被严格限定：

1. **AI 推理扩展：** 仅当用户发起深度分析时，将必要的上下文（脱敏）发送至 LLM API
2. **外部数据服务：** 如真太阳时校准需要地理位置数据库
3. **未来扩展预留：** 云备份、多设备同步为后期功能，V1 不做

**数据上传最小化规则：**
- 只上传当前分析所需的最小上下文
- 不上传完整人生档案
- 所有上传数据标注临时性，分析完成后不持久化在云端

### 8.1.4 Windows APP 作为 V1 入口

V1 以 **Windows 桌面应用** 为首发平台（ADR-015）：

**选型考量因素：**

| 方案 | 跨平台 | 本地能力 | 学习成本 | 推荐度 |
|------|--------|----------|----------|--------|
| 待确认 | 待确认 | 待确认 | 待确认 | — |

> ⚠️ **具体技术框架选型** ——【待确认】  
> 候选：Tauri（Rust 前端）、Electron（JS 生态）、Flutter Desktop、C# WinUI3

**Windows APP 作为首发平台的优势：**
- 本地资源访问无限制（文件系统、SQLite 等）
- 离线运行能力自然支持
- 与 Windows 生态集成（通知、任务栏、系统托盘）
- 适合"本地优先"架构落地

### 8.1.5 AI Agent 总体设计

V1 采用 **单核心知命 AI Agent 架构**（ADR-016）：

```
┌──────────────────────────────┐
│       知命 AI Agent          │
│                              │
│  ┌────────────────────────┐  │
│  │   核心对话引擎          │  │
│  │   - 上下文管理          │  │
│  │   - 对话状态跟踪        │  │
│  └────────┬───────────────┘  │
│           │                  │
│  ┌────────┴───────────────┐  │
│  │   能力路由              │  │
│  │   ┌─────┐ ┌─────┐ ┌──┐ │  │
│  │   │命理  │ │档案  │ │… │ │  │
│  │   │分析  │ │管理  │ │  │ │  │
│  │   └─────┘ └─────┘ └──┘ │  │
│  └────────────────────────┘  │
│                              │
│  ┌────────────────────────┐  │
│  │   工具/插件接口         │  │
│  │   (预留多 Agent 扩展)   │  │
│  └────────────────────────┘  │
└──────────────────────────────┘
```

**Agent 内部能力划分：**
- **对话管理：** 维护与用户的长期对话上下文，管理知命角色的语气与风格（长期上下文采用分层记忆架构，详见 `docs/19_分层记忆架构设计.md`）
- **命理分析：** 调用排盘引擎结果，结合 LLM 进行八字分析
- **档案管理：** 读取/更新用户人生档案，处理三层数据体系
- **工具调用：** 根据需要调用外部工具（排盘库、LLM API 等）

**未来向多 Agent 演进的预留设计：**
- Agent 内部按模块组织 Prompt，方便未来拆分为独立 Agent
- 工具 / 插件接口标准化
- 日志和监控按能力维度分开记录

### 8.1.6 模块化架构思想

所有模块遵循以下设计原则：

1. **高内聚低耦合：** 模块间通过定义良好的接口通信，不直接依赖内部实现
2. **本地替换优先：** 每个模块优先考虑本地实现，云端仅为增强选项
3. **可测试：** 模块边界清晰，支持独立单元测试
4. **渐进增强：** 基础功能可离线运行，联网自动启用增强能力

> **来源：** ADR-017（核心业务服务与客户端解耦原则）

### 8.2.1 架构总览

在 V1 架构中，将原本单体内聚的 APP 拆分为两个协同层：

```

│         │                 │                    │            │

│  │  排盘引擎      │  │  AI 分析引擎   │  │  档案管理系统     │  │
│  │  - 四柱排盘    │  │  - 分析编排    │  │  - 三层数据存储   │  │
│  │  - 真太阳时    │  │  - LLM 调用   │  │  - 版本历史管理   │  │

│  │  - 命理规则库  │  │  Agent 编排  │  │  - LLM API 调用  │  │
│  │  - 逻辑推理机  │  │  - 对话管理  │  │  - 地理数据查询   │  │

└─────────────────────────────────────────────────────────────┘

### 8.2.2 Windows APP 职责

Windows APP 专注于**表示层**职责，不直接处理业务逻辑：

| 职责 | 说明 | 是否涉及业务逻辑 |
|------|------|-----------------|
| **UI 渲染** | 所有用户界面展示（对话、排盘、档案） | ❌ 纯展示 |
| **用户交互** | 输入采集、点击响应、手势控制 | ❌ 纯事件 |
| **本地 IPC 调用** | 将用户请求转发至 Python 核心服务 | ❌ 纯转发 |
| **数据缓存** | 本地 UI 状态缓存（不缓存核心数据） | ❌ 纯缓存 |
| **离线状态管理** | 监控连接状态，展示离线/在线指示器 | ❌ 纯状态 |
| **安装/更新管理** | APP 更新、核心服务版本管理 | ⚠️ 仅版本检查 |
| **系统集成** | 通知、任务栏、系统托盘 | ❌ 纯集成 |

**Windows APP 不直接处理：**
- 排盘计算（转发至核心服务）
- AI 分析（转发至核心服务）
- 档案数据持久化（转发至核心服务）
- 命理规则推理（转发至核心服务）

### 8.2.3 Python 核心服务职责

Python 核心服务作为**独立的本地进程**，承载所有业务逻辑：

| 职责 | 说明 | 依赖 |
|------|------|------|
| **排盘引擎** | 四柱排盘、真太阳时校准、大运流年计算 | 排盘库（本地） |
| **AI 分析引擎** | 八字分析编排、LLM 提示词管理、分析结果结构化 | LLM API（云端） |
| **知命 Agent 编排** | 对话上下文管理、能力路由、多轮交互状态跟踪 | 内部模块 |
| **档案管理系统** | 三层数据体系存储、版本历史管理、数据层级流转 | 本地数据库 |
| **规则推理引擎** | 命理规则库查询、逻辑推理、组合规则匹配 | 规则定义（本地） |
| **外部服务网关** | LLM API 调用管理、超时处理、重试策略、降级方案 | 云端 API |
| **配置管理** | 服务配置、用户偏好、模型选择 | 配置文件 |

**Python 核心服务不处理：**
- UI 渲染（由 APP 负责）
- 用户事件处理（由 APP 负责）

### 8.2.4 客户端与业务逻辑边界

**API 契约定义：**

```
┌─────────────────────┐        IPC 请求        ┌──────────────────────┐
│  Windows APP        │ ──────────────────────→ │  Python 核心服务     │
│  (表示层)            │                        │  (业务逻辑层)         │
│                     │ ←────────────────────── │                      │
│  - 仅处理 UI 状态    │        IPC 响应        │  - 所有业务计算       │
│  - 不处理业务逻辑    │                        │  - 数据持久化         │
│  - 不直接访问数据库  │                        │  - 外部 API 调用      │
└─────────────────────┘                        └──────────────────────┘
```

**边界规则：**

1. **APP 不直接访问数据库** — 所有数据操作通过核心服务的 API 进行
2. **APP 不包含命理分析逻辑** — 排盘、推理、分析全部委托给核心服务
3. **APP 不直接调用 LLM** — AI 分析经由核心服务统一调度
4. **核心服务不包含 UI 代码** — 不产生任何用户界面输出
5. **通信协议标准化** — 所有 API 定义在契约文档中（OpenAPI / protobuf）

**典型请求流程（以"八字分析"为例）：**

```
用户输入出生信息
    │
    ▼
Windows APP 采集输入 → 序列化请求 → 通过 IPC 发送至核心服务
    │
    ▼
Python 核心服务接收请求 → 调用排盘引擎 → 生成四柱八字
    │
    ▼
核心服务调用 AI 分析引擎 → 构造 Prompt → 调用 LLM API
    │
    ▼
LLM API 返回结果 → 核心服务结构化分析结果 → 存入档案数据库
    │
    ▼
核心服务将结果序列化为 IPC 响应 → 返回至 APP
    │
    ▼
Windows APP 接收响应 → 解析 → UI 渲染展示
```

### 8.2.5 未来 Web 扩展能力

解耦架构天然支持未来平台扩展：

**V2 — Web 端扩展：**

```
                    ┌──────────────────────────────────────┐
                    │       Web 前端 (React/Vue)            │
                    └──────────────┬───────────────────────┘
                                   │
                    ┌──────────────┴───────────────────────┐
                    │      Web API 网关 (Nginx/反向代理)     │
                    └──────────────┬───────────────────────┘
                                   │
          ┌────────────────────────┼────────────────────────┐
          │                        │                        │
          ▼                        ▼                        ▼
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│  Python 核心服务   │    │  Python 核心服务   │    │  Python 核心服务   │
│  (本地 Windows   │    │  (服务器部署)      │    │  (服务器部署)      │
│   APP 伴随进程)   │    │                   │    │                   │
└─────────────────┘    └─────────────────┘    └─────────────────┘
    用户 A                   用户 B                   用户 C
```

**扩展路径：**

| 阶段 | 平台 | 客户端 | 核心服务部署 | 数据存储 |
|------|------|--------|-------------|----------|
| V1 | Windows | Windows APP（本地 IPC） | 本地伴随进程 | 本地数据库 |
| V2 | Web | 浏览器（HTTP API） | 云端部署 | 云端数据库 |
| V3 | 移动端 | iOS/Android App（HTTP API） | 云端部署 | 云端数据库 |

**核心复用点：**

- **API 接口复用：** V1 定义的 IPC API 契约直接映射为 Web API endpoint
- **业务逻辑复用：** 排盘引擎、AI 分析、档案管理全部复用，无需重写
- **AI 模型统一：** 所有客户端共享同一套 AI 分析服务，模型升级一次生效

**过渡策略：**

```
V1 ──── APP ↔ 本地 Python 服务 (localhost IPC)
                │
                ▼
V2 ──── APP ↔ 本地 Python 服务 (向后兼容)
        Web ↔ 云端 Python 服务 (复用同一套代码)
                │
                ▼
V3 ──── APP ↔ 本地 Python 服务 或 云端服务 (可选)
        Web ↔ 云端 Python 服务
        Mobile ↔ 云端 Python 服务
```

## 8.3 V1客户端技术架构选择

> **来源：** ADR-018

### 8.3.1 PySide6 + Vue 渐进式混合架构

在 ADR-017（核心服务与客户端解耦）基础上，V1 客户端的 UI 技术架构采用 **PySide6 + Vue 渐进式混合架构**。

核心思想：
- **PySide6** 作为 Windows 桌面应用的根基，负责系统集成和本地应用管理
- **Vue** 作为渐进式 Web UI 能力，在需要复杂交互时逐步引入
- **Python 核心服务** 保持独立，与前端技术栈无关

### 8.3.2 技术职责边界

| 技术层 | 负责内容 | 不负责内容 |
|--------|----------|-----------|
| **Windows APP（PySide6）** | 用户入口、界面容器、系统交互（通知/任务栏/进程管理） | 业务逻辑计算、数据持久化、AI 分析 |
| **Vue UI 模块（渐进引入）** | 富交互页面、AI 聊天界面、高级报告展示、数据可视化 | 系统集成、本地文件管理、进程生命周期管理 |
| **Python 核心服务** | 排盘计算、AI Agent 编排、用户档案管理、知识库与规则推理 | UI 渲染、用户事件处理 |

### 8.3.4 未来 Web 扩展路线

PySide6 + Vue 架构天然支持从桌面端向 Web 端的平滑过渡：

```
V1（桌面首发）
  Windows APP（PySide6）──→ Vue UI（可选）──→ Python 核心服务（本地）
                           │
                           ▼
V2（Web 扩展）
  Web APP（Vue）─────────────────────────────────→ Python 核心服务（云端）
  Windows APP（PySide6）──→ Vue UI ──────────────→ Python 核心服务（本地）
                           │
                           ▼
V3（全平台）
  Web APP（Vue） ────────────────────────────────→ Python 核心服务（云端）
  Mobile（Vue）  ────────────────────────────────→ Python 核心服务（云端）
  Windows APP ───→ Vue UI ──────────────────────→ Python 核心服务（可选本地）
```

**核心复用策略：**
- Python 核心服务始终不变，仅改变部署方式（本地进程 → 云端服务）
- Vue UI 模块在桌面端和 Web 端可高度复用
- PySide6 层仅 Windows 桌面端特有，Web 端和移动端无需此层

### 8.3.5 PySide6 × Vue 集成机制

> **来源：** ADR-020

基于 ADR-018（PySide6 + Vue 混合架构）与 ADR-017（客户端解耦），本小节明确 PySide6 与 Vue 的**集成方案**（ADR-018 后续影响第 3 条遗留的专项决策）：

**V1 采用 QWebEngineView 嵌入式集成：**

1. **渲染方式** — Vue 应用作为 PySide6 `QWebEngineView` 的内嵌网页渲染，Vue 构建产物以本地静态资源或本地 HTTP 服务（localhost）提供。
2. **宿主集成通道** — PySide6 与内嵌 Vue 之间经 **Qt WebChannel** 双向调用（窗口事件、通知、进程生命周期）。
3. **业务通信通道** — Vue 业务交互遵循 ADR-019，通过 **HTTP API + WebSocket（`/ws/v1/session`）** 调用本地 Python 核心服务。
4. **双层通信职责** — WebChannel = 宿主集成（窗口级），HTTP/WS = 业务通信（服务级），互不耦合。

```
Windows APP（PySide6 主窗口）
  ├── 原生控件区（通知 / 任务栏 / 进程管理）
  └── QWebEngineView（内嵌 Vue 模块，按需引入）
        │  Qt WebChannel（宿主集成）
        └── Vue 业务页（AI 聊天 / 报告 / 可视化）
              │  HTTP API + WebSocket /ws/v1/session（ADR-019）
              ▼
        Python 核心服务（本地）
```

**渐进引入原则：** V1 初期以 PySide6 原生控件起步，Vue 仅在复杂交互场景按需经 `QWebEngineView` 嵌入（ADR-018 后续影响第 1 条）。整体 Windows APP 框架选型（Tauri/Electron/Flutter）仍为【待确认】，详见 P1-001（08_架构 8.1.4 / 8.7 对齐）。

## 8.5 数据三层体系

**来源：** ADR-012

### 层级定义

| 层级 | 英文 | 定义 | 存储标记 | 可信度 |
|------|------|------|----------|--------|
| **事实** | Fact | 用户明确确认的真实信息 | `confidence: fact` | ⭐⭐⭐ |
| **推测** | Inference | AI 基于命理规则的分析推论 | `confidence: inference` | ⭐⭐ |
| **待验证** | Pending | AI 提出的假设，需用户反馈 | `confidence: pending` | ⭐ |

### 层级流转规则

```
待验证 ←→ 推测 ←→ 事实
  │          │        │
  ▼          ▼        ▼
未确认    部分确认   用户确认
```

- **待验证 → 推测：** AI 反复确认后仍无法获得用户反馈，但命理分析支持
- **推测 → 事实：** 用户明确确认
- **推测 → 待验证：** 用户否认，但 AI 认为有其他可能性
- **事实不可逆：** 用户确认的事实不能自动降级，修改需通过版本历史

## 8.6 数据版本历史机制

**来源：** ADR-013

### 核心原则

## 8.9 客户端通信架构

> **来源：** ADR-019（客户端与核心服务通信协议设计）

### 8.9.1 通信设计原则

客户端与核心服务之间的通信遵循以下设计原则：

1. **客户端不直接访问业务模块** — 所有业务请求必须通过 API 通道，客户端不直接导入或调用业务模块内部实现
2. **所有业务请求经过 API 层** — API 层作为统一入口，负责请求路由、参数校验、响应格式化
3. **通信协议面向未来 Web 扩展设计** — V1 的 localhost 通信协议应天然兼容未来远程 Web API，避免后期协议迁移成本

### 8.9.2 双通道通信模型

V1 采用 **HTTP API + WebSocket 双通道通信架构**（ADR-019）：

**HTTP API 通道：**

负责请求/响应模式的业务操作，适用于有明确开始和结束的同步/异步请求。

| 类型 | 示例 | 协议 |
|------|------|------|
| 用户数据 | 用户资料 CRUD、档案查询 | HTTP REST |
| 业务请求 | 八字排盘、深度分析、报告生成 | HTTP REST |
| 系统管理 | 配置同步、版本检查、健康检查 | HTTP REST |

**WebSocket 通道：**

负责实时通信和流式输出的场景，适用于需要持续连接或推送数据的交互模式。

| 类型 | 示例 | 协议 |
|------|------|------|
| 实时聊天 | AI 角色对话、多轮交互 | WebSocket |
| 流式输出 | AI 回复逐 Token 推送、报告分段生成 | WebSocket |
| 状态通知 | 长任务进度推送、分析过程状态更新 | WebSocket |

### 8.9.4 未来 Web 扩展

双通道架构天然支持从本地通信向远程 Web API 的平滑过渡：

| 阶段 | 架构模式 | 说明 |
|------|----------|------|
| **V1（当前）** | PySide6 Client + localhost API | 桌面应用通过 localhost HTTP/WebSocket 调用本地核心服务 |
| **V2（Web 扩展）** | Vue Web Client + HTTP API | Web 前端通过远程 HTTP/WebSocket 调用云端核心服务 |
| **核心服务** | Python Core Service | 不变 —— 同一套业务逻辑，仅部署位置从本地变为云端 |

**过渡路径：**

```
V1 ──── 本地桌面模式
  PySide6 APP → localhost HTTP/WebSocket → Python 核心服务（本地进程）

V2 ──── Web 扩展模式（新增，不取代 V1）
  Vue Web APP → 远程 HTTP/WebSocket → Python 核心服务（云端部署）
  PySide6 APP → localhost HTTP/WebSocket → Python 核心服务（本地进程，向后兼容）

V3 ──── 全平台模式
  Web / Mobile APP → 远程 HTTP/WebSocket → Python 核心服务（云端部署）
  Windows APP → localhost HTTP/WebSocket → Python 核心服务（可选本地或云端）
```

**核心复用点：**
- API 接口契约（路由、请求/响应格式）在 V1 和 V2 之间保持一致
- Python 核心服务的业务模块无需修改，仅增加远程部署方式
- WebSocket 的实时通信机制在本地和远程模式下均可工作
