## 1️⃣ 文档目的

### 1.1 为什么要这份文档

本项目已进入 Phase 2 末期，即将进入代码开发（Phase 3）阶段，AI Agent 的协作场景将从**单 AI 串行执行**逐步转向**多 AI 并行协作**。

[16_AI协作治理体系审计报告](./16_AI协作治理体系审计报告.md) 明确指出以下三个核心缺口：

| 审计发现 | 严重程度 | 本文档对应章节 |
|---------|---------|--------------|
| ❌ 无 AI 身份标识约定 → 多 AI 并行时无法追溯修改来源 | 最弱维度(5/10) | 附录C |
| ❌ 无跨 AI 交接协议 → AI 交接时无上下文转移机制 | 最弱维度(5/10) | 第🔟章 |
| ❌ 无冲突裁决机制 → 不同 AI 给出分歧意见时无标准裁决流程 | 最弱维度(5/10) | 第9️⃣章 |

本文档旨在系统性填补上述三个缺口。

### 1.2 与 docs/12 的关系

[12_AI协作开发规范.md](./12_AI协作开发规范.md) 第二章定义了按**任务类型**划分的五种功能角色（文档撰写者、方案分析师、代码开发者、评审者、测试者），适用于单次任务的角色声明。

本文档定义了按**治理职能**划分的五种 Agent 角色，适用于多 AI 并行场景下的权限划分与协作治理。

| 维度 | docs/12（功能角色） | docs/17（治理角色） |
|------|-------------------|-------------------|
| 划分依据 | 任务类型 | 治理职能 |
| 适用场景 | 单 AI 执行单任务 | 多 AI 并行协作 |
| 核心关注 | 做什么、输出什么 | 能改什么、不能改什么、如何裁决 |
| 角色举例 | 文档撰写者、代码开发者 | Architect Agent、Audit Agent |

**同一 AI 在一次会话中可以同时承担两套角色**，但需分别声明。例如：当前会话以 `Architect Agent + 方案分析师` 身份执行架构分析任务。

## 2️⃣ 角色体系总览

### 2.1 五角色矩阵

| # | 角色 | 职责定位 | 核心交付物 | 权限等级 |
|---|------|---------|-----------|---------|
| 1 | **Architect Agent** | 架构设计与技术决策 | 架构文档、ADR Draft、技术方案 | W(受限) |
| 2 | **Developer Agent** | 编码实现与缺陷修复 | 源代码、测试用例 | W(代码) |
| 3 | **Audit Agent** | 质量审计与风险识别 | 审计报告、问题清单 | W(报告) |
| 4 | **Product Agent** | 需求分析与产品设计 | PRD、用户旅程、流程文档 | W(产品文档) |
| 5 | **Knowledge Agent** | 知识库建设与数据管理 | 知识库内容、数据字典 | W(知识文档) |

### 2.2 角色选取原则

1. **按需选取** — 每次任务开始时声明当前角色
2. **一人多角** — 同一 AI 可顺序承担多个角色，但需明确声明切换
3. **角色互斥** — 在同一任务中，**Audit Agent 不得与 Developer Agent 为同一 AI**（避免自我审计）
4. **角色追溯** — 每份输出文件必须标注产出角色的 Agent ID

## 3️⃣ Architect Agent（架构师 Agent）

### 职责

- 设计系统整体架构与技术方案
- 制定技术选型，权衡方案利弊
- 编写和更新架构设计文档
- 创建 ADR Draft，提交产品负责人确认
- 确保跨模块的架构一致性
- 响应其他 Agent 的架构疑问

### 输入

- 已确认的 ADR（ADR-001~ADR-020）
- 现有架构设计文档（`docs/08_系统架构设计.md`）
- 产品需求文档和用户旅程
- API 接口契约（`docs/13_API接口契约设计.md`）
- WebSocket 通信协议（`docs/14_WebSocket实时通信协议设计.md`）
- 技术调研资料

### 输出

- 架构设计文档 / 更新
- ADR Draft（标记为 Draft 状态）
- 技术选型对比方案（标记为"建议"）
- 架构一致性检查报告
- 对 Developer Agent 的技术指导

### 可修改文件范围

| 路径模式 | 允许操作 | 限制条件 |
|---------|---------|---------|
| `docs/08_系统架构设计.md` | 修改、扩展 | 不得与已 Approved ADR 冲突 |
| `docs/13_API接口契约设计.md` | 修改、扩展 | 需注明修改理由 |
| `docs/14_WebSocket实时通信协议设计.md` | 修改、扩展 | 需注明修改理由 |
| `docs/**/ADR Draft 新增` | 创建 | 标记为 Draft，不可自行升为 Approved |
| `docs/其他技术相关文档` | 修改 | 仅限 Draft 状态的文档 |

### 禁止操作

| 禁止事项 | 理由 |
|---------|------|
| ❌ 直接修改已 ✅ Approved 的 ADR | 文档契约红线，必须通过 ADR 质疑流程 |
| ❌ 修改业务代码（`src/` 目录） | 超出架构职责范围 |
| ❌ 自行将 ADR Draft 标记为 Approved | 需产品负责人确认 |
| ❌ 自行填补【待确认】内容 | 违反 12_规范第一章原则 |

## 4️⃣ Developer Agent（开发者 Agent）

### 职责

- 根据架构设计与 API 契约编写业务代码
- 修复已识别或报告的 Bug
- 编写和维护单元测试与集成测试
- 确保代码质量（可读性、性能、安全）
- 将运行结果和执行报告反馈给团队

### 输入

- 架构设计文档（`docs/08_系统架构设计.md`）
- API 接口契约（`docs/13_API接口契约设计.md`）
- WebSocket 通信协议（`docs/14_WebSocket实时通信协议设计.md`）
- 技术方案 / 任务描述
- 产品需求 / 用户故事

### 输出

- 源代码（`src/` 目录下的业务代码）
- 测试用例（`tests/` 目录下的测试代码）
- 执行报告（含运行结果、测试覆盖率）
- Bug 修复说明
- 开发过程中发现的架构问题反馈

### 可修改文件范围

| 路径模式 | 允许操作 | 限制条件 |
|---------|---------|---------|
| `src/**` | 创建、修改 | 不得超出任务范围 |
| `tests/**` | 创建、修改 | 不得修改被测源码（测试框架配置除外） |
| 项目配置文件（`*.json`, `*.yaml`, `*.cfg` 等） | 修改 | 需与架构方案一致 |
| `README.md` | 修改 | 仅限技术相关补充 |
| `docs/09_开发规范.md` | 补充 | 仅限编码实践相关，不改架构内容 |
| `docs/10_测试规范.md` | 补充 | 仅限测试实践相关 |

### 禁止操作

| 禁止事项 | 理由 |
|---------|------|
| ❌ 修改架构文档（`docs/08_`） | 架构决策权归 Architect Agent |
| ❌ 创建新的 API 端点 | 需经 Architect Agent 审批 — 防止未受控的 API 扩散 |
| ❌ 修改 ADR 文档 | 超出开发者权限 |
| ❌ 修改产品定位文档 | 超出编码职责 |
| ❌ 删除未在任务范围内的文件 | 需先报告并等待确认 |
| ❌ 引入未经 Architect 评估的依赖 | 防止技术债务累积 |

## 5️⃣ Audit Agent（审计 Agent）

### 职责

- 执行阶段门审计，评估是否可以进入下一阶段
- 检查跨文档一致性（文档 vs ADR vs 架构 vs 代码）
- 识别架构性、安全性、性能风险
- 按 P0/P1/P2 三级对问题分级
- 提出修复建议并跟踪问题闭环

### 输入

- 项目全部文档（`docs/` 目录）
- 已确认的 ADR（`docs/04_ADR_产品决策记录.md`）
- 代码库（如有 Phase 3+ 的代码）
- 上一阶段的审计报告（如有）
- AI 协作日志（`docs/11_AI协作日志.md`）

### 输出

- 审计报告（含审计范围、方法、发现、分级风险、建议）
- 问题跟踪表（跨会话持续跟踪）
- 修复建议清单（分 P0/P1/P2）
- 阶段门通过/拒绝/有条件通过结论

### 可修改文件范围

| 路径模式 | 允许操作 | 限制条件 |
|---------|---------|---------|
| `docs/<编号>_Phase<X>架构审计报告.md` | 创建、修改 | 审计报告内容独立 |
| `docs/<编号>_AI协作治理体系审计报告.md` | 创建、修改 | 审计报告内容独立 |
| 对审计发现的问题列表 | 创建 | 仅记录发现，不修改被审计对象 |

### 禁止操作

| 禁止事项 | 理由 |
|---------|------|
| ❌ 修改已确认的 ADR | 审计是只读检查，非修改权限 |
| ❌ 修改架构设计文档 | 修复建议应通过 Architect Agent 执行 |
| ❌ 修改业务代码 | 修复建议应通过 Developer Agent 执行 |
| ❌ 修改产品定位或 Persona 文档 | 超出审计职责 |
| ❌ 自行修复审计发现的问题 | 审计与修复必须分离角色（自我审计无效） |

> **关键原则：** Audit Agent 发现 → 记录 → 分级 → 建议修复，但不执行任何修改。修改由相应角色的 Agent 在收到指令后执行。

## 7️⃣ Knowledge Agent（知识 Agent）

### 职责

- 建设和维护命理知识库（四柱八字、神煞、大运流年等）
- 管理和维护数据字典与数据模型文档
- 整理和验证命理数据的准确性
- 建立知识图谱关系
- 确保 AI 输出的命理内容有据可查

### 输入

- 命理典籍和数据来源
- 数据模型设计文档
- 知识图谱结构定义
- 已确认的 ADR（数据相关：ADR-012、ADR-013 等）
- 命理排盘引擎接口定义

### 输出

- 知识库内容（结构化 / 非结构化知识）
- 数据字典 / 数据模型文档更新
- 知识图谱关系定义
- 数据来源引用清单
- 命理知识验证报告

### 可修改文件范围

| 路径模式 | 允许操作 | 限制条件 |
|---------|---------|---------|
| 知识库相关文件（路径待定——Phase 3 确定后补充） | 创建、修改 | 数据来源需可追溯 |
| `docs/数据模型相关文档`（创建后） | 创建、修改 | 不与 ADR 冲突 |
| 数据字典文件 | 创建、修改 | 字段定义需明确 |
| 命理规则 / 排盘配置 | 补充 | 需注明参考来源 |

### 禁止操作

| 禁止事项 | 理由 |
|---------|------|
| ❌ 修改产品定位文档（`docs/02_`） | 产品决策权归 Product Agent / 产品负责人 |
| ❌ 修改 AI Persona 设定（`docs/05_`） | 角色人格设定需产品负责人确认 |
| ❌ 修改技术架构 | 架构决策权归 Architect Agent |
| ❌ 直接修改已 Approved ADR | 文档契约红线 |
| ❌ 引入未经验证的命理数据 | 防止误导性内容进入产品 |
| ❌ 自行决定数据合规策略 | 涉及 PIPL/GDPR，需产品负责人确认 |

## 8️⃣ AI Agent 权限模型

### 8.1 三级权限体系

| 权限 | 符号 | 含义 | 适用范围 |
|-----|------|------|---------|
| **修改权限** | **W** | 可以创建和修改指定范围内的文件 | 按角色划定路径模式（见第3~7章） |
| **审批权限** | **A** | 可以批准或拒绝 Draft 文档升为 Approved | **仅产品负责人可持有**，AI Agent 不可持有 |
| **只读权限** | **R** | 可以读取但不可修改文件内容 | 跨角色参考时默认权限；未明确授予 W 的文件默认为 R |

### 8.2 权限分配矩阵

| 文件类型 \ 角色 | Architect | Developer | Audit | Product | Knowledge |
|----------------|:---------:|:---------:|:-----:|:-------:|:---------:|
| ADR (Approved) | R | R | R | R | R |
| ADR (Draft) | **W** | R | R | **W** | R |
| 架构文档 `docs/08_` | **W** | R | R | R | R |
| API 契约 `docs/13_` | **W** | R | R | R | R |
| WS 协议 `docs/14_` | **W** | R | R | R | R |
| 源代码 `src/` | R | **W** | R | R | R |
| 测试代码 `tests/` | R | **W** | R | R | R |
| 审计报告 | R | R | **W** | R | R |
| 产品文档 `docs/01_-02_` | R | R | R | **W** | R |
| UX 文档 `docs/06_` | R | R | R | **W** | R |
| AI 流程 `docs/07_` | R | R | R | **W** | R |
| 知识库文件 | R | R | R | R | **W** |
| 协作日志 `docs/11_` | **W** | **W** | **W** | **W** | **W** |
| 开发规范 `docs/09_` | **W** | **W** | R | R | R |
| 测试规范 `docs/10_` | **W** | **W** | R | R | R |

> **注解：** 所有 Agent 均需在协作日志中记录自己的执行记录，故 `docs/11_` 统一为 W。

### 8.3 权限继承规则

1. **升级继承** — 被授予 W 权限的 Agent 自动拥有该文件范围内的 R 权限
2. **例外上报** — 如确需修改权限矩阵外的文件，必须先记录到协作日志并等待产品负责人确认
3. **权限降级** — 产品负责人可在任务级别临时降级某 Agent 权限（如：限制某次审计仅 R 权限，不产生审计报告文件）

### 8.4 权限违反处理

| 违规等级 | 情形 | 处理方式 |
|---------|------|---------|
| 🔴 一级违规 | 修改了已 Approved ADR | 立即回滚变更，在协作日志中记录违规，产品负责人评估影响 |
| 🟡 二级违规 | 修改了权限矩阵中仅 R 的文件 | 回滚变更，在日志中记录，产品负责人决定是否需修正 |
| 🟢 三级违规 | 修改了权限范围内但超出任务范围的文件 | 在日志中记录，下次任务前确认范围边界 |

## 9️⃣ 多 AI 冲突裁决机制

### 9.1 冲突类型

| 冲突类型 | 示例 | 常见场景 |
|---------|------|---------|
| **技术方案冲突** | 两个 AI Agent 提出不同的数据库选型 | 架构讨论、技术选型 |
| **接口定义冲突** | 一个 Agent 定义 RESTful，另一个建议 GraphQL | API 契约设计 |
| **流程冲突** | 一个 Agent 建议同步流程，另一个建议异步 | 业务逻辑设计 |
| **数据解释冲突** | 对同一命理规则的不同理解 | 知识库建设 |

### 9.2 裁决层级（从高到低）

当不同 AI Agent 对同一问题产生分歧时，按以下层级裁决：

```
优先级 1: ADR（已 Approved）
    ↑ 最高权威 — 已确认的产品/技术决策，不可绕过的约束
    ↓ 适用场景：方案与已确认 ADR 冲突时，以 ADR 为准

优先级 2: 架构文档
    ↑ 次高权威 — 已确认的架构设计
    ↓ 适用场景：实现方案与架构设计冲突时，以架构文档为准
    ↓ 注：架构文档必须与 ADR 一致；如两者冲突，需启动 ADR 质疑流程

优先级 3: API 契约
    ↑ 中高权威 — 已确认的接口定义
    ↓ 适用场景：实现细节与 API 契约冲突时，以契约为准

优先级 4: 实现建议
    ↑ 中低权威 — 未经架构文档确认的技术方案
    ↓ 适用场景：两个实现方案争议时，选与最上层约束冲突最少的

优先级 5: AI 普通建议
    ↑ 最低权威 — 无文档支撑的口头/对话式建议
    ↓ 适用场景：所有建议级别，仅作为参考意见，不做裁决依据
```

### 9.3 裁决流程

```
第1步：发现冲突
  ↓
第2步：定位层级 —— 冲突点属于哪一层的决策范畴？
  ↓
第3步：提取约束 —— 从对应层级文档中提取约束条件
  ↓
第4步：对比裁决 —— 与层级约束对比，符合者胜出
  ↓
第5步：记录裁决 —— 写入协作日志，附上引用文档路径和理由
```

### 9.4 同层级冲突处理

当冲突双方处于同一层级且无法直接引用更上层约束时：

1. **交叉评审** — 涉及的两个 Agent 分别输出评审意见
2. **投票机制** — 引入第三个 Agent（通常是 Audit Agent）作为仲裁方
3. **三方意见汇总** — 记录各自理由
4. **产品负责人终裁** — 如仍无法达成一致，升级至产品负责人
5. **记录决议** — 将终裁结果和理由写入协作日志

### 9.5 紧急裁决路径

当冲突影响开发进度时，可跳过交叉评审步骤，由产品负责人直接终裁：

```
发现冲突 → 影响评估（P0/P1/P2）→ 产品负责人决策 → 记录执行
```

### 9.6 裁决记录模板

每次冲突裁决后，需在 `docs/11_AI协作日志.md` 中追加以下结构：

```
--- 冲突裁决记录 ---
冲突日期：YYYY-MM-DD
冲突标题：[一句话描述]
参与方：[Agent ID A] vs [Agent ID B]
冲突类型：[技术方案 / 接口定义 / 流程 / 数据解释]
引用层级：[ADR-X / 架构文档章节 / API 契约章节]
裁决依据：[引用具体内容]
裁决结果：[胜出方 + 理由]
产品负责人确认：[是/否]
```

## 附录C：多 AI 身份标识约定

### C.1 Agent ID 注册规则

当多个 AI Agent 在同一项目中协作时，每个 AI 实例必须在开始任务前注册唯一的身份标识。

**格式：** `Agent-{日期}-{序号}`

| 字段 | 说明 | 示例 |
|------|------|------|
| `Agent` | 固定前缀 | Agent |
| `{日期}` | 当前日期，YYYYMMDD | 20260708 |
| `{序号}` | 当日流水号，从 01 开始 | 01 |

**完整示例：** `Agent-20260708-01`、`Agent-20260708-02`

### C.2 注册方式

在 `docs/11_AI协作日志.md` 中追加身份声明记录：

```
--- Agent 身份注册 ---
Agent ID:   Agent-20260708-01
AI 类型:    Claude Code (DeepSeek V4 Flash)
启动时间:   2026-07-08 14:00
本次治理角色: Architect Agent
本次功能角色: 方案分析师
任务来源:   产品负责人
```

### C.3 使用规则

1. **唯一性** — 同一日期内序号不可重复；跨日期允许相同序号
2. **声明即注册** — 首次在协作日志中声明后即生效
3. **追溯标识** — 每次修改文件时，在协作日志中标注 Agent ID
4. **会话内唯一** — 同一 AI 会话中若切换角色，Agent ID 不变，仅更新角色声明

> **补充说明：** 当前实现阶段以**人类可读的身份标识**为主。未来如需机器可读的身份签名（如 GPG 签名提交），可在 Phase 3 或 Phase 4 阶段补充完善。
