## 1. 审计目的

本次审计定位为 **Phase 2 → Phase 3 的阶段门审查**。项目当前处于产品设计与文档建设期（15 份文档、19 条 ADR，全为 Draft 状态），尚未进入代码开发。审计目标：

1. **验证架构一致性** — ADR 决策、架构分层、API 契约、WebSocket 协议之间的交叉一致性
2. **识别工程风险** — 按 P0/P1/P2 分级扫描阻止进入代码开发的问题
3. **评估开发准备度** — 7 维度评分体系判断是否可以安全进入 Phase 3
4. **产出行动清单** — 明确进入 Phase 3 前必须完成的前置条件

### 4.3 重要发现

## 5. API 与通信协议分析

### 5.1 HTTP API 评估

| 维度 | 得分 | 评价 |
|------|------|------|
| **完整性** | 7/10 | 36 个端点覆盖核心 CRUD + 异步任务，但缺少账号删除、密码修改、版本对比端点 |
| **REST 设计** | 7/10 | 资源命名合理，部分子路径（如 `/charts/{id}/analysis`）表达动作而非资源 |
| **一致性** | 6/10 | **3 处 P0 级内部矛盾**（详见 5.3 节） |
| **错误码体系** | ✅ 良好 | 6 段范围编码覆盖系统/鉴权/校验/资源/业务/限流，映射到 HTTP 状态码合理 |
| **版本策略** | ✅ 良好 | URL 路径版本化 + 兼容性规则表 + 6 个月弃用期，缺少 WS 协议版本策略 |

### 5.2 WebSocket 协议评估

| 维度 | 评价 |
|------|------|
| **消息类型** | 15 种（含 error）覆盖 AI 流式、任务推送、心跳、订阅。`task.resume`/`task.list` 缺失于正式类型表 |
| **路径设计** | ❌ **三处矛盾**：`/ws/v1/session` vs `/ws/v1/task/{task_id}` vs `/ws/v1/chat` |
| **心跳机制** | ✅ 合理（30s ping / 30s timeout / 90s idle），pong timeout = ping interval 无缓冲余量 |
| **重连机制** | ✅ 指数退避 (1s→30s, ×2, jitter ±500ms)，最多 10 次 |
| **Task 恢复** | ⚠️ 流程定义完整但 `task.resume`/`task.list` 不在正式消息类型表中，`sequence` 字段标为可选 |
| **错误码映射** | ⚠️ HTTP 错误码与 WS Close 帧码无交叉引用 |

### 5.3 P0 级跨文档冲突（6 项）

| # | 冲突 | 涉及文档 | 影响 |
|---|------|---------|------|
| 1 | **Task 字段名不一致**：HTTP 用 `type`、WS 定义用 `task_type`、WS 示例又用 `type` | `13_API` + `14_WS` | 客户端服务端解析错误 |
| 2 | **POST /reports 响应示例矛盾**：3.4 节 `status: pending`，7.3 节 `status: completed` | `13_API` | 实现者无法确定正确格式 |

| 5 | **HTTP 4.4 节异步任务示例自相矛盾**：type 字段与段落说明矛盾 | `13_API` + `14_WS` | 同节内数据自相矛盾 |
| 6 | **用户模块端点计数错误**：标注 8 个但表格实际 9 个 | `13_API` | 文档可信度问题 |

## 6. 数据模型准备度

### 6.1 评估结果

| 维度 | 结果 |
|------|------|
| **API 字段覆盖度** | 73% — 核心实体主要字段已定义 |
| **Schema 设计准备度** | 🟡 **Conditional** — 满足 3 个条件后可进入 |
| **数据复杂度** | 🔴 Complex — 6 大实体的多层次关系 |
| **存储建议** | V1 SQLite (本地优先) → V2 PostgreSQL (云端扩展) |

### 6.2 Schema 设计前置条件

1. ✅ 数据库选型确认（建议 SQLite）
2. ❌ 版本快照格式定义（推荐 JSON Patch RFC 6902）
3. ❌ 知识库内部 Schema 补充（当前仅定义 API 表面）
4. ❌ 数据三层体系状态机形式化定义

### 6.3 缺失的关键字段

- **用户档案：** `nickname`、`birth_time_unknown` 标记、`timezone` 字段
- **八字排盘：** 用神/忌神结构化定义、格局类型枚举、纳音/神煞字段
- **分析报告：** `section.metadata` 完整结构、置信度分数体系
- **知识库：** 内部内容结构、引用/来源结构、知识图谱关系定义

## 9. Phase 3 进入建议

### 建议：🟡 Conditional（有条件进入 Phase 3 — Phase 2 修复阶段）

**总分 56/100（不及格线 60%），存在 6 项 P0 问题。**

### 否定的理由

- 总分低于及格线，文档一致性仅 4/10（40%），是最大的薄弱环节
- **AI 上下文窗口管理缺失**和 **PySide6/Vue 集成未决策** 将直接导致编码后大规模返工
- 6 项 P0 跨文档矛盾（Task 字段名、WS 路径、API 示例）会在实现阶段引发解析错误

### 有条件肯定的理由

- 产品设计体系完整（14 份文档 + 19 条 ADR）
- 架构分层清晰（6 层模型），API 覆盖面广（36+ endpoints）
- 错误码体系和版本策略设计精良
- 所有 P0 问题都是 **文档层面的不一致和设计空缺**，而非根本性方向错误
- 一旦修复前置条件，即可安全进入开发
