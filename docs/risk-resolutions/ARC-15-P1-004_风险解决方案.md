## 根因分析

### 当前离线分级定义

`08_系统架构设计.md` §8.1.2 当前定义：

| 网络状态 | 可用功能 |
|----------|----------|
| 🟢 在线 | 全部功能（含 AI 深度分析） |
| 🟡 有限连接 | 排盘、档案浏览、基础分析 |
| 🔴 离线 | 排盘、档案浏览 |

问题在于「有限连接」状态缺少以下细节：
1. **队列行为** — 发起的请求是自动排队、静默失败、还是提示用户？
2. **重试策略** — 网络恢复后如何重试排队请求？立即重试？批量重试？
3. **冲突解决** — 离线状态下用户修改了本地数据，但云端（如未来 V2 有同步）同时被修改，谁优先？
4. **UI 反馈** — 不同分级下用户界面的指示器如何变化？
5. **API 端点映射** — 哪些具体 API 端点在各个分级下可用/不可用/降级？

### 为什么需要细化

桌面 APP 的离线体验是核心 UX 差异点（与 Web APP 相比）。知命作为本地优先 APP，用户在以下场景会频繁经历网络变化：
- 地铁/电梯（短暂离线）
- 信号弱区域（有限连接，高延迟/低带宽）
- 飞行模式（完全离线）
- 切换 Wi-Fi（网络漂移）

在这些场景下，用户期望：
- 不会丢失已输入的数据
- 离线时能继续浏览已有档案
- 网络恢复后自动同步，无需手动操作
- 清晰的 UI 指示器告知当前状态

## 已确认方案：方案 A — 3 级基础细化（V1 采用）

### 决策信息

| 字段 | 内容 |
|------|------|
| **决策方** | 产品负责人（Product Owner） |
| **决策时间** | 2026-07-16 |
| **确认方案** | 方案 A：3 级基础细化 |
| **决策依据** | 见下方「决策理由」 |

### 方案内容

保持现有的 **3 级分层（Online / Limited / Offline）**，为每一级明确定义以下五个维度，补充到 `08_系统架构设计.md` §8.1.2：

| 维度 | Online | Limited | Offline |
|:----:|--------|---------|---------|
| **队列策略** | 无需排队，请求直达 | 写操作自动进入内存队列（最大 50 条）；队列内容持久化至本地 SQLite；读操作优先尝试网络，失败回退缓存 | 所有写操作进入本地队列（内存 + SQLite 持久化）；读操作全部走本地缓存 |
| **重试策略** | 不适用 | 网络恢复后自动触发 FIFO 回放，每条队列间隔 500ms；单条失败最多重试 3 次（间隔 2s / 5s / 10s），全部失败后标记为待处理 | 同 Limited 重试策略，网络恢复后统一回放 |
| **冲突解决规则** | 不适用（始终同步最新数据） | **本地优先**：本地修改时间戳晚于云端则覆盖云端，反之忽略；同步冲突提示写入操作日志不影响当前操作 | 同 Limited |
| **API 端点可用性** | 全部 API 端点可用 | 排盘、档案读写、本地规则分析 → 可用；AI 深度分析（LLM 调用）→ 排队，网络恢复后自动执行；云端资源查询 → 降级为本地缓存 | 仅排盘和档案浏览（本地缓存） → 可用；所有远端 API → 不可用，自动排队 |
| **UI 指示器** | 顶部状态栏绿色圆点 + "已连接" | 顶部状态栏黄色圆点 + "有限连接"，队列中有待发送项时显示小数字角标；首次进入 Limited 时 toast 提示"部分功能暂不可用，网络恢复后自动同步" | 顶部状态栏灰色圆点 + "离线模式"，底部工具栏提示"离线模式 - 仅可浏览已有档案" |

### 决策理由

产品负责人确认选择 **方案 A (3 级基础细化)** 作为 V1 的离线策略，理由如下：

1. **V1 目标市场验证** — V1 阶段主要面向命理爱好者和早期体验用户，而非企业级用户。该群体对离线体验的预期为「不丢数据、能用即可」，而非 Notion/Obsidian 级的精细化离线 UX。
2. **开发资源聚焦** — Phase 3 的核心工作流量较大（AI 动态分析 6 步流程、档案管理等），离线功能不应成为 Phase 3 的资源瓶颈。方案 A 实现成本低，能快速关闭当前风险项。
3. **用户场景匹配** — 桌面 APP（V1 载体为 Windows 客户端）的离线场景以「短暂断网」为主，3 级分层 + 有限连接的队列/重试机制足以覆盖 90% 的用户体验需求。
4. **升级路径清晰** — 方案 A 的 3 级分层结构是方案 B (5 级) 和方案 C (3+3 子状态) 的超集。未来 V2 可在当前框架上平滑扩展，无需推翻重来。

> **后续升级方向（V2 及以上）：** 若用户反馈显示"有限连接"状态下的体验不足，可参照方案 B (5 级) 将 Limited 拆分出 Degraded / Minimal 子级，或在方案 A 的 API 可用性矩阵中增加端点级别的降级策略。

## 曾评估的替代方案

### 方案 B：5 级精细分层

**核心思路：** 将 3 级扩展为 5 级（Online → Degraded → Limited → Minimal → Offline），覆盖从完全在线到完全离线的全频谱，每级绑定更精确的网络条件阈值。

| 优势（为何当时被考虑） | 劣势（为何 V1 未采用） |
|------------------------|------------------------|
| 更精细的网络感知，用户体验更平滑 | 实现复杂度大幅提升（5 种状态的状态机 + UI 指示器） |
| Degraded 级可降级使用本地推理而非完全禁用 AI | 网络探测算法需要额外工作量（不仅是 `navigator.onLine`） |
| Minimal 级提供只读模式保护数据一致性 | 测试覆盖 5 种状态的组合爆炸 |
| 各分级过渡更平滑，用户感知不会跳变 | V1 场景下过度设计，开发成本远超收益 |

**决策结果：** 不采用。保留为 V2/V3 移动端场景的候选升级方案。

### 方案 C：3+3 子状态方案（3 主级 x 3 子级）

**核心思路：** 保持对外展示的 3 级主分级（UI 简洁），内部细化每级 3 种子状态，共 9 种内部行为模式，系统自动在子状态间转换。

| 优势（为何当时被考虑） | 劣势（为何 V1 未采用） |
|------------------------|------------------------|
| 对外 UX 简洁（3 种状态图标），对内逻辑精细（9 种子状态行为） | 内部实现复杂度最高（状态机 + 子状态转换逻辑） |
| 子状态转换由系统自动完成，用户无感知 | 子状态判定需要精确的带宽/延迟/断连频率阈值 |
| 兼顾方案 A 的 UX 简洁和方案 B 的控制精细度 | 测试复杂度高（9 种组合 x 转换路径） |
| | 子状态对开发者不透明，调试困难 |
| | 在 V1 阶段属于严重的过度设计 |

**决策结果：** 不采用。保留为产品成熟期（Phase 4 及以上）精细运营的候选方案。
