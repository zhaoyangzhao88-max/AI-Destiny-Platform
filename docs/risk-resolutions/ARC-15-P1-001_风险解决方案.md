## 根因分析

### 冲突位置

| 位置 | 当前内容 | 与 ADR-018 的冲突 |
|------|---------|-----------------|
| **8.1.4 Windows APP 作为 V1 入口** 选型表 | 全部「待确认」；候选：Tauri、Electron、Flutter Desktop、C# WinUI3 | ADR-018 已决策 PySide6+Vue，无上述候选 |
| **8.7 技术栈初步建议** 前端行 | React / Vue / Flutter | ADR-018 已确定 PySide6 为主框架 + Vue 渐进引入 |
| **8.7 技术栈初步建议** 后端行 | Python (FastAPI) / Node.js | ADR-018 已确定 Python 核心服务 |
| **8.3.2 渐进引入原则** 末句 | "整体 Windows APP 框架选型（Tauri/Electron/Flutter）仍为【待确认】，详见 P1-001" | 此处已自引用本风险，说明作者已知冲突但未修复 |

### 根本原因

`08_系统架构设计.md` 的撰写时间线如下：
1. **v0.1 初始版本**：编写 8.1.4 和 8.7 节时，技术选型尚未决策，采用「待确认」占位是合理做法
2. **ADR-018 决策后**（2026-07-08）：8.3 节已更新为 PySide6+Vue 架构（8.3.1/8.3.2），但 8.1.4 和 8.7 节未被同步更新
3. **v0.2 部分更新**：8.3.2 末尾新增了自引用注释 `详见 P1-001`，表明问题已被识别但未被修复

这是一个典型的**部分更新遗漏**问题：架构文档的多个章节引用了同一组技术选型信息，但更新时只覆盖了核心章节（8.3），遗漏了较早的技术选型讨论章节（8.1.4）和汇总章节（8.7）。

### 影响范围

- 新 AI 读取 `08_系统架构设计.md` 时，会在 8.1.4 节读到 "待确认"，形成错误的第一印象
- 8.7 节作为技术栈汇总表，对开发者决策有重要参考价值，错误选项会误导实现
- 自引用注释 "详见 P1-001" 虽然标注了问题，但本质上是将本应修复的问题"外包"给风险跟踪，增加了读者的认知负担

## 解决方案

### 方案描述

对 `08_系统架构设计.md` 的两处冲突进行对齐修改，统一为 ADR-018 确定的 PySide6+Vue 混合架构，候选方案移入附录作为历史参考。

#### 修改 8.1.4 节

**当前位置**：`### 8.1.4 Windows APP 作为 V1 入口`

将当前「待确认」选型表：

```
| 方案 | 跨平台 | 本地能力 | 学习成本 | 推荐度 |
|------|--------|----------|----------|--------|
| 待确认 | 待确认 | 待确认 | 待确认 | —     |
```

替换为已决策的技术选型确认表：

```
| 方案 | 跨平台 | 本地能力 | 学习成本 | 决策状态 |
|------|--------|----------|----------|---------|
| PySide6 + Vue（混合架构） | Windows V1（Web V2）| ✅ 强（文件系统/SQLite/系统集成） | 中（需同时掌握 PySide6 + Vue）| ✅ 已决策（ADR-018）|
```

将候选方案文本：

```
> ⚠️ **具体技术框架选型** ——【待确认】  
> 候选：Tauri（Rust 前端）、Electron（JS 生态）、Flutter Desktop、C# WinUI3
```

替换为决策说明文本：

```
> ✅ **技术框架已决策**：PySide6 + Vue 渐进式混合架构（ADR-018，2026-07-08）  
> 详情见 [8.3 客户端技术架构选择](#83-v1客户端技术架构选择)  
> 候选方案（Tauri/Electron/Flutter/C#）已移入附录以供参考
```

#### 修改 8.7 节

**当前位置**：`## 8.7 技术栈初步建议`

将当前「待确认」建议表：

```
> ⚠️ **以下为初步建议，全部【待确认】**

| 层级 | 技术选项（建议） | 备注 |
|------|-----------------|------|
| **前端** | React / Vue / Flutter | 需要跨平台支持？ |
| **后端** | Python (FastAPI) / Node.js | 需要 AI 模型调用 |
| **数据库** | PostgreSQL + MongoDB | 结构化数据 + 档案快影 |
| **AI 模型** | LLM API（待定提供商） | 需要支持中文命理推理 |
| **排盘库** | 自研 / 引用 bazi-master-analysis 脚本 | 已有 Python 排盘脚本可供参考 |
```

替换为已决策的技术栈确认表：

```
> ✅ **V1 技术栈已确认**（ADR-014~018）

| 层级 | V1 技术方案 | 备注 | 决策依据 |
|------|------------|------|---------|
| **桌面框架** | PySide6 | Windows 桌面入口、系统集成、本地应用管理 | ADR-018 |
| **富交互 UI** | Vue（渐进引入） | 复杂交互页面、AI 聊天、数据可视化，按需经 QWebEngineView 嵌入 | ADR-018 |
| **后端服务** | Python (FastAPI) | 排盘计算、AI Agent 编排、用户档案管理、知识库 | ADR-017 / ADR-018 |
| **数据库** | SQLite（V1）→ PostgreSQL（V2） | V1 本地优先，V2 云端扩展 | ADR-014 |
| **AI 模型** | LLM API（待定提供商） | 需要支持中文命理推理 | ADR-013 |
| **排盘库** | 通过 BaziEngine 接口接入（见 8.8.1） | 具体实现待选型 | ARC-15-P1-006 |
```

#### 候选方案附录

在 `08_系统架构设计.md` 末尾新增附录：

```
## 附录 D：候选技术方案参考

> 以下候选方案在 ADR-018 决策过程中被评估但未选中，
> 保留以供未来版本（V2 Web 扩展、跨平台支持）参考。

### 前端框架候选
| 方案 | 未选中原因 |
|------|-----------|
| **Tauri（Rust 前端）** | Rust 学习曲线陡峭，与 Python 技术栈不统一；V1 无跨平台需求 |
| **Electron（JS 生态）** | 打包体积过大（Chromium 捆绑 100MB+），与本地优先架构理念不完全契合 |
| **Flutter Desktop** | UI 表现力强但 Python 集成复杂，生态系统成熟度不足 |
| **C# WinUI3** | 绑定 Microsoft 生态，不利于未来 Web/移动扩展 |

### 后端框架候选
| 方案 | 未选中原因 |
|------|-----------|
| **Node.js** | 与 Python 技术栈冲突，需要维护两套运行时 |
```

#### 清理 8.3.2 残余自引用

将 8.3.2 渐进引入原则末句：

```
整体 Windows APP 框架选型（Tauri/Electron/Flutter）仍为【待确认】，详见 P1-001（08_架构 8.1.4 / 8.7 对齐）。
```

修改为：

```
V1 Windows APP 技术框架已由 ADR-018 决策为 PySide6 + Vue 渐进式混合架构，不再开放选型讨论。
```
