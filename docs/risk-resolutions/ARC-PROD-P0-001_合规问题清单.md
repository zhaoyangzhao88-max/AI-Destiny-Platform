## 问题组一：AI 服务备案与算法合规

### 背景

知命 AI 的技术架构为：**Python 核心服务（本地进程）** → 通过 **HTTP API + WebSocket 双通道** 与表示层通信 → 按需调用 **第三方 LLM API** 进行 AI 深度分析。排盘引擎（四柱八字计算）在本地运行，不依赖云端。

架构详见 `docs/08_系统架构设计.md` §8.1.3（云端扩展原则）、§8.9（客户端通信架构）。

### 问题清单

#### Q1-1：第三方 LLM API 调用是否触发《生成式人工智能服务管理暂行办法》的备案义务？

**背景：**

知命的 AI 分析能力通过调用第三方 LLM API（如 OpenAI、国内大模型 API）实现。知命本身不训练、不微调基础模型，仅通过 Prompt 工程调度 LLM 完成命理分析。

**具体问题：**

1. 这种"调用第三方 API + Prompt 调度"的模式，是否属于《生成式人工智能服务管理暂行办法》管辖的"生成式人工智能服务提供者"？
2. 如果属于，知命 V1（Windows 桌面 APP，本地进程调用远程 API）是否需要履行算法备案义务？
3. 如果 LLM API 提供商（如国内大模型厂商）已经完成备案，知命是否可以复用其备案，还是需要独立备案？
4. 是否需要向用户明确告知"当前对话由 AI 生成"？告知方式有没有强制性要求（如实时标注 vs 首次告知）？

**参考法规：** 《生成式人工智能服务管理暂行办法》（2023）、《互联网信息服务算法推荐管理规定》

---

#### Q1-2：自建排盘引擎是否需要单独算法备案？

**背景：**

知命的排盘引擎（四柱八字计算）是自建的本地规则引擎，基于确定性算法（节气计算、干支转换等），不涉及机器学习或大模型。该引擎在用户本地设备上运行，不上传至云端。

**具体问题：**

1. 这种基于传统历法规则的确定性本地算法，是否属于《互联网信息服务算法推荐管理规定》中需要备案的"算法推荐技术"？
2. "本地运行、不收集用户数据"的特性是否影响备案义务的判断？
3. 如果排盘引擎未来升级为"规则引擎 + AI 辅助校正"的混合模式，备案要求是否会变化？

## 问题组二：多渠道分发与合规

### 背景

知命 V1 定位为 **Windows 桌面 APP**（ADR-015），通过安装包分发。V2/V3 规划扩展至 Web 端和移动端（小程序/APP）。分发渠道不同，合规要求可能不同。

详见 `docs/08_系统架构设计.md` §8.1.4（Windows APP 作为 V1 入口）、§8.2.5（未来 Web 扩展能力）。

### 问题清单

#### Q2-1：Windows 桌面 APP 分发是否需要前置审批？

**具体问题：**

1. Windows 桌面 APP（通过官网下载 / Microsoft Store 分发）是否需要申请《互联网信息服务许可证》（ICP 证）或其他前置审批？
2. 如果 APP 仅提供本地计算服务，AI 分析功能通过调用 API 实现，在许可证申请上是否与完全自建服务器的 Web 服务有区别？
3. 知命涉及"命理分析"内容，是否属于需要前置审批的特殊行业？是否需要《网络文化经营许可证》或其他行业许可？
4. 如果未来通过 Microsoft Store 分发，是否涉及跨境分发合规问题（如美国出口管制、中国数据出境）？

#### Q2-2：未来扩展至小程序/Web 渠道的合规差异

**具体问题：**

1. 如果未来上线微信小程序，微信平台对"命理/算命"类小程序的审核政策是什么？是否有已知的准入限制？
2. 微信小程序/Web 端与 Windows 桌面端在用户协议、隐私政策、未成年人保护等方面的要求有何差异？
3. Web 端是否需要额外的 ICP 备案/许可证申请？
4. 如果未来涉及付费功能（ADR-003 免费+付费模式），是否需要《增值电信业务经营许可证》？

## 问题组三：命理内容合规与 AI 角色边界

### 背景

知命 AI 的角色设定为"东方智慧 × 人工智能的人生顾问"（`docs/05_知命_Persona角色设定.md`），明确禁止替用户做重大决策、断言未来具体事件、制造焦虑或贩卖恐惧。产品定位为"去迷信化"（`docs/01_项目简介.md` §1.5）。

### 问题清单

#### Q3-1：现有"禁止行为"清单是否足以满足合规要求？

**背景：**

05_知命_Persona角色设定.md §5.7 定义了以下禁止行为：
- ❌ 替用户做人生重大决策
- ❌ 断言未来具体事件
- ❌ 制造焦虑或贩卖恐惧
- ❌ 冒充真人有神通力
- ❌ 泄露用户隐私信息
- ❌ 超出知识边界回答命理问题

**具体问题：**

1. 从合规角度，上述禁止行为清单是否需要补充？例如：
   - 是否需要强制性免责声明（如"本分析仅供娱乐/参考，不构成专业建议"）？
   - 是否需要对命理结论附加"确定性等级"标注（如 ADR-012 数据三层体系中的事实/推测/待验证标记）？
   - 是否需要明确禁止向未成年人提供命理分析服务？
   - 是否需要禁止涉及健康、医疗、法律等专业领域的建议？
2. 如果 AI 在对话中出现了超出禁止清单但仍有合规风险的内容（如用户诱导 AI 做出投资建议），责任归属于谁（产品运营方 / AI 模型提供商 / 用户）？

#### Q3-2：AI 命理分析是否属于"迷信内容"监管范畴？

**具体问题：**

1. 中国现行法规对"算命/命理"类互联网内容服务的监管态度是什么？是否存在明确的禁止性规定，还是处于灰色地带？
2. 知命以"人生档案顾问"定位，强调"去迷信化"（分析仅作参考，不替用户决策），这种定位在合规审查中是否有帮助？
3. 如果有用户投诉 AI 命理内容"传播迷信"，产品方可能面临什么法律风险？如何提前防范？
4. 境外 LLM API 生成的命理分析内容，是否受中国内容审核法规的管辖？

#### Q3-3：是否需要引入人工审核机制？

**具体问题：**

1. 对于 AI 生成的命理分析内容，法规是否要求配备人工审核？
2. 如果需要，人工审核的最低比例和审核标准是什么？
3. V1 阶段（单人开发团队，无专职审核人员）是否可以通过技术手段（关键词过滤、敏感话题阻断）替代人工审核？

## 问题组四：用户数据合规与本地优先架构的相容性

### 背景

知命 V1 采用 **"本地优先 + 云端扩展"** 混合架构（ADR-014）。核心档案数据存储在用户本地设备（SQLite 数据库），仅当用户发起 AI 深度分析时，将必要的上下文发送至云端 LLM API。数据上传遵循最小化规则（不上传完整人生档案，仅上传当前分析所需的最小上下文）。

详见 `docs/08_系统架构设计.md` §8.1.2（本地优先原则）、§8.1.3（云端扩展原则）、§8.5（数据三层体系）、§8.6（数据版本历史机制）。

### 问题清单

#### Q4-1：用户出生信息在个保法下的处理要求

**背景：**

用户注册时需提供出生日期、出生时间（可能精确到分钟）、出生地点、性别等信息。这些信息用于八字排盘和 AI 分析。其中出生时间允许用户填写"不知道"（ADR-011）。

**具体问题：**

1. 出生日期、精确出生时间、出生地点是否属于《个人信息保护法》中的"敏感个人信息"？判断标准是什么？
2. 如果属于敏感个人信息，需要满足哪些条件才能合法处理？（单独同意、目的明确、必要性说明等）
3. 出生时间未知（ADR-011）的情况下，用户填写"不知道"是否影响数据处理的合法性基础？
4. 未来如果通过 AI 与用户的对话内容推断出用户之前未声明的出生时间，这种"推断信息"在合规上如何处理？（涉及 ADR-012 数据三层体系中的"推测"层级）

#### Q4-2：本地优先架构与个保法的相容性

**具体问题：**

1. 用户数据全程存储在本地设备（不传输到服务器），是否意味着不适用《个人信息保护法》中关于"个人信息处理者"的跨境传输、存储期限等规定？
2. "本地存储"是否可以被视为一种合规措施（数据最小化、目的限制原则的体现）？
3. 如果用户数据仅存储在本地，产品方是否需要仍履行《个人信息保护法》中的"告知-同意"义务？告知的方式在桌面 APP 场景下可以是"隐私政策弹窗 + 首次启动引导"吗？
4. 本地 SQLite 数据库中的数据保护措施（加密要求、访问控制）需要达到什么级别？

#### Q4-3：LLM API 调用中的数据跨境与数据委托处理

**背景：**

当用户发起 AI 深度分析时，Python 核心服务将分析上下文（脱敏后的核心命理数据）发送至第三方 LLM API。数据可能传输至境外服务器（取决于 LLM API 提供商的部署位置）。

**具体问题：**

1. 向第三方 LLM API 发送分析上下文，属于《个人信息保护法》中的"委托处理"还是"向第三方提供个人信息"？两者的合规义务差异很大。
2. 如果 LLM API 提供商位于境外，是否触发《个人信息保护法》的**跨境传输**规定？是否需要通过安全评估、标准合同或认证？
3. 如果选择国内 LLM API 提供商（如百度文心、阿里通义千问），是否完全规避跨境传输问题？
4. "数据脱敏"应达到什么标准，才能降低向 LLM API 发送数据的合规风险？目前的技术方案是将出生信息转换为干支（脱敏为四柱数据，不发送原始出生时间），这种脱敏是否足够？
5. 如果 LLM API 提供商在对话过程中收集了用户输入用于模型训练（如默认的对话日志），知命是否需要主动告知用户并取得同意？

#### Q4-4：数据删除与 GDPR 相容性

**背景：**

知命未来可能面向海外用户（虽然 V1 聚焦中文用户）。ADR-014 的本地优先架构在数据删除方面需要特别考虑。

**具体问题：**

1. V1 的本地优先架构是否天然满足 GDPR 的"数据最小化"和"存储限制"原则？
2. 如果用户要求删除数据（被遗忘权），在全快照版本历史机制（`docs/08_系统架构设计.md` §8.6）下，如何处理历史版本快照中的用户数据？
3. 本地存储的数据删除请求，产品方是否需要提供工具让用户自行删除，还是需要远程指令删除？
4. 如果未来启用云同步/备份功能，数据删除的合规复杂度会如何变化？
