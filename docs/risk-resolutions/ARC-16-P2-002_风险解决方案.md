## 根因分析

### 问题定位

> 原文引用审计报告 §4.3.2：
> "| 5 | **废弃文档流程未定义** | 🟢 P2 | 🔴 Deprecated 状态存在于状态表中，但没有定义'什么情况下可以废弃一份文档、废弃后如何处理引用关系'"

### 根本原因

`12_AI协作开发规范.md` §5.2 的文档状态表定义了 🔴 Deprecated 状态，但仅简单标注"由产品负责人"可修改，缺少以下关键定义：

1. **判定标准** —— 什么条件下可以判定一份文档为 Deprecated？
2. **操作步骤** —— 标记为 Deprecated 需要执行哪些具体动作？
3. **引用处理** —— 废弃后如何处理其他文档对该文档的引用？
4. **通知机制** —— 谁需要知道这份文档已废弃？
5. **清理时间线** —— Deprecated 文档保留多久？何时可以物理删除？

当前文档体系中尚无文档需要废弃（所有文档均处于 Draft 或 Accepted 状态），但随着 Phase 3 代码开发的推进和文档体系的迭代，未来可能出现以下场景：
- 某份设计文档被新的设计取代（如 V1 设计 → V2 设计）
- 架构方向变更导致旧 ADR 不再适用
- 合并/拆分文档导致旧编号废止

### 影响范围

- 未来文档迭代时缺乏废弃操作的标准流程
- 被废弃文档的引用者（其他文档、AI、开发者）无法获取准确的废弃信息和替代指引
- 废弃文档长期存留在文档体系中，造成认知负担

---

## 解决方案

### 方案描述

在 `12_AI协作开发规范.md` §5.2「文档状态定义」中补充废弃文档流程，作为新的子节。流程定义包括：前置条件判定、操作步骤清单、通知机制、清理时间线。

---

#### 新增内容：废弃文档流程（插入到 §5.2 末尾）

```markdown
### 5.2.2 废弃文档流程

#### 5.2.2.1 废弃前置条件

文档可被标记为 🔴 Deprecated 当满足以下**任一**条件：

| # | 条件 | 说明 | 示例 |
|---|------|------|------|
| 1 | **替代文档已存在** | 有新的正式文档取代了该文档的内容和定位 | V2 架构设计文档取代 V1 架构文档 |
| 2 | **内容已过时** | 文档中的决策、描述或数据因项目方向调整已不再适用 | 某项技术选型被正式否决时，该技术选型文档 |
| 3 | **已被合并** | 该文档的内容已被吸收到其他文档中，独立存在不再必要 | 某专项设计文档的内容合并入主架构文档后 |
| 4 | **已被拆分** | 该文档的内容被拆分到多个新文档中，原文档编号废止 | 通用规范拆分为"前端规范"和"后端规范" |
| 5 | **范围被否决** | 文档所定义的模块/功能/阶段在项目决策中被正式取消 | 某计划中的功能模块被取消，其设计文档 |

> 废弃判定由产品负责人做出。AI Agent 可以提出废弃建议，但不得自行标记 Deprecated。

#### 5.2.2.2 废弃操作步骤

文档被判定为废弃时，须按以下步骤操作：

```
┌─────────────────────────────────────────────────────┐
│  ① 废弃判定         产品负责人确认满足前置条件      │
│         ↓                                            │
│  ② 文档头部更新     添加 Deprecated 标记 + 替代指引  │
│         ↓                                            │
│  ③ 添加废弃横幅     在文档顶部添加醒目提示          │
│         ↓                                            │
│  ④ 交叉引用修复     更新所有引用该文档的交叉引用    │
│         ↓                                            │
│  ⑤ 通知记录         在 11_协作日志 中记录废弃决定    │
│         ↓                                            │
│  ⑥ 状态同步         更新 00_文档地图（如有）         │
│         ↓                                            │
│  ⑦ 清理倒计时       进入保留期，到期后评估物理删除   │
└─────────────────────────────────────────────────────┘
```

**步骤详解：**

**步骤 ①：废弃判定**
- 产品负责人确认条件满足（见 5.2.2.1）
- 决定废弃后通知当前轮次的 AI Agent

**步骤 ②：文档头部更新**
```markdown
> **版本:** v<原版本号> | **日期:** YYYY-MM-DD | **状态:** 🔴 Deprecated
> **废弃日期:** YYYY-MM-DD
> **替代文档:** [新文档名称](./新文档路径.md)
> **废弃理由:** [简述——如"被 ADR-XXX 取代" / "内容已合并到 08_架构"]
```

**步骤 ③：添加废弃横幅**
在文档标题下方、目录之前，添加醒目提示横幅：

```markdown
> **🔴 本文档已废弃（Deprecated）**
>
> 自 YYYY-MM-DD 起本文档不再维护。
> **替代文档：** [新文档名称](./新文档路径.md)
> **变更记录：** `docs/11_AI协作日志.md` YYYY-MM-DD 条目
```

**步骤 ④：交叉引用修复**
- 搜索项目中引用此文档的所有位置（`grep "文档名" docs/`）
- 每条引用按以下方式处理：

| 引用场景 | 处理方式 |
|---------|---------|
| 引用具体内容（可找到替代来源） | 直接更新为引用替代文档的对应章节 |
| 引用文档整体（无直接替代） | 改为引用替代文档，并标注"相关内容已迁移" |
| 引用已废弃决策（无可替代） | 保留引用但添加 `（📌 该文档已废弃，所引用的决策可能不再有效）` 标注 |

**步骤 ⑤：通知记录**
在 `11_AI协作日志.md` 中追加以下记录：

```
--- 文档废弃记录 ---
废弃文档:    docs/XX_文档名.md
废弃日期:    YYYY-MM-DD
废弃理由:    [条件 1~5 中的对应条件]
替代文档:    docs/YY_新文档名.md
执行动作:    头部标记 / 横幅添加 / 交叉引用修复（N 处）
决策人:      产品负责人
```

**步骤 ⑥：状态同步**
- 如果存在 `docs/00_AI_CONTEXT_INDEX.md`（文档地图），同步更新废弃状态
- 在文档地图中对应条目添加 `🔴 Deprecated` 标记

**步骤 ⑦：清理时间线**
- 见下文 5.2.2.3 清理规则

#### 5.2.2.3 清理时间线

Deprecated 文档不会立即物理删除，而是经过一段保留期后评估：

| 阶段 | 时间 | 状态 | 可操作内容 |
|------|------|------|-----------|
| **保留期** | Deprecated 之日起 3 个 Sprint | 🔴 Deprecated | 可读取内容，不可修改；交叉引用修复期 |
| **评估点** | 3 个 Sprint 后 | — | 产品负责人评估是否仍需要保留 |
| **可删除** | 评估通过后 | — | 物理删除文件，或转为归档存储 |

**保留期内：**
- 文档保留在 docs/ 目录中，头部带 Deprecated 横幅
- AI 和开发者在搜索或引用时能看到废弃状态和替代指引
- 其他文档的交叉引用应在保留期内完成修复

**评估后决策：**

| 评估结论 | 操作 |
|---------|------|
| 不再需要 | 物理删除文件，在 `11_协作日志` 中记录删除 |
| 需保留参考 | 保留 Deprecated 状态，删除横幅下方的"替代指引"改为"历史参考"，移至 docs/archive/ 目录（如该目录存在） |
| 需要恢复 | 产品负责人可决定取消 Deprecated 标记，恢复为 Draft 或对应状态 |

> **重要：** 物理删除文件前，必须确保所有交叉引用已修复完成，并在 `11_协作日志.md` 中记录删除操作。禁止在未确认引用关系的情况下删除任何文件。

#### 5.2.2.4 废弃文档的特殊情况

**ADR 记录中的 ADR 废弃：**
- 单个 ADR（如 ADR-XXX）废弃不废弃整份 04_ADR_产品决策记录.md
- 在 04_ADR 的 ADR 条目头部将状态改为 🔴 Deprecated
- 在附录 ADR 状态索引表中同步更新
- 在废弃的 ADR 条目上方添加引用替代 ADR 的说明

**协作日志不废弃：**
- `11_AI协作日志.md` 作为历史记录保留，不标记 Deprecated
- 即使项目进入新阶段，旧日志仍作为可追溯的审计证据保留
```
