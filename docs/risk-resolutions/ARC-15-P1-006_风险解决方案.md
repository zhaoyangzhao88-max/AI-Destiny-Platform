### 2.2 bazi-consumer 应用层耦合度实测（Sprint 12）

- bazi-consumer 与 bazi-skill-dist **零进程耦合**，唯一交互点是一处 `fetch()` HTTP 调用（`src/runtime/api/analyze.ts`）

## 三、解决方案：抽象接口设计 + 已知缺陷的强制校验

### 3.1 接口设计（保留 v1.0 的骨架，新增缺陷防护）

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass
from typing import Optional
from datetime import datetime

@dataclass
class BaZiRequest:
    """排盘请求参数"""
    birth_year: int
    birth_month: int
    birth_day: int
    birth_hour: int
    birth_minute: int
    calendar_type: str              # "solar" | "lunar"
    gender: str                     # "male" | "female"
    longitude: float                # 经度（真太阳时校准，必填——见3.2）
    latitude: float                 # 纬度
    timezone: str                   # 时区标识
    birth_time_unknown: bool = False

@dataclass
class Pillar:
    heavenly_stem: str
    earthly_branch: str
    hidden_stems: list[str]
    na_yin: str

@dataclass
class BaZiResult:
    year_pillar: Pillar
    month_pillar: Pillar
    day_pillar: Pillar
    hour_pillar: Pillar
    day_master: str
    gender: str
    birth_time_unknown: bool
    great_fortune_start_age: int
    great_fortune_cycle: list[dict]
    yong_shen: Optional[str] = None      # 用神——新增，bazi-skill-dist 完全缺失此字段
    xi_shen: Optional[str] = None        # 喜神——新增
    ji_shen: Optional[str] = None        # 忌神——新增
    engine_name: str = ""
    engine_version: str = ""
    calculated_at: str = ""

class BaziEngine(ABC):
    """排盘引擎抽象基类

    任何实现（含未来引入的 bazi-skill-dist 重写版）必须满足：
    - 正确处理立春换年、节气换月（见 self_check 强制校验）
    - 起运年龄计算不得使用占位/硬编码节气数据
    - 用神/喜神/忌神字段可为空，但不可静默省略——必须显式返回 None 并在
      engine_info 中声明 supports_yongshen: bool
    """

    @abstractmethod
    def paipan(self, request: BaZiRequest) -> BaZiResult:
        pass

    @abstractmethod
    def validate(self, request: BaZiRequest) -> list[str]:
        pass

    @property
    @abstractmethod
    def engine_info(self) -> dict:
        """必须包含以下字段，供调用方判断引擎能力边界：
        {
            "name": str, "version": str,
            "handles_jieqi": bool,       # 是否正确处理节气换月
            "handles_lichun": bool,      # 是否正确处理立春换年
            "handles_true_solar_time": bool,  # 是否处理真太阳时
            "supports_yongshen": bool,   # 是否支持用神推导
        }
        """
        pass

    def self_check(self) -> list[str]:
        """引擎自检——用一组已知边界案例验证准确性，返回失败项列表。
        建议至少包含：2026-02-01（立春前，验证是否误判年柱）、
        2026-03-05（惊蛰前，验证是否误判月柱）。
        已知 bazi-skill-dist 在这两个案例上会失败，
        任何候选引擎接入前必须先跑通此方法。
        """
        return []
```

### 3.2 为什么把 `longitude`/`timezone` 设为必填、新增 `self_check()`

这不是通用最佳实践式的"锦上添花"，是**直接针对 Sprint 11 发现的具体缺陷**做的强制约束：

- bazi-skill-dist 的时柱计算函数名涉及"真太阳时定位纠偏"，但实际代码里根本没有经度参数——接口层面把 `longitude` 设为必填，可以在编译期就挡住这类"名不副实"的实现混进来
- `self_check()` 强制任何新接入的引擎先跑一遍已知边界案例（立春/节气临界点），如果 bazi-skill-dist 的重写版本某天又退化回占位实现，这一步能立刻暴露，而不是要等到线上用户投诉才发现

### 3.3 工厂模式（沿用 v1.0）

```python
class BaziEngineFactory:
    _engines: dict[str, type[BaziEngine]] = {}

    @classmethod
    def register(cls, name: str, engine_cls: type[BaziEngine]) -> None:
        cls._engines[name] = engine_cls

    @classmethod
    def create(cls, name: str, **kwargs) -> BaziEngine:
        if name not in cls._engines:
            raise ValueError(f"未知引擎: {name}")
        engine = cls._engines[name](**kwargs)
        failures = engine.self_check()
        if failures:
            raise RuntimeError(f"引擎 {name} 自检未通过: {failures}")
        return engine
```

### 3.4 推荐落地路径

沿用 Sprint 11 的路径 C（复用架构与数据模型，重写计算核心），具体拆分：

| 可直接复用 | 必须重写 | 新增设计 |
|---|---|---|
| `BaziChart`/`Pillar` 数据模型思路 | 四柱计算核心（换用真正处理节气的万年历库） | 用神/喜神/忌神推导引擎 |
| 藏干表常量 | 起运年龄计算（替换占位节气 provider） | `self_check()` 边界案例校验机制 |
| 十神计算逻辑 | — | `engine_info` 能力声明字段 |
| 大运顺逆规则 | — | — |

工作量估算：6-10 周（含用神引擎从零设计），受益于 bazi-consumer 前端零耦合，**前端改动为零，后端改动局限于计算核心**。
