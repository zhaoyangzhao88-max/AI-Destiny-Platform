## 根因分析

### 问题定位

> 原文引用（[sprint13-risk-application-and-extraction.md](../sprints/sprint13-risk-application-and-extraction.md) §ARC-16-P1-001）：
>
> **问题 #1：** 三份文档的"先读清单"不一致
> - CLAUDE.md 列出 P0=5 份，HANDOVER 列出 Step1=5 份但清单与 CLAUDE 有微小排序差异，12_规范 §3.1 列出不同子集（HANDOVER + ADR + 开发状态 + 日志）

### 三份清单差异明细

| 文档 | 位置 | 列出文档 | 数量 | 差异点 |
|------|------|---------|:----:|--------|
| **CLAUDE.md** | §开发前必读文档 | 5 P0 + 8 P1（含 HANDOVER/01/02/04/05 + 03/06/07/08/11/12/13/14） | **13** | 最完整，含 P0/P1 优先级区分 |
| **HANDOVER.md §6.1** | §6.1 必须遵守 | 01 + 02 + 04 + 05 + HANDOVER | **5** | 同 Step1，不含 03/06/07/08/11/12/13/14 |
| **HANDOVER.md §7 Step 1** | §7 启动说明 Step 1 | 同 §6.1 的 5 份 + HANDOVER | **5** | 与 §6.1 一致但不含 03/06/07/08/11 |
| **HANDOVER.md §7 Step 2** | §7 Step 2 | 03 + 06 + 07 + 08 | **4** | 不含 11/12/13/14，12 本身的规范不在本清单 |
| **12_规范 §3.1 Step 1** | §3.1 通用工作流 | HANDOVER + ADR + 开发状态 + 协作日志 | **4** | 最小子集，不含 01/02/05 |
| **12_规范 Appendix B** | 附录 B 文档快速索引 | 10 个"想做什么→先看什么"条目 | **10** | 用途导向，非统一清单 |

### 根本原因

1. **缺乏单一权威清单（SSOT）**：CLAUDE.md 在项目根目录，HANDOVER 在 docs/ 目录，12_规范也在 docs/ 目录——三者各自独立维护"先读清单"，无交叉引用机制。编写者各自按需列出子集，积累多个版本后差异越来越大。

2. **HANDOVER 意图重叠**：HANDOVER.md 既要充当"项目总览"又要充当"AI 启动指南"还要充当"文档索引"，同一文件内 §6.1（协作规则章节）、§7 Step 1、§7 Step 2 三处分别列出不同子集，读者难以判断哪份是完整的。

3. **增量维护无协调**：每次新文档加入（如 13_API 契约、14_WebSocket 协议、12_规范本身）时，各入口文档分别更新，但更新策略不同（CLAUDE.md 全部追加，HANDOVER §3 追加但不更新 §6.1/§7，12_规范仅更新 §5.1 文档体系总览）。

### 影响范围

- 新 AI 进入项目时，如仅读取 HANDOVER §6.1 的 5 份清单，会遗漏 03/06/07/08/11/12/13/14 等 8 份关键文档
- 如仅读取 12_规范 §3.1，会遗漏 01/02/05 等核心 P0 文档
- 即使读取多份入口文档，读者仍需自行交叉比对才能得到完整清单——认知负担大，出错概率高
- 跨 AI 交接时，不同 AI 可能因读取不同入口文档而形成不同的"项目认知基座"

## 解决方案

### 方案描述

**核心原则：CLAUDE.md 为唯一真实来源（SSOT），HANDOVER 和 12_规范 改为交叉引用。**

#### 修改 1：HANDOVER.md §6.1（必须遵守）—— 替换为交叉引用

将当前独立列出的 5 份文档列表，替换为指向 CLAUDE.md 的标准引用：

```
1. **所有参与本项目的 AI（包括但不限于 Claude Code、ChatGPT、其他大模型）必须先读取 CLAUDE.md 中列出的全部 P0 核心文档**。
   - 完整文档清单见 `CLAUDE.md` §开发前必读文档
   - 当前 P0 清单包含：HANDOVER.md、01_项目简介.md、02_产品定位与愿景.md、04_ADR_产品决策记录.md、05_知命_Persona角色设定.md
   - 清单如有更新，以 `CLAUDE.md` 为准
```

#### 修改 2：HANDOVER.md §7 Step 1（核心文档）—— 替换为交叉引用

将当前 Step 1 的独立文件列表，替换为：

```
### Step 1：阅读核心文档

先读取 `CLAUDE.md`（项目根目录）中 §开发前必读文档 列出的全部 **P0** 文档：
- 完整清单以 CLAUDE.md 为准，当前 5 份 P0 文档涵盖项目背景、产品方向、已确认决策和 AI 角色设定
- 读取后可在 `docs/11_AI协作日志.md` 中记录已读确认
```

#### 修改 3：HANDOVER.md §7 Step 2（当前状态）—— 替换为交叉引用

将当前 Step 2 的独立文件列表，替换为：

```
### Step 2：了解当前状态

再读取 `CLAUDE.md` §开发前必读文档 中列出的 **P1** 文档：
- 完整清单以 CLAUDE.md 为准，当前 8 份 P1 文档涵盖开发状态、UX 设计、分析流程、架构框架、协作规范等
- 特别关注 `docs/03_当前开发状态.md` 了解最新的待办事项
```

#### 修改 4：12_AI协作开发规范.md §3.1 Step 1（通用工作流）—— 对齐 CLAUDE.md

将当前 Step 1 的读取清单：

```
│  - HANDOVER.md           │
│  - 相关 ADR              │
│  - 当前开发状态           │
│  - 历史协作日志           │
```

替换为：

```
│  - 读取 CLAUDE.md §开发前必读文档 的 P0 清单（全部）  │
│  - 读取 CLAUDE.md §开发前必读文档 的 P1 清单（按需）  │
│  - 确认相关 ADR 最新状态                              │
│  - 读取历史协作日志了解近期活动                        │
```

#### 修改 5：12_AI协作开发规范.md Appendix B（文档快速索引）—— 增加 SSOT 注释

在 Appendix B 表格顶部添加注释行：

```
> 📌 **必读清单以 `CLAUDE.md` §开发前必读文档 为准，本表为用途导向的快速索引，非完整清单。**
```

#### 修改 6：CLAUDE.md §开发前必读文档 —— 确认 SSOT 地位

在表格下方添加自我声明注释：

```
> 📌 **本清单为必读文档的唯一权威版本。其他文档中的引用清单请以本表为准。**
> 如需修改（新增/移除文档），需同步更新本表 HANDOVER.md §6.1、§7 以及 12_规范 §3.1、§5.1、Appendix B 的引用。
```
