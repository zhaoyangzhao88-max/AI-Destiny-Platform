## 根因分析

### 问题定位

> 原文引用审计报告 §4.4.2：
> "| 1 | **ADR-014~019 状态歧义** | 🟡 P1 | 6 条 ADR 标注为 📝 Draft，但已在架构文档和 API 设计中作为'已确定方案'被引用。这种'半确认'状态对后续 AI 造成混淆——到底该将其视为已确认决策还是讨论草稿？"

### 根本原因

项目在 Phase 2 期间完成了 6 条架构相关 ADR 的编写（ADR-014~019），并在编写完成后立即应用到架构文档和 API 契约设计中。这些 ADR 在技术上经过充分推敲——每一条都包含完整的 Context/Decision/Consequences 结构——但由于项目当时处于"文档建设期"而非"决策确认期"，产品负责人尚未对它们进行正式的状态升级。

这产生了 **"实际已确认 vs 形式仍 Draft"** 的矛盾：

| 维度 | 形式状态 | 实际使用 |
|------|---------|---------|
| ADR 自身头部 | ✅ Draft | — |
| 架构文档引用 | — | 作为已确定方案引用 |
| API 契约引用 | — | 作为设计基础引用 |
| 后续 ADR 依赖 | — | ADR-018 依赖 ADR-017，ADR-019 依赖 ADR-017/018 |

上述矛盾的根本原因是：**项目缺乏从 Draft 到 Approved / Accepted Concept 的标准化升级路径**（该问题已在 ARC-16-P1-007 中解决，ADR 批准流程已建立）。

### 影响范围

- **对 AI 决策的影响：** 当 AI 读取 04_ADR 时看到 Draft 标记，可能认为方案未定而重新讨论已确定的架构方向
- **对引用文档的影响：** 08_架构、13_API、14_WS 中大量引用这些 ADR，如果 ADR 状态不明确，整条引用链的可靠性受损
- **对后续开发的影响：** Phase 3 编码依赖这些 ADR 作为实现依据，状态模糊可能延误开发
