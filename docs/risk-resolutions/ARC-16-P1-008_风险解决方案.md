## 根因分析

### 问题定位

> 原文引用审计报告 §4.5.2：
> "审计发现问题后，没有定义谁负责跟踪这些问题的解决状态、以什么频率更新。15_报告列出了 10 个前置条件，但无后续跟踪记录"

> 原文引用审计报告 §5.2 P1#8：
> "审计发现问题后无跟踪流程确保解决，15_报告的 10 个前置条件无后续更新"

### 根本原因

在审计机制的设计中，规范（`12_AI协作开发规范.md` 第四章）定义了完整的"发现问题→分级→报告"流程，但缺少了最关键的一步——**问题发现后的生命周期管理**。具体缺失环节：

1. **无跟踪载体** — 发现的问题分散在审计报告中，没有统一的跟踪文件来集中管理状态
2. **无负责人分配** — 谁负责解决哪个问题、何时解决无指定
3. **无状态更新机制** — 问题从发现到关闭没有状态流转定义
4. **无频率约定** — 跟踪表多久更新一次、谁来更新无要求

这使得每次审计变为"一次性快照"而非"持续改进闭环"——审计报告生成后就沉睡在 docs/ 中，问题是否已解决无人跟进。

### 影响范围

- 16 份审计报告中共产生 32 个风险条目（P0=6, P1=17, P2=9），均需跟踪
- 15_报告列出 10 个 Phase 3 前置条件，无跟踪则进度不可见
- 未来每次新增审计都会叠加更多问题，缺乏跟踪机制将导致问题越积越多

## 解决方案

### 方案描述

**本风险已被 `docs/18_架构风险跟踪表.md` 的创建直接解决。** 该文件是本问题的完整修复方案，自创建之日起即承担了审计问题的生命周期跟踪职责。

以下验证该文件对审计问题跟踪全部需求的覆盖情况：

| 跟踪需求 | docs/18 覆盖情况 | 对应章节 |
|---------|-----------------|---------|
| **风险唯一标识** | 每个条目有 ARC-ID | §4 / §5 所有条目 |
| **风险描述** | 每个条目有"问题描述"字段 | 所有条目 |
| **风险等级** | P0/P1/P2 三级 + 色标 | §3 + 所有条目 |
| **当前状态** | 5 状态生命周期（Open→In Progress→Resolved→Verified→Closed） | §2 + 每个条目 |
| **负责人角色** | 每个条目有"负责人角色"字段 | 所有条目 |
| **来源追溯** | 每个条目标注来源报告 + 章节编号 | 所有条目 |
| **关闭条件** | 每个条目有"关闭条件"字段 | 所有条目 |
| **建议措施** | 每个条目有"建议措施"字段 | 所有条目 |
| **修改记录** | 修复方案、修改文件、修改日期 | 已解决条目均有 |
| **状态流转规则** | 5 状态不可跳转、可驳回、可暂缓 | §2 |
| **关闭流程** | 标准 5 步流程 + 各等级验证标准 | §7 |
| **更新纪律** | 频率约定（新风险→立即，状态变更→每次） | §7.3 |
| **关闭记录模板** | 标准模板 | §7.4 |
| **Dashboard** | 总量/按来源/按负责人/按状态统计 | §6 |

所有 13 项审计问题跟踪需求均已覆盖。docs/18 是一份自洽的、可执行的风险生命周期管理文件。
