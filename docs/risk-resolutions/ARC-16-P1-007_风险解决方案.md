## 根因分析

### 直接原因

`12_AI协作开发规范.md` 第 5.3 节（ADR 管理规则）当前定义了以下流程：

| 状态 | 定义 |
|------|------|
| ✅ Approved | 已确认，不可直接修改，需通过 ADR 变更流程 |
| ✅ Accepted Concept | 概念已确认，细节待定。产品负责人 + AI（讨论层面）可修改 |
| ✅ Draft | 草稿，可自由修改。AI 和产品负责人均可 |
| 🟡 Draft（待审核）| 等待产品负责人确认。产品负责人 |
| 🔴 Deprecated | 已废弃。产品负责人 |

但缺少以下关键定义：
1. **什么条件** — Draft → Approved 或 Draft → Accepted Concept 的判定标准
2. **多少时间** — 产品负责人确认的时限约定（如 48h/5 个工作日）
3. **超时怎么办** — 产品负责人超过时限未响应时的替代处理机制
4. **谁可以催办** — 超时催办的责任人和催办方式

### 根本原因

项目在 Phase 1/2 阶段，产品负责人同时承担了"产品定义者 + 决策者 + 执行者"的多重角色，ADR 的创建和确认实际上由同一个人完成。在这种单人工作模式下，"产品负责人确认"是一个即时动作，不需要明确的时限或替代机制。

但随着项目进入 Phase 3（代码开发）和潜在的多人协作场景，ADR 的提出者和审批者将分离，上述假设不再成立。ADR 审批可能因产品负责人工作繁忙、不在线、优先度冲突等原因被延迟，从而阻塞下游开发决策。

### 影响范围

- Phase 3 启动后，技术实现依赖 ADR 决策，审批延迟直接影响开发效率
- ADR-014~019 当前均为 Draft 状态（ARC-16-P1-006），6 条未确认的方案已在架构文档中被引用，急需批准流程
- 未来多 AI Agent 协作场景下，Agent 提出的 ADR 需经产品负责人确认，流程缺失将成为瓶颈

---

## 解决方案

### 方案描述

在 `12_AI协作开发规范.md` 第 5.3 节中新增 ADR 批准流程的完整定义，包含条件、时限、超时替代机制和催办流程。

---

#### 新增：ADR 批准流程（插入到 5.3 节末尾）

```markdown
### 5.3.4 ADR 批准流程

#### 5.3.4.1 批准条件

产品负责人批准 ADR（Draft → Approved）时，应确认以下条件：

| # | 条件 | 说明 |
|---|------|------|
| 1 | **问题陈述清晰** | ADR 的 Context 部分完整描述待解决的问题背景和驱动力 |
| 2 | **方案明确** | Decision 部分给出具体、可执行的技术/产品决策，而非模糊方向 |
| 3 | **影响分析完整** | Consequences 部分列出优点、代价、后续影响，至少各 2 条 |
| 4 | **不与其他 Approved ADR 冲突** | 快速交叉对比已有 Approved ADR，无逻辑矛盾 |
| 5 | **版本兼容** | 如果是技术选型 ADR，确认不破坏已确定的架构方向（ADR-014/015/016 等） |

> 满足全部 5 项条件的产品负责人可直接批准。
> 满足条件 1~3 但不满足 4~5 的，应标记为 **Accepted Concept** 并记录待解决的问题。

#### 5.3.4.2 批准时限

| ADR 等级 | 时限 | 计时起点 |
|----------|------|---------|
| **🔴 P0（阻止级）** | **12 小时内** | 产品负责人收到通知之时起 |
| **🟡 P1（优先级）** | **48 小时内** | 产品负责人收到通知之时起 |
| **🟢 P2（优化级）** | **5 个工作日内** | 产品负责人收到通知之时起 |

> **通知方式：** ADR 提交人（AI Agent 或人类）在协作日志 `docs/11_AI协作日志.md` 中追加记录，
> 标注 `【ADR审批请求】ADR-XXX（等级）`，同时通过项目约定的即时通信渠道通知产品负责人。
>
> **时限定义：** "12 小时内"指自然时间的 12 小时，含非工作日；
> "5 个工作日"指排除周末和法定节假日的 5 天。

#### 5.3.4.3 超时替代机制

若产品负责人在上述时限内未做出响应，按以下规则自动处理：

| 场景 | 超时处理 | 后续操作 |
|------|---------|---------|
| **P0 ADR** 超时 | 自动升为 ✅ **Accepted Concept**，允许开始设计/原型验证 | 产品负责人返回后需在 24h 内复审并确认（Approved）或驳回（Back to Draft） |
| **P1 ADR** 超时 | 自动升为 ✅ **Accepted Concept**，允许基于此方案开始设计 | 产品负责人返回后需在 5 个工作日内复审 |
| **P2 ADR** 超时 | 保留 🟡 **Draft（待审核）** 状态，不自动升级 | AI 可在协作日志中记录"超时：P2 ADR 待审批"，待产品负责人下次工作时处理 |

> **重要：** 超时自动升级为 Accepted Concept **不等于** Approved。Accepted Concept 的定位是"概念已确认，细节待定"——允许开始设计工作，但不应视为最终决策。
> 产品负责人返回后仍需对超时升级的 ADR 进行确认或驳回。

#### 5.3.4.4 催办机制

| 角色 | 催办条件 | 催办方式 |
|------|---------|---------|
| **提交 ADR 的 AI** | 超过时限 50% 未响应 | 在协作日志中追加 `【催办】ADR-XXX 审批，已超过时限 XX%，请产品负责人确认` |
| **Audit Agent** | 任意 ADR 超过时限未处理 | 在下次审计中标注"ADR 审批延迟"，作为治理风险记录 |
| **其他 AI Agent** | 依赖该 ADR 决策的工作受阻 | 可在协作日志中追加 `【受阻】ADR-XXX 未审批，当前工作 YYY 阻塞` 说明影响 |
```

#### 同步修改：Accepted Concept 使用指南（补充 4.2 节）

当前 `12_AI协作开发规范.md` 4.2 节对 Accepted Concept 的定义为"概念已确认，细节待定"，但未说明可否据此开始设计。新增以下说明：

> **Accepted Concept 状态使用指南：**
>
> ✅ **可开始的设计/开发活动：**
> - 技术方案的设计文档编写
> - 原型开发（Prototype）和可行性验证
> - 依赖于该决策的辅助模块设计（在决策被确认前保持模块独立）
> - 接口契约的前期设计（标注"依赖 ADR-XXX 确认后定稿"）
>
> ❌ **不可开始的开发活动：**
> - 基于该 ADR 的生产级代码开发（不可进入测试/发布流程）
> - 对外部的技术承诺（如"V1 将支持 X 特性"的公开声明）
> - 对其他已确认 ADR 的修改
>
> **从 Accepted Concept 到 Approved 的条件：**
> 原型验证通过 + 产品负责人书面确认 = 升级为 Approved
