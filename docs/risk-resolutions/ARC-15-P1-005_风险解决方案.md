## 风险概述

| 字段 | 内容 |
|------|------|
| **风险 ID** | ARC-15-P1-005 |
| **问题描述** | `13_API接口契约设计.md` §3.1 通用约定和 §4.5 鉴权方式描述了传统的 Bearer Token / JWT 认证模型（用户名+密码登录，Token 有效期 24h，无 Refresh Token 机制），但该模型是为 Web/移动端设计的——知命 V1 是本地优先的桌面 APP，用户注册/登录流程在桌面场景下显得过重，且登录流程与本地优先架构的「离线可用」核心原则存在矛盾 |
| **等级** | 🟡 P1 |
| **影响** | 执行传统的用户名+密码注册/登录流程会破坏本地优先的使用体验——用户下载 APP 后第一件事不是使用产品，而是注册账号、等待 Token 认证，这与"本地排盘、离线可用"的产品承诺不符 |
| **建议措施** | V1 采用设备绑定简化认证，JWT 仅用于云端通信 |
| **当前状态** | ✅ Approved — 已确认方案 A（设备绑定简化认证）|
| **决策人** | 产品负责人 |

## 根因分析

### 核心矛盾

`13_API.md` 当前的认证模型直接复用了一般的 Web APP API 设计模式：

| 认证要素 | 当前设计 | 桌面 APP 场景问题 |
|---------|---------|-----------------|
| 注册流程 | 用户名 + 密码 + 出生信息 | 用户首次打开 APP 被要求注册，增加转化摩擦 |
| 登录流程 | POST /login 换取 JWT | 每次启动 APP 需要认证，离线时无法登录 |
| Token 管理 | JWT + 24h 有效期 | Token 过期后需重新登录，打断使用体验 |
| Refresh Token | V1 不实现 | Token 过期后用户被迫重新输入密码 |
| 无鉴权端点 | 仅 health/register/login | 离线状态下大部分核心功能（排盘/档案浏览）被鉴权阻挡 |

### 产品的实际认证需求

知命 V1 作为本地优先的桌面 APP，实际的认证需求是：

1. **本地操作应无需认证** — 排盘、档案浏览、本地分析是纯本地操作，不应需要任何网络 Token
2. **云端服务需轻量认证** — AI 深度分析（调用 LLM API）和真太阳时校准才需要身份识别
3. **首次使用零摩擦** — 用户下载后应能立即开始使用，而不是先注册
4. **离线可用** — 即使完全离线，核心功能不受影响

### 当前文档约束

`13_API.md` §4.5 明确写道：
- 所有端点除标注「无鉴权」外均需 `Authorization: Bearer <token>`
- 无鉴权端点仅 3 个：health / register / login
- Token 有效期 24h，无刷新机制

这意味着离线状态下，用户无法获取有效 Token，从而无法操作任何 API 端点——包括排盘和档案浏览这样纯本地的功能。

## 解决方案

### 已确认方案

**方案 A — 设备绑定简化认证**（由产品负责人确认采纳）

**方案描述：** APP 首次启动时自动生成唯一的设备密钥（Device Key），存储在本地安全存储中。该设备密钥即本地操作的身份凭证，无需用户注册即可使用全部本地功能。JWT 仅在调用云端服务（LLM API、真太阳时校准）时才需要，用户在首次使用云端功能时才触发注册/登录流程。

**认证流程：**

```
APP 首次启动:
  1. 生成设备密钥 (device_key) → 存入本地安全存储
  2. 设备密钥自动签名本地请求 → 核心服务识别设备身份
  3. 用户可立即使用全部本地功能（排盘/档案/本地分析）

用户首次使用云端功能（如 AI 深度分析）:
  1. APP 提示"此功能需要联网账号，是否注册/登录？"
  2. 用户注册 → 获得 JWT → JWT + device_key 绑定
  3. 后续云端调用使用 JWT，本地调用仍使用 device_key
  
Token 过期:
  1. 本地功能不受影响（device_key 永不过期）
  2. 云端功能需刷新 Token：设备绑定用户可无感刷新（通过 device_key 换取新 JWT）
```

**决策理由：**

| 维度 | 评估 |
|------|------|
| **零摩擦体验** | 打开即用，注册不再是强制步骤，完美满足"下载即用"的产品承诺 |
| **离线可用** | device_key 本地存储，不依赖网络，核心功能（排盘/档案/本地分析）离线完全可用 |
| **实现成本** | 仅需 Windows DPAPI/Credential Store 存储 device_key，无需额外服务端基础设施 |
| **云端认证可延迟触发** | 用户用到云端功能时才注册，转化漏斗更自然，减少注册流失 |
| **安全性** | device_key 双通道 + JWT 云端分离，本地无明文密钥暴露风险 |
| **V2 可扩展性** | 当前方案为设备绑定；V2 多设备/云同步场景需扩展（见下方 V2 扩展条件） |

**涉及实现要点：**

- 客户端需实现设备密钥的生成、安全存储（Windows DPAPI / Credential Store）和签名逻辑
- 服务端需实现 device_key 到 JWT 的绑定和刷新接口
- `13_API接口契约设计.md` §3.1 / §4.5 需按本方案更新，区分本地认证和云端认证

---

### V2 扩展条件

> **适用范围声明：** 本文确认的"设备绑定简化认证"方案**仅适用于纯桌面单机场景**。

若知命未来扩展至 **Web 端或移动端**（如 ADR-015 规划的 V2/V3），需重新评估认证体系，原因如下：

| 场景 | 设备绑定方案的限制 |
|------|-------------------|
| **Web 端** | 浏览器无安全硬件存储（DPAPI/Credential Store），无法安全存储设备密钥 |
| **移动端** | iOS/Android 有安全存储但跨设备同步场景与设备密钥绑定矛盾 |
| **多设备用户** | 同一用户在不同设备上会有不同的 device_key，身份合并需要额外机制 |
| **V2 云同步** | 云端数据同步需要统一的用户身份而非设备身份 |

**建议的 V2 评估时机：** 当产品路线图确认进入 Web 或移动端开发时，重新进行认证体系选型，评估 OAuth 2.0 / OpenID Connect 等 Web 标准认证方案。

### 曾评估的替代方案

以下方案在决策过程中被评估但未采纳，保留记录供未来参考。

#### 方案 B：传统 JWT 认证（当前设计，始终在线）

保持 `13_API.md` §4.5 的原始设计——每次使用均需 JWT 认证，用户首次使用时必须注册/登录。

| 优点 | 缺点 |
|------|------|
| 实现最简单，标准 JWT 流程 | 首次使用摩擦大，必须先注册再使用 |
| 与 Web APP 认证模型一致，V2 可复用 | **离线不可用**：无网络时无法获取 Token，全部功能被锁定 |
| 用户身份明确，每一次操作都可追溯 | Token 无刷新机制，24h 后必须重新登录 |
| 跨设备认证天然支持 | **与 ADR-014「本地优先」核心原则严重冲突** |

**不采纳原因：** 与本地优先架构核心原则存在根本性矛盾，不适合 V1 产品定位。

#### 方案 C：Hybrid 本地优先认证（设备密钥 + JWT 无感刷新）

在方案 A 的基础上增强认证链路，实现设备密钥驱动的 JWT 无感刷新机制。

| 优点 | 缺点 |
|------|------|
| 最佳用户体验：首次零摩擦 + Token 无感刷新 | 实现复杂度最高，需 device_key 远程验证接口 |
| 安全等级高，分层安全设计 | device_key 丢失需要额外账户恢复流程 |
| 无 Token 过期打断 | V1 用户量小时可能过度设计 |
| 为 V2 跨设备认证奠定基础架构 | 开发工作量显著高于方案 A |

**不采纳原因（V1 阶段）：** V1 用户量小、场景简单，方案 A 已满足全部核心需求。方案 C 的实现成本在 V1 阶段不必要，建议 V2/V3 阶段评估升级至此方案。
