## 根因分析

### 问题定位

> 原文引用审计报告 §4.5.2：
> "| 2 | **无复审流程** | 🟡 P1 | 审计发现问题→修复→需要'复审'确认修复有效——但规范中没有定义复审的执行方式（快速确认审计 vs 完整二次审计）"

> 审计报告 §7.1-B 进一步建议：
> "同时定义复审流程：P0 问题解决后需执行一次快速确认审计（可由同一 AI 在解决后立即执行，确认后更新跟踪表状态）。"

### 根本原因

`12_AI协作开发规范.md` 第四章（阶段性审计机制）定义了完整的审计流程——从触发条件（§4.1）、审计目标（§4.2）、审计原则（§4.3）、风险分级（§4.4）、报告格式（§4.5）到阶段门规则（§4.6），覆盖了"如何发现问题"。但缺少了审计闭环的关键一环——**"发现问题 → 修复 → 确认修复有效"的复审流程**。

当前状态：发现问题 → 修复 → ❌ 无标准化确认步骤。

这导致：
1. **P0 问题：** 修复后无人验证，可能引入新问题或修复不完全
2. **P1/P2 问题：** 修复状态依赖产品负责人记忆，无系统化跟踪
3. **审计报告前置条件：** 15_报告列出的 10 个前置条件无后续跟踪记录，部分已修复但无正式确认

### 影响范围

- 所有审计发现的问题在"已修复"后缺乏质量门
- 18_架构风险跟踪表 中的风险状态流转（Resolved → Verified → Closed）缺少 Verified 环节的执行定义
- Phase 进入决策（阶段门）的质量保证依赖于"修复确认"的可靠性

---

## 解决方案

### 方案描述

在 `12_AI协作开发规范.md` §4.6（阶段门规则）之后新增 §4.7「复审流程」，按风险等级定义不同的复审方式。同时同步更新 `18_架构风险跟踪表.md` §7.2 的验证标准以保持一致。

---

#### 新增 §4.7：复审流程

在 §4.6 阶段门规则之后新增以下章节：

```markdown
### 4.7 复审流程

#### 4.7.1 目的

复审是在风险/问题修复后，通过审计确认修复正确性、完整性的标准化流程。
它是"发现问题 → 修复 → 确认有效"闭环的最后一步，确保每个修复都经过验证方可关闭。

#### 4.7.2 复审等级

按风险等级定义不同的复审方式：

| 等级 | 复审方式 | 执行者 | 执行时机 | 深度要求 |
|------|---------|--------|---------|---------|
| **🔴 P0** | 独立快速确认审计 | Audit Agent（不可与修复者为同一 Agent） | 修复完成后 24h 内 | 完整复审修复方案 + 修复结果；逐条验证关闭条件 |
| **🟡 P1** | 纳入下次阶段审计 / 产品负责人直接确认 | Audit Agent 或 产品负责人 | 下次阶段审计时，或产品负责人指定时间 | 确认修复满足关闭条件即可；无需完整二次审计 |
| **🟢 P2** | 由下次定期审计验证 | Audit Agent | 下次例行审计或阶段审计时 | 简略确认状态更新；可在审计报告中合并验证 |

#### 4.7.3 P0 独立快速确认审计

P0 问题修复后必须执行独立快速确认审计，流程如下：

```
P0 修复完成（Resolved）
    │
    ▼
通知 Audit Agent（不可为原修复者）
    │
    ▼
执行快速确认审计：
  ① 读取修复方案 → 确认修复逻辑正确
  ② 读取修改文件 → 确认修改完整（无遗漏/无过度修改）
  ③ 逐条验证关闭条件 → 确认每条条件均已满足
  ④ 回归检查 → 确认修复未引入新问题
    │
    ▼
输出复审记录 → 更新跟踪表状态（Resolved → Verified）
    │
    ▼
产品负责人最终确认 → 关闭（Verified → Closed）
```

**核心要求：**
- **独立性：** 执行复审的 Audit Agent 不得与执行修复的 Agent 为同一 AI
- **完整性：** 覆盖全部关闭条件，不遗漏
- **记录：** 复审结果需在 `11_AI协作日志.md` 中记录

#### 4.7.4 P1/P2 复审

- **P1：** 已修复的 P1 问题由产品负责人直接确认，或在下次阶段审计中批量验证
- **P2：** 已修复的 P2 问题纳入下次定期审计或阶段审计的范围，由 Audit Agent 统一验证
- 产品负责人可在任何时候要求对特定 P1/P2 问题执行独立复审

#### 4.7.5 复审记录格式

每次复审后，需在 `11_AI协作日志.md` 中追加以下记录：

```
--- 复审记录 ---
风险 ID:       ARC-XX-YYY-NNN
风险标题:      [原问题描述摘要]
原修复方案:    [简述]
复审方式:      [独立快速确认审计 / 阶段审计 / 产品负责人确认]
复审结论:      [通过 / 有条件通过 / 未通过]
复审日期:      YYYY-MM-DD
执行复审:      [Audit Agent / 产品负责人]
未通过原因:    [如未通过，说明]
```

#### 4.7.6 与风险跟踪表的关系

复审流程与 `18_架构风险跟踪表.md` 的风险状态流转直接对应：

```
风险跟踪表状态      复审流程对应
────────────────────────────────────
Resolved  ──→  修复完成，等待复审
    │
    ▼
复审执行   ──→  4.7.2 按等级执行对应复审
    │
    ├── 通过 → Verified（跟踪表同步更新）
    │
    └── 未通过 → 退回 In Progress（跟踪表同步更新 + 记录原因）
    │
    ▼
产品负责人确认 → Closed（跟踪表同步更新 + 11_日志记录）
```

#### 4.7.7 复审触发条件

复审在以下条件下自动触发：

- **P0 问题：** 修复后自动触发，Audit Agent 应在 24h 内响应
- **P1 问题：** 在下次阶段审计时自动纳入复审范围
- **P2 问题：** 在下次定期审计时统一验证
- **产品负责人指定：** 产品负责人可随时指定某问题需要立即复审

> **超时处理：** 若 P0 问题修复后在 48h 内未触发复审，Audit Agent 应在协作日志中备注 `【复审超时】ARC-XXX 修复后 48h 未复审`，并主动安排复审。
```
