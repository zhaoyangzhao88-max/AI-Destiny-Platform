## 根因分析

### 问题定位

> 原文引用审计报告 §4.6.2：
> "| 4 | **规范语言偏向 Claude Code** | 🟢 P2 | 多处出现'Claude Code 沙箱'、'Claude Code / OpenCode 等'、'Claude Code 项目入口规则'等 Claude 特定表述 |
> | 5 | **无针对非 Claude AI 的适配说明** | 🟢 P2 | 如 ChatGPT 没有'Plan 模式'，不适用 CLAUDE.md 的 Plan 模式要求——规范未说明其他 AI 应如何映射"

### Claude 特定表述扫描结果

对核心治理文档 `docs/12_AI协作开发规范.md` 进行全文扫描，发现以下 Claude 特定表述：

| # | 位置 | 原文 | 类型 | 建议替换 |
|---|------|------|------|---------|
| 1 | §3.3.2 L218 | `开发代理（Claude Code / OpenCode 等）` | 工具名枚举 | `开发代理（如 Claude Code、OpenCode 等）` — 替换为通用表述 |
| 2 | §6.5 Git 提交规则 L662 | `Co-Authored-By: Claude <noreply@anthropic.com>` | 署名格式 | 替换为通用 AI Agent 署名格式（见方案） |

此外，其他文档中存在的 Claude 特定表述（不在本次直接修改范围内，但建议同步处理）：

| 文件 | 原文 | 建议替换 |
|------|------|---------|
| `CLAUDE.md`（项目根目录） | `Plan 模式要求` | 保留（CLAUDE.md 是 Claude Code 入口文件，Plan 模式是其原生功能，无需替换） |
| `HANDOVER.md` | `适用于包括但不限于 Claude Code、ChatGPT、其他大模型` | 保留（这是包容性声明，列举具体工具名是合理的） |
| `docs/00_AI_CONTEXT_INDEX.md` | `-- CLAUDE.md — Claude Code 项目入口规则与 Plan 模式要求` | `-- CLAUDE.md — 项目入口规则（CLAUDE.md）与 Plan 模式要求` |

### 根本原因

项目以 Claude Code 为主要 AI 协作工具进行开发，文档中存在工具特定术语是自然的结果。此问题属于"表述层面的通用性不足"，不涉及治理体系的结构性缺陷。随着未来引入其他 AI 工具（ChatGPT、Cursor、Windsurf 等），这些特定术语会降低规范的适用性。

---

## 解决方案

### 方案描述

对 `docs/12_AI协作开发规范.md` 中的 Claude 特定表述进行精准替换，同时对涉及"AI 执行环境"的概念做通用化表述升级。

#### 替换映射表

| 原文 | 替换为 | 位置 | 说明 |
|------|--------|------|------|
| `"Claude Code / OpenCode 等"` | `"Claude Code、OpenCode 等 AI 执行环境"` | §3.3.2 L218 | 添加"AI 执行环境"作为修饰语，使表述更通用 |
| `"Co-Authored-By: Claude <noreply@anthropic.com>"` | `"Co-Authored-By: <AI Agent 标识> <<agent-email>>"` | §6.5 L662 | 署名格式通用化，实际使用时替换为当前 Agent 的名称和邮件 |
| `"禁止在 AI 沙箱内"` | （已为通用表述，无需修改） | §1.3 L64 | 当前写的是"AI 沙箱"而非"Claude Code 沙箱"，无需修改 |

#### Plan 模式映射说明

对于非 Claude Code 的 AI 工具，"Plan 模式"在多处被引用。建议在 `12_AI协作开发规范.md` §3.1 通用工作流步骤②后补充以下说明：

```markdown
> **Plan 模式适配说明：** "Plan 模式"在 Claude Code 中原生支持。使用其他 AI 工具时，
> 应将其映射为等效的"方案规划"步骤——即正式执行前，先输出一份结构化 Plan
> （目标、影响分析、受影响的 ADR、修改文件列表），等待产品负责人确认后再执行。
```

### 操作步骤

#### Step 1：更新 `12_AI协作开发规范.md` §3.3.2

修改 L218：
```
原文：当用户无法直接上传项目文件时，AI 应通过开发代理（Claude Code / OpenCode 等）执行文件读取任务

修改：当用户无法直接上传项目文件时，AI 应通过开发代理（Claude Code、OpenCode 等 AI 执行环境）执行文件读取任务
```

#### Step 2：更新 `12_AI协作开发规范.md` §6.5 署名模板

修改 Git 提交模板中的署名格式，使其不绑定具体 AI 工具：

```
原文：
Co-Authored-By: Claude <noreply@anthropic.com>

修改：
Co-Authored-By: <Agent 名称/ID> <agent-email>
```

并在示例中注明：*替换 `<Agent 名称/ID>` 为当前 AI Agent 的身份标识（如 `Agent-20260716-01`），替换 `<agent-email>` 为项目约定的 Agent 邮箱（如 `agent@zhiming-ai.dev`）。*

#### Step 3：补充 Plan 模式适配说明

在 §3.1 通用工作流步骤②（任务分析 & Plan）下方补充适配说明段落。
