## 根因分析

### 问题链

```
ADR-012 定义了数据三层体系（fact/inference/pending）
    ↓
§2.3 要求 AI 标注"信息来源"和"确定性分级"
    ↓        ← 缺失环节
§8.1 报告模板无对应字段
    ↓
实际协作日志中从未出现这些信息（"说了但没做"）
```

### 直接原因

`12_AI协作开发规范.md` 的 §2.3 节（AI 能力边界声明）和 §8.1 节（执行报告规范）由不同的编辑者/编辑轮次编写：

- **§2.3** — 强调了 AI 的信息标注义务，包含「确定性分级」和「信息来源标注」两条要求
- **§8.1** — 定义了协作日志的模板结构，包含任务信息、修改清单、关键决策、问题等已有字段

两份独立编写的章节缺少交叉一致性校验，导致 §2.3 的要求未在 §8.1 中找到对应落地字段。

### 根本原因

这是典型的**规范内部脱节**问题：一个章节定义了"该做什么"，另一个章节定义了"该怎么记"，但两者没有对齐。本质上是规范文档缺乏跨章节的自洽性校验机制。治理审计（ARC-16-P1-008）已指出此问题，本风险是治理审计的具体表现实例之一。

### 影响范围

- 截至 Sprint 19 之前的所有协作日志（`11_AI协作日志.md` 中的数十条记录）均缺少信息来源和确定性分级字段
- 后续 AI 读取历史日志时无法快速区分哪些信息是确定性事实、哪些是推测性分析
- ADR-012 的数据三层体系在"AI 协作治理"维度的要求在规范层脱节——ADR-012 定义了产品的数据分层，但治理规范本身的记录也应当遵循同样的分层理念

---

## 解决方案

### 方案描述

在 `12_AI协作开发规范.md` §8.1 报告模板中新增两个字段；同时在 §2.3 中补充说明这些字段与 ADR-012 的关联。

---

### 1. §8.1 报告模板修改

在报告模板的「任务信息」表格中，在现有字段后新增两行：

**修改前：**

```markdown
### 任务信息

- **任务来源：** [产品负责人 / 用户指令]
- **任务描述：** [一句话描述本次任务]
- **AI 角色：** [文档撰写者 / 方案分析师 / 代码开发者 / 评审者 / 测试者]
- **使用的认知框架：** [CoT / 系统思维 / 第一性原理等]
```

**修改后：**

```markdown
### 任务信息

- **任务来源：** [产品负责人 / 用户指令]
- **任务描述：** [一句话描述本次任务]
- **AI 角色：** [文档撰写者 / 方案分析师 / 代码开发者 / 评审者 / 测试者]
- **使用的认知框架：** [CoT / 系统思维 / 第一性原理等]
- **信息来源：** [项目内文档 / 外部知识 / 模型训练数据]
- **确定性等级：** [事实性陈述 / 分析性推理 / 推测性建议]
```

#### 字段填写规则

| 字段 | 可选值 | 说明 |
|------|--------|------|
| **信息来源** | `项目内文档` | 基于项目文档体系（docs/*.md）、代码库中的已有信息 |
| | `外部知识` | 基于外部知识库、网络搜索、第三方资料、命理经典典籍 |
| | `模型训练数据` | 基于 LLM 预训练知识（模型自身的训练数据中涵盖的通用知识）|
| | `混合` | 综合多种来源（如"项目内文档 + 外部知识"）|
| **确定性等级** | `事实性陈述` | 直接来自项目文档或 ADR 的确定性信息、可验证的排盘计算结果 |
| | `分析性推理` | 基于已有信息做出的推理分析（建议方案、设计决策、影响分析）|
| | `推测性建议` | 基于经验和模式的推测性建议（"建议考虑 X 方案"、"未来可能需要 Y"）|

#### 与 ADR-012 的关系

两个字段直接从 ADR-012（数据三层体系）派生而来，是 ADR-012 在治理规范执行层面的映射：

| ADR-012 体系 | → | 本规范字段 |
|-------------|:---:|-----------|
| `fact`（事实性）| → | `确定性等级: 事实性陈述` |
| `inference`（分析性）| → | `确定性等级: 分析性推理` |
| `pending`（推测性）| → | `确定性等级: 推测性建议` |
| 信息来源标注 | → | `信息来源: 项目内文档/外部知识/模型训练数据` |

> **说明：** 规范 §2.3 中定义的确定性分级是 ADR-012 在 AI 协作治理维度的具体应用，两者共同遵循"区分事实与推断"的核心原则。规范记录中的确定性等级用于帮助后续 AI 快速评估协作日志中各条信息的可信度层次。

#### 使用示例

```markdown
### 任务信息

- **任务来源：** 产品负责人
- **任务描述：** 编写 API 接口契约设计文档 v0.1
- **AI 角色：** 文档撰写者
- **使用的认知框架：** 第一性原理 + 系统思维
- **信息来源：** 项目内文档 + 外部知识（RESTful API 设计规范）
- **确定性等级：** 分析性推理（设计建议，标注了【待确认】项）
```

---

### 2. §2.3 补充 ADR-012 引用

在 §2.3（AI 能力边界声明）中，将当前列表：

```markdown
1. **信息来源标注** — 基于项目内文档 / 基于外部知识 / 基于模型训练数据
2. **确定性分级** — 事实性陈述 / 分析性推理 / 推测性建议
3. **不确定时处理** — 不确定时需向产品负责人提问，而非自行推断
```

扩展为（增加来源说明）：

```markdown
1. **信息来源标注** — 基于项目内文档 / 基于外部知识 / 基于模型训练数据
   - 此字段与 ADR-012（数据三层体系）共用确定性分级原则，用于在协作记录中标注信息的来源可信度
   - 具体字段定义见 §8.1 报告模板的「任务信息」说明
2. **确定性分级** — 事实性陈述 / 分析性推理 / 推测性建议
   - 映射自 ADR-012 的三层体系：`fact → 事实性陈述，inference → 分析性推理，pending → 推测性建议`
   - 分级目的是帮助后续 AI 快速评估历史协作信息中的可信度层次，避免将推测性建议当作确定性事实使用
3. **不确定时处理** — 不确定时需向产品负责人提问，而非自行推断
```
