## 根因分析

### 审计发现总结

`15_Phase2架构审计报告.md` §6.3（数据模型准备度评分）指出 API 契约现有数据模型在以下四个模块中存在字段缺口：

| 模块 | 审计评分 | 主要缺口 |
|------|---------|---------|
| 用户档案（User） | ⚠️ 2/5 | 无 nickname、birth_time_unknown 标记、timezone 未统一 |
| 八字排盘（Bazi） | ⚠️ 2/5 | 用神/忌神为字符串而非结构化对象；无格局枚举；无纳音/神煞 |
| 分析报告（Report） | ⚠️ 1/5 | section.metadata 结构不完整；置信度分数体系未定义 |
| 知识库（Knowledge） | ⚠️ 1/5 | 术语内部内容结构未定义；引用/来源结构缺失；知识图谱关系未定义 |

### 根本原因

项目处于产品设计和文档建设期（Phase 2），API 契约的优先目标为**确定接口通信模式、路由设计、数据流向**等架构层内容，数据模型层面的细粒度设计被推迟到 Schema 阶段。这是合理的阶段划分决策，但随着进入 Phase 3（开发实现），字段级别的一致性必须补齐。

### 影响范围

- 开发者在实现数据层（SQLite Schema、Repository 模式）时将面对字段缺失、类型不明确等问题
- 前端开发者因字段结构不完整可能做出错误的数据绑定
- 字段定义不清晰可能导致不同模块间数据交换不一致
- 部分字段（如用神结构化、置信度体系）需要产品负责人确认具体设计，存在依赖关系

---

## 解决方案

### 方案描述

为每个模块提供详细的缺失字段规范。对于需要产品负责人设计决策的复杂结构（用神结构化定义、置信度体系），标注 **【待产品负责人确认具体结构】**，不做硬编码假设。

### 1. 用户档案（User）模块 — 缺失字段

#### 1.1 当前状态

当前 `GET /api/v1/users/me/profile` 响应（§3.2）中缺少以下字段：

- `basic_info.nickname` — 用户昵称（显示用）
- `basic_info.birth_time_unknown` — 出生时间未知标记
- `basic_info.timezone` — 时区字段（未在 user profile 层统一）

#### 1.2 补充字段定义

新增到 `GET /api/v1/users/me/profile` 响应结构的 `basic_info` 对象：

| 字段 | 类型 | 必填 | 默认值 | 描述 | 示例 |
|------|------|------|--------|------|------|
| `basic_info.nickname` | string | ❌ | `null` | 用户显示昵称，1-32 个字符，不提供则默认用 username | `"天命追寻者"` |
| `basic_info.birth_time_unknown` | bool | ✅ | `false` | 出生时间未知标记。`true` 表示用户未提供或不知道自己出生的具体时间（时/分），排盘时作特殊处理。`birth_hour` 和 `birth_minute` 为 `null` 时此字段应为 `true` | `true` |
| `basic_info.timezone` | string | ❌ | `"Asia/Shanghai"` | IANA 时区标识符（`tz database`）。用于排盘时的真太阳时计算和出生信息的地理位置校准。注册接口已有 `birth_place` 但未独立存储 timezone。注册时若未提供，可通过 `birth_place` 反查默认时区 | `"Asia/Shanghai"` |

#### 1.3 涉及修改的位置

- `docs/13_API接口契约设计.md` §3.2 — `POST /api/v1/users/register` Request 增加 `nickname`（可选）、`timezone`（可选）
- `docs/13_API接口契约设计.md` §3.2 — `GET /api/v1/users/me/profile` Response `basic_info` 增加上述三个字段

### 2. 八字排盘（Bazi）模块 — 缺失字段

#### 2.1 当前状态

当前 `POST /api/v1/bazi/charts` 响应（§3.3）中：

- **用神**：`yong_shen: "金"` — 为简单字符串，无推导方法、来源、置信度信息
- **喜神/忌神**：同样为简单字符串
- **格局**：未出现在响应中，即使 AI 分析阶段会推导格局
- **纳音**：四柱各有纳音（如年柱"庚午"纳音"路旁土"），但当前仅输出天干地支和藏干，无纳音
- **神煞**：未出现在响应中

#### 2.2 补充字段定义

##### 2.2.1 用神/喜神/忌神结构化

**当前：**
```json
"yong_shen": "金",
"xi_shen": "水",
"ji_shen": "火"
```

**改为结构化对象【待产品负责人确认具体结构】：**

```json
"yong_shen": {
  "element": "金",
  "method": "扶抑法",
  "explanation": "日主戊土偏旺，取金泄土生水为用",
  "confidence": 0.85,
  "data_tier": "inference",
  "alternative": [
    {"element": "水", "method": "调候法", "confidence": 0.65, "data_tier": "inference"}
  ]
},
"xi_shen": {
  "element": "水",
  "relation_to_yong": "金生水，喜神生用神",
  "confidence": 0.75,
  "data_tier": "inference"
},
"ji_shen": {
  "element": "火",
  "reason": "火生土，加剧日主偏旺之势",
  "confidence": 0.80,
  "data_tier": "inference"
}
```

| 字段 | 类型 | 必填 | 描述 |
|------|------|------|------|
| `yong_shen.element` | string | ✅ | 五行属性：`金/木/水/火/土` |
| `yong_shen.method` | string | ✅ | 推导方法：`扶抑法/调候法/通关法/病药法` |
| `yong_shen.explanation` | string | ❌ | 人类可读的推导理由 |
| `yong_shen.confidence` | float | ❌ | 置信度 0.0-1.0 |
| `yong_shen.data_tier` | string | ✅ | `fact/inference/pending` |
| `yong_shen.alternative` | array | ❌ | 备选用神方案（学派分歧或多方法推导不一致时） |

> **说明：** 用神推导是八字分析的核心且存在学派分歧，其结构化定义的具体字段、推导方法枚举值、置信度计算方式需产品负责人确认。当前方案给出最小可行结构，允许未来扩展。

##### 2.2.2 格局类型枚举

在 `GET /api/v1/bazi/charts/{chart_id}` 响应中新增字段：

| 字段 | 类型 | 必填 | 描述 | 示例 |
|------|------|------|------|------|
| `pattern` | string | ❌ | 命局格局名称 | `"杀印相生格"` |
| `pattern_category` | string | ❌ | 格局大类枚举 | `"normal"`（正格）/ `"special"`（特殊格）/ `"none"`（无格局）|

**格局类型枚举值（建议，待产品负责人确认）：**

- 正格：`正官格`、`七杀格`、`正印格`、`偏印格`、`正财格`、`偏财格`、`食神格`、`伤官格`、`建禄格`、`月刃格`
- 特殊格：`从杀格`、`从财格`、`从儿格`、`从旺格`、`化气格`、`专旺格`
- 复合描述：如 `杀印相生格`、`食神制杀格`、`伤官佩印格`

##### 2.2.3 纳音字段

在 `pillars` 的每个柱对象中新增：

| 字段 | 类型 | 必填 | 描述 | 示例 |
|------|------|------|------|------|
| `na_yin` | string | ❌ | 该柱干支的纳音五行 | `"路旁土"` |

**示例修改：**
```json
"pillars": {
  "year": {
    "heavenly_stem": "庚",
    "earthly_branch": "午",
    "hidden_stems": ["丁", "己"],
    "na_yin": "路旁土"
  }
}
```

##### 2.2.4 神煞字段

在排盘响应中新增：

| 字段 | 类型 | 必填 | 描述 | 示例 |
|------|------|------|------|------|
| `shen_sha` | object | ❌ | 神煞列表，key为神煞名称，value为该神煞出现的柱 | 见下 |

```json
"shen_sha": {
  "天乙贵人": ["hour"],
  "文昌贵人": ["day"],
  "桃花": ["year", "hour"],
  "驿马": ["month"]
}
```

> **说明：** 神煞体系在不同命理流派中存在差异（如《渊海子平》与《三命通会》的神煞定义不完全一致）。当前建议仅收录被广泛认可的约 15-20 种神煞，具体列表待产品负责人确认。

#### 2.3 涉及修改的位置

- `docs/13_API接口契约设计.md` §3.3 — `POST /api/v1/bazi/charts` Response 修改用神/喜神/忌神为结构化、新增 pattern / pattern_category / shen_sha / na_yin 字段
- `docs/13_API接口契约设计.md` §3.3 — `GET /api/v1/bazi/charts/{chart_id}` Response 同步更新

### 3. 分析报告（Report）模块 — 缺失字段

#### 3.1 当前状态

当前 `GET /api/v1/reports/{report_id}` 响应（§3.4）中：

- `sections[].confidence` 为简单字符串枚举（`high/medium/low`），无数值体系定义
- `sections[].metadata` 仅在最后可信度标注节出现（`fact_count/inference_count/pending_count`），其他节无 metadata
- 缺少 `sections[].source` 字段标明该节信息来源

#### 3.2 补充字段定义

##### 3.2.1 section.metadata 完整结构

在每个 `section` 对象中扩展 `metadata`：

```json
{
  "section_id": "s2",
  "title": "旺衰与用神",
  "content": "日主戊土生于申月...",
  "confidence": "medium",
  "data_tier": "inference",
  "order": 2,
  "metadata": {
    "fact_count": 3,
    "inference_count": 5,
    "pending_count": 1,
    "source": "bazi_engine.v1",
    "related_sections": ["s1", "s3"],
    "key_terms": ["日主", "旺衰", "用神"],
    "generated_at": "2026-07-08T12:05:00Z",
    "ai_model": "deepseek-chat-v3",
    "review_status": "unreviewed"
  }
}
```

| metadata 字段 | 类型 | 必填 | 描述 |
|-------------|------|------|------|
| `fact_count` | int | ❌ | 该节中事实性陈述的数量 |
| `inference_count` | int | ❌ | 该节中分析性推理的数量 |
| `pending_count` | int | ❌ | 该节中推测性建议的数量 |
| `source` | string | ❌ | 该节内容生成来源（`bazi_engine.v1` / `ai.deepseek-chat-v3` / `user_input` / `manual`）|
| `related_sections` | array[string] | ❌ | 关联的 section_id 列表，用于报告内交叉引用 |
| `key_terms` | array[string] | ❌ | 该节涉及的关键命理术语列表 |
| `generated_at` | string | ❌ | ISO 8601 该节生成时间 |
| `ai_model` | string | ❌ | 生成该节内容的 AI 模型标识 |
| `review_status` | string | ❌ | 审核状态：`unreviewed` / `confirmed` / `disputed`（用户反馈触发）|

##### 3.2.2 置信度分数体系【待产品负责人确认具体结构】

当前 `confidence` 为字符串枚举（`high/medium/low`），建议引入数值与枚举的映射体系：

| 字符串值 | 数值范围 | 含义 | 适用场景 |
|---------|---------|------|---------|
| `high` | 0.80-1.00 | 高置信度：确定性算法输出或多源一致确认 | 排盘结果、用户已确认的事实 |
| `medium` | 0.50-0.79 | 中置信度：单一学派推理或 AI 单次分析 | 用神推导、格局判定（无用户反馈时）|
| `low` | 0.00-0.49 | 低置信度：推测性建议、多学派分歧 | 流年预测、事件推测、无确认的 pending 项 |

**数值计算建议：**

- **排盘层（fact）**：固定为 `1.0` / `high`，排盘是确定性算法
- **推理层（inference）**：由共识引擎的多学派一致度决定——4 学派一致 → 0.85，3 学派一致 → 0.70，2 学派分歧 → 0.50
- **推测层（pending）**：AI 自行标注，默认 ≤ 0.40

> **待确认：** 置信度的数值计算方式、是否需要独立于 data_tier 的三层体系、是否允许客户端过滤（如只显示 confidence ≥ 0.60 的内容）。

#### 3.3 涉及修改的位置

- `docs/13_API接口契约设计.md` §3.4 — `GET /api/v1/reports/{report_id}` Response 中 `sections[].metadata` 扩展为完整结构
- `docs/13_API接口契约设计.md` §3.4 — 补充置信度分数体系的枚举定义

### 4. 知识库（Knowledge）模块 — 缺失字段

#### 4.1 当前状态

当前知识库相关接口（§3.6）：

- `GET /api/v1/knowledge/terms` — 返回 `{term_id, name, category, summary, related_terms}`，缺少内部定义内容
- `GET /api/v1/knowledge/terms/{term_id}` — 无当前定义，内容结构待设计
- `POST /api/v1/knowledge/search` — 返回同 terms 列表格式
- `GET /api/v1/knowledge/references` — 定义存在但 response 结构未设计

#### 4.2 补充字段定义

##### 4.2.1 术语详情结构（`GET /api/v1/knowledge/terms/{term_id}`）

新增端点响应结构：

```json
{
  "term_id": "t_001",
  "name": "十神",
  "category": "ten_gods",
  "summary": "十神是八字命理中用于分析日主与其他天干关系的十种分类...",
  "definition": "详细的术语定义文本，包含起源、用法、注意事项...",
  "aliases": ["十星", "六神"],
  "related_terms": [
    {"term_id": "t_002", "name": "正官", "relationship": "子类别"},
    {"term_id": "t_003", "name": "七杀", "relationship": "子类别"}
  ],
  "sources": [
    {"name": "渊海子平", "chapter": "卷三·论十神", "citation": "...原文引用..."},
    {"name": "三命通会", "chapter": "卷五·十神篇", "citation": "...原文引用..."}
  ],
  "examples": [
    {"scenario": "年柱庚金为食神", "description": "表示命主聪明才艺...", "data_tier": "fact"}
  ],
  "visibility": "public",
  "data_tier": "fact"
}
```

| 字段 | 类型 | 必填 | 描述 |
|------|------|------|------|
| `definition` | string | ✅ | 完整定义文本，支持 Markdown 格式 |
| `aliases` | array[string] | ❌ | 别名列表 |
| `related_terms` | array[object] | ❌ | 关联术语（含关系类型）|
| `sources` | array[object] | ❌ | 经典典籍引用来源列表 |
| `examples` | array[object] | ❌ | 应用示例 |
| `visibility` | string | ✅ | `public` / `premium`（免费/付费用户可见）|
| `data_tier` | string | ✅ | ADR-012 三层标记 |

##### 4.2.2 引用/来源结构

在知识库中统一引用来源的数据结构：

| 字段 | 类型 | 必填 | 描述 | 示例 |
|------|------|------|------|------|
| `source_id` | string | ✅ | 来源唯一标识 | `src_001` |
| `name` | string | ✅ | 典籍/文章名称 | `"渊海子平"` |
| `author` | string | ❌ | 作者 | `"徐子平"` |
| `era` | string | ❌ | 成书年代 | `"宋代"` |
| `chapter` | string | ❌ | 章节引用 | `"卷三·论十神"` |
| `citation` | string | ❌ | 原文引用文本 | `"原文：..."` |
| `reliability` | string | ❌ | 典籍可靠度：`canonical`(经典) / `reference`(参考) / `modern`(现代) | `"canonical"` |
| `data_tier` | string | ✅ | ADR-012 三层标记 | `"fact"` |

##### 4.2.3 知识图谱关系定义【待产品负责人确认具体结构】

当前 `related_terms` 仅以字符串列表形式存在（如 `["正官", "七杀"]`），缺少关系语义。建议扩展为带关系类型的结构：

| 关系类型 | 含义 | 示例 |
|---------|------|------|
| `hierarchy.parent` | 父类别 | 十神 → 正官 |
| `hierarchy.child` | 子类别 | 正官 → 十神 |
| `synonym` | 同义词 | 十神 = 十星 |
| `antonym` | 对立/相反 | 正官 ↔ 七杀 |
| `derivation` | 派生关系 | 日主 → 十神（日主是十神计算的基准）|
| `reference` | 引用/参考 | 用神 → 扶抑法 |
| `conflict` | 学派分歧 | 子平派用神 ≠ 梁湘润派用神 |
| `see_also` | 参见 | 格局 → 用神 |

> **待确认：** 知识图谱的存储方式（SQLite 关系表 / JSON 字段内嵌 / 独立图数据库 light 实现）、关系类型枚举的完整列表、图谱数据的维护机制（AI 自动构建 vs 专家手动录入）。

#### 4.3 涉及修改的位置

- `docs/13_API接口契约设计.md` §3.6 — `GET /api/v1/knowledge/terms/{term_id}` 新增完整 Response 结构定义
- `docs/13_API接口契约设计.md` §3.6 — `GET /api/v1/knowledge/references` 新增 Response 结构定义
- `docs/13_API接口契约设计.md` §3.6 — 补充 Source 对象和关系类型的枚举定义
