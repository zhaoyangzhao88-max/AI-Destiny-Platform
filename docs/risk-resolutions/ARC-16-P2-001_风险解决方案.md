## 根因分析

### 问题定位

> 原文引用审计报告 §4.3.2：
> "| 4 | **文档版本号规则不明确** | 🟢 P2 | 大部分文档标记为 v0.1，部分为 v0.2，但无版本号递增规则（何时升 minor？何时升 major？）"

### 根本原因

当前项目文档体系（15 份文档）使用了语义化版本号（SemVer）格式 `v<major>.<minor>`，但缺乏版本号变更的具体指导规则。具体表现为：

1. **v0.1 → v0.2 的触发条件未定义** —— 什么程度的修改应该递增 minor？是"任何修改"还是"结构性修改"？
2. **v0.x → v1.0 的触发条件未定义** —— v1.0 是"首次审阅通过"还是"内容完整"？
3. **v1.x → v2.0 的触发条件未定义** —— 什么构成"大版本更新"？
4. **不同状态文档的版本规则未区分** —— Draft、Approved、Deprecated 文档的版本号应如何管理？

当前实际使用中的版本号：
- 大部分文档（01/02/03/04 等）= v0.1
- 08_系统架构设计.md = v0.2（经历了一次较大修改）
- 15_Phase2架构审计报告.md = v1.0（完成报告后的标记）
- 18_架构风险跟踪表.md = v1.0（同上）

可以看出 v1.0 在实际使用中对应"首次完成/报告定稿"，v0.2 对应"有重要更新"，但规则从未被正式定义。

### 影响范围

- 跨文档引用时无法通过版本号判断文档的成熟度
- 新 AI 或新开发者无法通过版本号快速理解文档变更历史
- 随着文档进入 Phase 3（代码开发期），文档更新频率增加，无规则将导致版本号随意使用

---

## 解决方案

### 方案描述

在 `12_AI协作开发规范.md` §5.2「文档状态定义」中补充文档版本号规则。采用 SemVer 语义化版本号格式 `v<major>.<minor>`，并针对项目不同文档状态定义明确的递增触发条件。

---

#### 新增内容：文档版本号规则（插入到 §5.2 文档状态表之后）

```markdown
### 5.2.1 文档版本号规则

所有项目文档采用语义化版本号 **`v<major>.<minor>`** 格式。
版本号与文档状态独立——同一份文档在不同状态下可能有不同的版本号节奏。

#### 版本号范围定义

| 版本范围 | 适用阶段 | 含义 | 示例 |
|---------|---------|------|------|
| **v0.x** | Draft 阶段 | 草稿迭代，内容持续修改中。x 从 1 开始递增 | v0.1 → v0.2 → v0.3 |
| **v1.0** | 首次确认 | 文档完成首次审阅/批准/报告完成，内容已稳定 | Draft → Approved 时设为 v1.0 |
| **v1.x** | 确认后修订 | 对已确认文档的局部修改（非结构性变更） | v1.0 → v1.1 → v1.2 |
| **v2.0+** | 结构性变更 | 文档结构、核心决策或范围发生重大变化 | v1.x → v2.0（需配合 ADR 变更流程） |

#### 版本号递增条件

**v0.x → v0.x+1（minor 递增，Draft 阶段）：**

满足以下任一条件即应递增 minor 版本号：
- 文档内容发生实质性修改（不仅仅是错别字修正或格式调整）
- 新增或删除了一个完整章节
- 核心数据（字段定义、API 路由、状态码等）发生变化
- 文档中的图表/架构图发生变更
- 简单格式调整、错别字修正、标点符号统一 **不递增** minor 版本号

> **v0.x 不设上限：** 文档在 Draft 阶段的版本号可以持续递增（v0.9 → v0.10 → v0.11），无需强求在某个数值升为 v1.0。

**v0.x → v1.0（首次确认）：**

满足**全部**以下条件：
1. 文档完成了目的定义的全部内容（如"全部章节已填充，无占位符"）
2. 文档通过了产品负责人或指定评审者的正式审阅
3. 文档状态从 🟡 Draft 变更为 ✅ Accepted Concept 或 ✅ Approved
4. 对于审计报告类文档：审计执行完成，报告输出定稿

**v1.x → v1.x+1（确认后修订）：**

满足以下任一条件：
- 对已确认文档的局部章节进行修改（非结构性）
- 新增示例、备注、引用链接等补充内容
- 根据反馈修正事实性错误
- 同步更新以反映关联文档的变化

> 已确认文档（v1.0+）的 minor 版本递增前应在 `docs/11_AI协作日志.md` 中记录修改摘要和影响范围。

**v1.x → v2.0+（结构性变更）：**

满足以下任一条件：
- 文档所指涉的核心决策发生变更（如架构方向改变）
- 文档结构发生重新组织（章节重排、内容拆分/合并）
- 文档适用的范围发生重大变化（如从"V1 专用"扩展到"V1/V2 通用"）
- 超过 30% 的内容被重写或替换
- 多个连续 minor 版本累积了足够多的变更（建议在 v1.9 后评估是否升为 v2.0）

> 从 v1.x 升为 v2.0 通常需要配合 ADR 变更流程，并在协作日志中记录结构性变更的缘由。

#### 版本号与文档状态的对应关系

| 文档状态 | 典型版本范围 | 说明 |
|---------|------------|------|
| 🟡 Draft | v0.1 ~ v0.x | 草稿阶段，持续递增 minor |
| 🟡 Draft（待审核） | v0.x | 草稿阶段，等待产品负责人确认 |
| ✅ Accepted Concept | v0.x 或 v1.0 | 概念已确认，可标记为 v1.0 或保持高于 v0.5 的 Draft 版本 |
| ✅ Approved | v1.0+ | 已确认后至少 v1.0 |
| 🔴 Deprecated | 保留原版本号 + Deprecated 标记 | 版本号不随废弃变化 |

#### 版本号声明位置

每个文档的头部 meta 块中必须包含版本号字段：

```markdown
> **版本:** v<major>.<minor> | **日期:** YYYY-MM-DD | **状态:** <Draft/Approved/Deprecated>
```

每次版本号变更时同步更新日期字段。

#### 例外情况

- **ADR 记录（04_ADR_产品决策记录.md）：** ADR 本身有独立的决策编号体系，文档级版本号仅反映 ADR 集合的编辑状态
- **协作日志（11_AI协作日志.md）：** 日志按时间追加，使用日志条目编号而非版本号
- **风险跟踪表（18_架构风险跟踪表.md）：** 版本号反映跟踪表的编辑迭代，不反映其中风险条目的状态
```
