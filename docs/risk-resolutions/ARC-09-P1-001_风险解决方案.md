## 根因分析

### 实际 git 现状（A1 实测，2026-07-16）

| 检查项 | 结果 |
|--------|------|
| 项目目录下是否有独立 `.git` | ✅ 有（`./git` 存在；`git rev-parse --show-toplevel` → `E:/VSCODE/AI-Destiny-Platform`） |
| 是否隶属于父目录 `E:\VSCODE` 大仓库 | ❌ 否（`cd ..` 后 `git status` / `rev-parse` 全部空输出，exit 128，父目录非 git 仓库） |
| `docs/` 是否已被 git 追踪 | 部分已追踪（`CLAUDE.md`、`04/08/09/10/11/12/13/18_*`、`HANDOVER.md` 为 *modified*）；其余（sprint 报告、`risk-resolutions/`）为 *untracked* |

### 文档为何与现实脱节

`09_开发规范.md` §9.4 当前标注为「【待确认】—— 建议从 `E:\VSCODE` 父仓库中分离独立仓库」，属早期框架性推测；项目初始化时实际已直接建立独立仓库，但该推测性描述未随真实状态同步更新。`HANDOVER.md` §1「父 git 仓库 `E:\VSCODE\.git`（当前在此大仓库中，建议独立）」、§4.1「Git 仓库独立方案 … 当前在大仓库中」、§5 下一步「P0 ── Git 仓库独立」均沿用了同一过时假设。

### 关键结论

> **A3 原拟推荐「为项目初始化独立 `.git`」的前提（项目在共享仓库中）不成立。** 项目本就是独立仓库，不存在需要拆分的父仓库。因此本方案**不执行 `git init` / submodule**，而是将动作替换为**修订过时文档**。

## 解决方案

### 推荐方案：修订过时文档（不拆分、不 git init）

经 A1 实测，项目独立 git 仓库**已具备**，无需任何仓库结构变更。真正待办是消除文档与现实的不一致。

#### 具体修订步骤

1. **`09_开发规范.md` §9.4 Git 工作流**
   - 移除「建议从 `E:\VSCODE` 父仓库中分离独立仓库」与「【待确认】」标注
   - 改为说明：「项目已是独立 git 仓库（`.git` 位于 `E:\VSCODE\AI-Destiny-Platform\`），父目录 `E:\VSCODE` 非 git 仓库；如未来需要多项目协同再评估 `git submodule`，不建议在单人项目阶段引入」

2. **`HANDOVER.md` §1 项目概要**
   - 「父 git 仓库：`E:\VSCODE\.git`（当前在此大仓库中，建议独立）」→「独立 git 仓库（无父仓库）」

3. **`HANDOVER.md` §4.1 需产品负责人决策**
   - 「Git 仓库独立方案 … 当前在大仓库中」→「已完成：项目为独立 git 仓库，无需拆分」

4. **`HANDOVER.md` §5 下一步工作**
   - 移除「P0 ── Git 仓库独立」或标注为「✅ 已完成」

#### 为什么不再拆分

- 项目已是独立仓库，**没有父仓库可拆**；强行 `git init` 或 submodule 反而会破坏现有结构、丢失已提交的 2 个 commit 历史
- 当前处于文档建设期、尚无业务代码，正是「已独立」而非「待独立」的状态，拆分成本窗口期的论点（针对「尚未独立」场景）在此不适用

#### 若 A1 发现被父仓库追踪（实测未出现）才需的迁移步骤

> 本任务实测未触发此分支；保留作为兜底说明。

若未来某次实测发现项目确被父仓库 git 追踪（即 `git rev-parse --show-toplevel` 指向 `E:\VSCODE`），则：
- 因当前无业务代码提交、历史价值低，建议**直接开新仓库**（`git init` 于项目目录，放弃历史）而非 `git filter-repo`（迁移复杂度高、收益低）
- 不采用 submodule（给单人项目增加不必要的操作复杂度）
