## 风险概述

| 字段 | 内容 |
|------|------|
| **风险 ID** | ARC-15-P2-002 |
| **问题描述** | `10_测试规范.md` 当前为框架性占位符（§10.1 明确标注「本文件为框架性说明，具体测试策略待技术选型和核心逻辑确定后完成」），仅有概念性描述（§10.2 测试挑战、§10.3 测试层级、§10.4 命理测试的特殊性），未针对命理领域的特殊性设计可执行的测试策略 |

> **反模式警示：** 外部项目 bazi-skill-dist（Sprint 11 深度扫描）的排盘算法被评估为 2/5 不可用于生产，其核心原因之一就是 **核心排盘算法零测试覆盖**。该项目的 `tests/` 目录中，28 个存活测试文件覆盖的是基础设施层（runtime_standard/replay/observability），而四柱计算、十神、大运、学派编译器、共识引擎等**核心命理逻辑从未有过测试**。这正是本方案要避免的反模式。

## 根因分析

### 直接原因

`10_测试规范.md` 的撰写时间处于产品设计阶段，当时项目尚无代码（仅占位符），因此测试规范停留在框架性概念层面，缺少可执行的具体策略。§10.4「命理测试的特殊性」仅提供了一组单侧示例，未形成完整的测试策略体系。

### 根本原因

测试规范的细化依赖于技术选型和核心逻辑的确定——测试什么、如何测试取决于具体的技术实现。在 Phase 2（文档建设期）将测试规范留为占位符是合理的阶段决策。但进入 Phase 3（代码开发）前，必须提供足够的测试指导，否则开发者自行猜测测试策略导致质量不稳定。

### bazi-skill-dist 反模式参考

bazi-skill-dist 的测试分布清晰地展示了**命理项目测试的反模式**：

```
反模式特征：
├── ✅ 基础设施层大量测试（runtime_standard 80个、replay 140个）
├── ✅ 情感安全层高质量测试（cos 748行）
├── ✅ 可观测试（130个）、快照（70个）
├── ❌ 四柱计算测试：零
├── ❌ 十神/藏干测试：零
├── ❌ 大运/起运测试：零
├── ❌ 学派编译器测试：零（曾有25个测试文件，已全部删除）
├── ❌ 共识引擎测试：零
└── ❌ 核心排盘算法测试：从未存在
```

**教训：** 命理项目的测试不能仅关注基础设施的"健壮性"，必须为核心业务逻辑单独设计测试体系。基础设施测试再多也无法保证命理计算结果的正确性。

---

## 解决方案

### 方案描述

在 `10_测试规范.md` 中新增 **§10.5「命理特殊性测试策略」**，从黄金数据集、一致性测试、状态机测试、边界案例四个维度定义可执行的测试策略。

#### 新增 §10.5 完整内容

```markdown
## 10.5 命理特殊性测试策略

### 10.5.1 概述

命理项目的测试与通用软件测试有本质差异：

- **排盘是确定性的**：相同输入 → 相同输出，适合完全自动化断言测试
- **推理（AI 部分）是非确定性的**：LLM 输出每次不同，适合结构校验而非精确断言
- **命理规则有学派差异**：同一命局在不同学派理论下结论可能不同，测试需指定"测试参考学派"

因此命理测试需要独立的测试策略体系，不能简单套用通用测试分类。

### 10.5.2 黄金数据集

#### 定义

黄金数据集是一组**事先验证正确的命理案例集合**，用于作为测试的"标准答案"。每个案例包含完整输入和已知正确的预期输出，由命理专家或从经典典籍中整理提取。

#### 数据集结构

每条黄金案例的结构：

```json
{
  "case_id": "golden_001",
  "description": "1990年庚午年命主综合案例",
  "input": {
    "birth_year": 1990,
    "birth_month": 8,
    "birth_day": 15,
    "birth_hour": 12,
    "birth_minute": 0,
    "gender": "male",
    "birth_place": "北京",
    "timezone": "Asia/Shanghai",
    "longitude": 116.4,
    "latitude": 39.9,
    "calendar_type": "solar"
  },
  "expected_output": {
    "year_pillar": {"heavenly_stem": "庚", "earthly_branch": "午"},
    "month_pillar": {"heavenly_stem": "甲", "earthly_branch": "申"},
    "day_pillar": {"heavenly_stem": "戊", "earthly_branch": "寅"},
    "hour_pillar": {"heavenly_stem": "戊", "earthly_branch": "午"},
    "day_master": "戊",
    "yong_shen_candidates": ["金", "水"],
    "pattern": "杀印相生格"
  },
  "source": "渊海子平·卷四",
  "verified_by": "产品负责人/命理专家（待确认）",
  "created_at": "2026-07-16"
}
```

#### 数据集规模要求

| 阶段 | 最少案例数 | 覆盖要求 |
|------|-----------|---------|
| Phase 3 开发启动 | 10 | 覆盖 5 种主流日主 + 正格/特殊格各至少 1 例 |
| Phase 3 中期 | 50 | 覆盖全部 10 日主 + 全部 10 正格 + 常见特殊格 |
| V1 发布前 | 200+ | 覆盖所有已知命理变化组合 + 边界条件 |

#### 数据集的维护与版本

- 黄金数据集是独立于测试代码的 JSON/YAML 文件集合，存放在 `tests/golden_dataset/` 目录下
- 数据集本身进行版本控制（Git），每新增/修改案例需经产品负责人或命理专家确认
- 每个案例的 `source` 字段应追溯至经典典籍或专家确认的原始出处

### 10.5.3 一致性测试

#### 定义

一致性测试验证**确定性逻辑在相同输入下始终产生相同输出**。这是命理测试的基石，因为排盘、五行统计、大运顺逆等全都是确定性算法。

#### 测试范围

| 测试目标 | 被测模块 | 验证方式 | 自动化程度 |
|---------|---------|---------|-----------|
| 四柱排盘 | BaziEngine.paipan() | 输入相同 → 四柱相同 | 完全自动化 |
| 藏干映射 | ChartParser | 输入单个地支 → 输出固定藏干列表 | 完全自动化 |
| 十神计算 | ten_god_count() | 日主 + 天干 → 固定十神 | 完全自动化 |
| 大运方向 | dayun_rules.resolve_direction() | 性别+年干 → 固定顺逆 | 完全自动化 |
| 起运年龄 | start_age_calculator | 输入相同 → 起运年龄相同 | 完全自动化（需先修复节气依赖）|
| 五行计数 | element_count() | 输入四柱 → 固定计数 | 完全自动化 |
| 格局判定 | pattern识别 | 输入相同 → 格局相同 | 完全自动化（规则引擎）|

#### 执行策略

- 对每个确定性函数：随机生成 100 组输入，每组执行 3 次，验证 3 次结果完全一致
- 对于四柱排盘：黄金数据集的每个案例执行 5 次，验证每次结果与预期输出完全一致
- 对函数的纯函数性质做**幂等性断言**：`assert compute(input) == compute(input)` 在不同时间/环境下成立

#### 避免 bazi-skill-dist 反模式

bazi-skill-dist 的教训表明：**不可假定"当前正确"的命理算法会一直正确**。一致性测试必须在 CI 中持续运行：

- 每次代码修改后自动运行全部一致性测试（耗时 < 30s 则全局运行，> 30s 可先运行黄金数据集子集）
- 任何修改排盘、五行、大运相关代码的 commit 必须附带一致性测试通过证明
- 如果修改导致黄金数据集案例失败，**必须更新数据集中的预期输出并注明理由**

### 10.5.4 状态机测试

#### 定义

状态机测试验证知命八字分析 6 步流程（ADR-009）的状态转换是否正确。

#### 状态机定义

6 步流程 + Task 状态机：

```
Step 1: 数据校验        → status: pending, stage: data_parsing
Step 2: 排盘计算        → status: processing, stage: four_pillars
Step 3: AI 初步分析     → status: processing, stage: ai_inference
Step 4: AI 动态追问     → status: processing, stage: ai_dynamic_qa
Step 5: 用户反馈验证    → status: processing, stage: user_feedback
Step 6: 最终报告生成    → status: processing, stage: report_gen
完成                     → status: completed
失败（任意步骤）         → status: failed
取消                     → status: cancelled
超时                     → status: timed_out
```

见 `14_WebSocket实时通信协议设计.md` §4.3 Task 状态机。

#### 测试用例

| 测试场景 | 初始状态 | 输入/事件 | 预期新状态 | 验证点 |
|---------|---------|-----------|-----------|--------|
| 正常完成 | pending | Step 1→2→3→4→5→6 逐级推进 | completed | 每一步 progress 递增、stage 切换正确 |
| 中途取消 | processing | 用户发送取消请求 | cancelled | 资源清理、不可恢复 |
| AI 调用失败 | processing | Step 3 调用 LLM 超时 | failed | error.code=10003, error.failed_stage="ai_inference" |
| 用户反馈 reject | processing (Step 5) | 用户标记 pending 为 fact | processing (Step 3 重新推理) | 触发 AI 重新分析，progress 回退 |
| 超时终止 | processing | 超过 300s 无完成 | timed_out | 推送 task.failed 含 timed_out 原因 |
| 重复提交 | pending | 相同 chart_id + analysis_type 再次提交 | 返回已有 task_id | 幂等性保障，不创建重复任务 |

#### 状态转换覆盖率要求

- 状态机中所有**合法转换**（pending → processing → completed 等正向路径）100% 覆盖
- 所有**非法转换**（如 completed → processing 的反向）测试断言拒绝
- 所有**异常转换**（任何步骤 → failed/cancelled/timed_out）至少 1 个用例

### 10.5.5 命理边界案例

#### 定义

命理计算中存在多个特殊边界情况，这些情况在通用测试中容易被忽略，但在命理领域不可缺失。

#### 边界案例清单

| 边界类型 | 案例示例 | 测试点 | 是否在黄金数据集中 |
|---------|---------|--------|-----------------|
| **立春换年** | 2026-02-03 立春日前后的时辰 | 立春前用乙巳年柱，立春后用丙午年柱 | ✅ 必须包含 |
| **节气换月** | 2026-03-05 惊蛰日前后的时辰 | 惊蛰前用正月庚寅，惊蛰后用二月辛卯 | ✅ 必须包含 |
| **闰月** | 2023-06-20（闰二月） | 闰月是否做特殊处理、排盘是否受影响 | ⚠️ 建议包含 |
| **未知时辰** | birth_hour=null, birth_time_unknown=true | 程序不崩溃，输出按子时参考或省略时柱 | ✅ 必须包含 |
| **跨时区** | 出生地在美国纽约（UTC-5） | 真太阳时校准正确，与北京时间的排盘结果不同 | ⚠️ 建议包含 |
| **跨日界线** | 出生在东十二区与西十二区边界 | 日柱不因日界线导致日期错误 | ❌ V1 可暂缺 |
| **公历/农历切换** | calendar_type="lunar" | 农历日期正确转换为公历后再排盘 | ✅ 必须包含 |
| **午夜时辰** | birth_hour=23 或 0 | 23:00-00:59 属于子时，日柱算同一天还是下一天 | ✅ 必须包含 |
| **出生年边界** | 1900-01-01 或当年末 | 跨世纪边界、跨甲子周期（60年轮回）的排盘正确性 | ⚠️ 建议包含 |
| **性别特殊** | 非二元性别或未提供 | 排盘需要性别决定阴阳顺逆 | ✅ 必须包含 |

#### 边界测试自动化策略

- **确定性边界**（立春、节气、闰月、午夜时辰等）：完全自动化断言测试
- **非确定性边界**（未知时辰、跨时区等）：结构校验 + 合理性检查（程序不崩溃 + 输出格式正确 + 无 NaN/None）
- 每个边界条件作为独立测试函数，便于失败时的精准定位

### 10.5.6 测试自动化与 CI 集成

#### 执行层级

| 层级 | 触发时机 | 执行范围 | 最大耗时 | 失败阻断级别 |
|------|---------|---------|---------|------------|
| L1 快速 | 每次保存文件（watch模式）| 黄金数据集子集（5条）+ 一致性（10条）| < 10s | ❌ 不阻断开发 |
| L2 完整 | 每次 git commit pre-hook | 黄金数据集全量 + 一致性全量 + 状态机 + 边界 | < 60s | ⚠️ 阻断 commit |
| L3 深度 | 每次 CI Pipeline（PR/merge）| L2 + 随机生成 100 组输入 + 幂等性验证 | < 5min | 🔴 阻断合并 |

#### 覆盖率要求

- 排盘核心模块（四柱计算、藏干、十神）：**行覆盖率 ≥ 95%**
- 大运相关模块（顺逆、起运、干支序列）：**行覆盖率 ≥ 90%**
- 状态机逻辑（Task 状态转换）：**行覆盖率 100%**
- AI 分析模块（LLM 调用编排）：**结构校验覆盖率 100%**（不要求行覆盖率，因其为 LLM 调用）
```
