## 根因分析

### 版本历史机制的核心矛盾

版本历史的本质是在「存储成本」和「回溯成本」之间做权衡：

| 维度 | 全快照 | 全增量（纯 JSON Patch） | 混合方案 |
|------|--------|------------------------|---------|
| 存储成本 | 高（每版本存全量） | 低（仅存 diff） | 中 |
| 回溯读取速度 | 快（直接读取） | 慢（需逐个 apply patch 到基础版本） | 中（偶尔读快照） |
| 实现复杂度 | 低（直接存 JSON） | 中（JSON Patch 构建/apply 逻辑） | 中高（patch + checkpoint 管理） |
| diff 对比 | 需单独实现 | 天然支持 | 天然支持 |

没有"完美"方案，只有适合场景的权衡。V1 的用户档案特点是：
- **写频率低**：用户不会频繁修改出生信息（每月几次量级）
- **读频率中**：用户查看档案、回溯历史（每周几次量级）
- **diff 对比需求明确**：用户比较「修改前 vs 修改后」是核心 UX 需求（ADR-013）
- **存储非瓶颈**：V1 本地 SQLite，JSON 字段存储成本极低（即使每次存全快照，一年也远 < 100MB）

## 解决方案

### 已确认方案

**方案 A：纯快照方案（Snapshots Only）** — 经产品负责人确认，V1 采用全快照存储。

每次修改用户档案时，存储一份完整的档案 JSON 快照。两个快照之间的 diff 由查询时动态计算生成。

**决策理由：**

| 维度 | 说明 |
|------|------|
| **实现复杂度最低** | 每条版本记录直接存完整 `snapshot` JSON，无 patch 构建/apply 逻辑，无需 checkpoint 管理，SQLite 单表 + JSON 字段即可实现，开发和测试成本最低 |
| **回溯性能最优** | 回溯任意版本直接读一条记录即可，无需沿 patch 链逐个 apply，对「历史查看」核心 UX 场景最友好 |
| **V1 存储成本可忽略** | 单用户档案 JSON < 100KB，本地 SQLite 存储，即使按每月 5 次修改、每年 60 个版本计算，单用户年存储量 < 6MB。V1 用户规模下完全不是瓶颈 |
| **无累积误差风险** | 全快照方案不存在 patch 链累积浮点/精度误差的问题，数据可靠性最高 |
| **技术独立性** | 无需引入外部库（如 `jsonpatch` / `json-diff`），减少依赖链 |

**方案约束：**
- 历史版本记录为仅追加写（append-only），不支持物理删除（如需 GDPR 合规删除需全表扫描 + 重建）
- 大面积 AI 批量更新时快照体积可能突变，需在服务端做合理分段
- diff 对比由服务端在查询时动态计算（逐字段对比两个快照），不在存储层做

**执行事项：**
1. 移除 `08_系统架构设计.md` §8.6 中「具体存储方案 ——【待确认】」标注，替换为本决策
2. `13_API接口契约设计.md` §3.2 版本端点增加全快照存储约束说明
3. 数据库版本历史表（`version_history`）的 `snapshot` 字段定义为 `TEXT` / `JSON` 类型，每条记录独立存储完整 JSON

### 曾评估的替代方案

以下方案在决策过程中被评估，因不适合 V1 场景而未被采纳，保留记录以供 V2 扩展时参考。

#### 选项 B：增量混合方案（JSON Patch + 每 30 次快照 Checkpoint）

使用 JSON Patch（RFC 6902）存储增量变更。基线版本存全量快照，后续版本仅存 patch diff，每积累 30 次 patch 后生成一个新的全量快照 checkpoint。

| 维度 | 评价 |
|------|------|
| **优点** | 存储效率最高；diff 天然存在无需额外计算；30 次 checkpoint 保证回溯上限；RFC 6902 标准成熟 |
| **缺点** | 实现复杂度最高（patch 构建/apply/checkpoint 全流程）；V1 版本数极低，复杂方案收益几乎为零；需引入外部库依赖 |
| **未采纳原因** | V1 场景下复杂度的代价远超存储收益，建议 V2 用户量大时重新评估 |

#### 选项 C：纯增量方案（JSON Patch Only，无快照）

仅存储 JSON Patch，不保留任何全量快照。回溯时从第一个版本开始，逐个 apply 所有 patch 直到目标版本。

| 维度 | 评价 |
|------|------|
| **优点** | 存储空间最小；架构最"干净"；patch 即审计日志 |
| **缺点** | 回溯性能随版本增长线性恶化；patch 链单点故障风险高；diff 对比复杂度高于预期 |
| **未采纳原因** | 回溯性能退化与「历史查看」核心 UX 目标冲突，不适合以版本回溯为主的用户档案场景 |
