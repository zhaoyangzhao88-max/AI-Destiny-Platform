## 根因分析

### 为什么部署方案必须在编码前决策

`08_系统架构.md` §8.2 定义了 Windows APP 与 Python 核心服务解耦架构，核心服务作为独立进程运行。这个架构带来了部署上的独特挑战：

| 挑战 | 说明 | 影响 |
|------|------|------|
| **Python 运行时依赖** | 用户机器可能没有 Python 3.12，或版本不兼容 | 无法运行核心服务 |
| **多进程打包** | 两个独立进程（PySide6 APP + Python 核心服务）需打包为单一安装包 | 安装/卸载流程复杂 |
| **版本兼容性** | APP 与核心服务的版本需保持兼容矩阵 | 升级必须协调 |
| **进程管理** | 核心服务作为子进程的启动/停止/崩溃恢复策略 | 用户体验问题 |
| **调试与日志** | 打包后的核心服务如何输出日志和错误信息 | 问题排查困难 |

## 解决方案

### 已确认方案（方案 A — PyInstaller Sidecar）

**产品负责人已确认采用方案 A（PyInstaller Sidecar）作为 V1 核心服务部署方案。** 以下是方案细节与决策依据。

#### 方案描述

Python 核心服务使用 PyInstaller 打包为独立的 `core-service.exe`（单文件或单目录模式），随 APP 安装包一同分发。PySide6 UI 进程在启动时以子进程方式启动 `core-service.exe`，通过 localhost IPC 通信。

#### 决策依据

| 维度 | 评价 |
|------|------|
| **技术可行性** | PyInstaller 是 Python 打包最成熟的工具之一，社区支持完善，对 fastapi/uvicorn 等依赖兼容性好 |
| **用户体验** | 用户零配置——无需安装 Python 运行时，安装包解压即用 |
| **工程复杂度** | 中等——需配置 hidden imports 和子进程生命周期管理，但无 C/C++ 胶水代码 |
| **版本管理灵活性** | 高——核心服务与 APP 可独立升级，Major 版本锁定确保兼容 |
| **未来演进** | 架构与 Web V2 一致——侧车进程模式可直接迁移为远程服务 |
| **机会成本** | 方案 B（嵌入式 DLL）和方案 C（预安装运行时）分别在复杂度和用户体验上不可接受 |

#### 版本兼容策略建议

```
核心服务版本: MAJOR.MINOR.PATCH (SemVer)
APP 版本:     MAJOR.MINOR.PATCH (SemVer)

兼容矩阵:
  APP v1.x.x → Core v1.y.z (y >= y_min, z >= z_min)
  APP v2.x.x → Core v2.y.z (y >= y_min, z >= z_min)

规则:
  - Major 版本降级：需要确认兼容性表
  - Minor 版本：APP 和 Core 各自独立升级，Major 不变即可
  - Patch 版本：完全独立，向前兼容
  - Core Major 升级视为架构变更，需经过 ADR 流程
```

#### 打包流程

```
APP 安装过程中:
  1. 解压 APP.exe + core-service.exe + 依赖资源
  2. 注册 APP 快捷方式
  3. 首次启动时校验 core-service.exe 完整性

APP 启动流程:
  1. PySide6 启动
  2. PySide6 以子进程方式启动 core-service.exe
  3. 等待核心服务 health check 通过（轮询 localhost:port /health）
  4. 启动完成，APP 进入就绪状态

APP 更新流程:
  1. 新版本安装包同时更新 APP.exe 和 core-service.exe
  2. 核心服务 Major 版本升级时，APP 强制升级
  3. 核心服务 Minor/Patch 升级时，APP 可选升级
```

#### 方案优缺点

**优点：**
- 用户零配置：无需安装 Python 运行时，捆绑所有依赖
- 版本独立管理：核心服务与 APP 可独立升级，Major 版本锁定确保兼容
- 子进程管理灵活：PySide6 可监控核心服务状态，崩溃后自动重启
- 便于调试：开发模式下可直接运行 .py 脚本，打包模式运行 .exe

**缺点：**
- 打包体积较大：Python 解释器 + 依赖库 ≈ 50-100MB 额外体积
- PyInstaller 的隐藏导入（hidden imports）需手动配置，遗漏会导致运行时错误
- 多进程架构增加了进程间通信和生命周期的管理复杂度
- 打包后的可执行文件反编译风险（纯 Python 字节码）

#### 适用场景

- V1 桌面 APP 快速配送
- 核心服务依赖的 Python 库较多（排盘引擎、fastapi、uvicorn 等）

### 曾评估的替代方案

以下方案在决策过程中被评估但未采纳，保留记录供后续阶段参考。

#### 方案 B：嵌入式 Python DLL（V3/V4 体积优化阶段再考虑）

**核心思路：** 将 Python 3.12 运行时以 DLL 形式嵌入到 PySide6 APP 进程中，函数直接调用替代 IPC。

| 核心优点 | 核心缺点 |
|---------|---------|
| 打包体积最小（~5MB 额外） | 实现复杂度最高，需 C/C++ 胶水代码 |
| 通信零延迟，无序列化开销 | Python GIL 与 PySide6 事件循环交互需谨慎处理 |
| 单一进程，无子进程管理 | 核心服务崩溃将导致整个 APP 崩溃 |
| | 无法独立升级核心服务 |
| | Web V2 时需要将嵌入的逻辑重新独立，增加解耦成本 |

**未采纳理由：** 复杂度高、隔离性差、与未来 Web V2 架构方向矛盾，不适合 V1 快速交付。若 V3/V4 阶段对体积有极致要求可重新评估。

#### 方案 C：预安装 Python 运行时（不适合商业桌面产品）

**核心思路：** 不打包 Python 运行时，要求用户自行安装 Python 3.12，APP 在首次运行时检测/引导安装。

| 核心优点 | 核心缺点 |
|---------|---------|
| 打包最简单，安装包 < 20MB | 用户体验最差——普通用户难以自行安装 Python |
| Python 开发者友好，可直接编辑脚本 | 多版本共存问题、环境差异大、调试困难 |
| 符合开源社区分发习惯 | 安装失败率高（路径、权限、环境变量问题） |
| | 不适合商业产品分发 |

**未采纳理由：** 用户体验不可接受，不符合商业桌面产品要求。
