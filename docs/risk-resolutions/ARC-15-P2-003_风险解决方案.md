## 根因分析

### 当前配置（§3.3 心跳参数）

| 参数 | 当前值 | 说明 |
|------|--------|------|
| ping 间隔 | 30s | 客户端发送 `heartbeat.ping` 的周期 |
| **pong 超时** | **30s** | 客户端等待 `heartbeat.pong` 的超时时间（问题值）|
| 连接空闲超时 | 90s | 服务端关闭无活动连接的阈值 |
| 重连初始间隔 | 1s | 第一次重连等待时间 |
| 重连最大间隔 | 30s | 指数退避上限 |

### 问题分析

当 `pong_timeout = ping_interval` 时，出现以下问题：

1. **无缓冲余量**：客户端发送 ping 后必须在 30s 内收到 pong，否则立即判定服务端不可用。但 `ping_interval=30s` 本身是发送间隔——这意味着 ping 传输耗时 + 服务端处理耗时 + pong 回传耗时必须 < 30s，否则触发断连
2. **Windows 桌面 APP 的特殊性**：即使服务端在同一台机器上通过 localhost 通信，以下场景仍可能造成瞬态延迟：
   - Windows 系统进入睡眠/休眠后恢复（suspend/resume），网络栈重建需要几百毫秒到数秒
   - CPU 被高负载任务抢占（如大文件 I/O、Windows Update 后台扫描）
   - Python GC（Garbage Collection）触发 STW（Stop-The-World）暂停
   - FastAPI 的异步事件循环单次循环被长时间协程阻塞
3. **重连成本**：一次误判断连会触发指数退避重连（1s → 2s → 4s → 8s → 16s → 30s...），每次重连还需重新鉴权（JWT 验证），在极端情况下可能形成"断连-重连-再断连"的震荡循环

### 为什么 2 倍是合理值

- **60s = 2 × 30s**：为 ping 往返留出 30s 的缓冲窗口。正常情况下 ping/pong 在 localhost 上的往返时间 < 1ms，60s 的超时为瞬态异常提供了充裕的恢复时间
- **与业界实践一致**：WebSocket 通用实践中，pong_timeout 通常为 ping_interval 的 2-3 倍
- **不会降低异常检测灵敏度**：即使 60s 的超时，相比 TCP 层面的数分钟超时仍然非常激进。客户端在 60s 内未收到 pong 则判定断连，仍能保证在 2 个心跳周期内发现异常
- **不会与服务端空闲超时冲突**：服务端空闲超时配置为 90s（§3.3 表第三行），客户端的 60s pong 超时在服务端 90s 判定之前触发，时序合理

## 解决方案

### 方案描述

将 `14_WebSocket实时通信协议设计.md` §3.3 心跳参数表中的 pong 超时值从 30s 修改为 60s。

#### 修改内容

**修改前：**

| 参数 | 值 | 说明 |
|------|-----|------|
| ping 间隔 | 30s | 客户端发送 `heartbeat.ping` 的周期 |
| pong 超时 | 30s | 客户端等待 `heartbeat.pong` 的超时时间 |
| 连接空闲超时 | 90s | 服务端关闭无活动连接的阈值 |

**修改后：**

| 参数 | 值 | 说明 |
|------|-----|------|
| ping 间隔 | 30s | 客户端发送 `heartbeat.ping` 的周期 |
| pong 超时 | **60s** | 客户端等待 `heartbeat.pong` 的超时时间（设置为 ping 间隔的 2 倍，为网络抖动提供缓冲）|
| 连接空闲超时 | 90s | 服务端关闭无活动连接的阈值 |

#### 同步检查其他引用

检查 `14_WebSocket实时通信协议设计.md` 中是否有其他位置引用了 pong 超时值：

- §3.2 阶段详情·⑤ heartbeat：客户端 30 秒未收到 pong 则判定服务端不可用（此处的"30 秒"也应同步修改为"60 秒"）

#### 时序校验

验证修改后各超时值的时序关系：

```
t=0s   客户端发送 heartbeat.ping
              │
t=0~30s  ←── 正常情况：服务端应在 30s ping 间隔内回复 pong
              │
t=30s   客户端准备发送下一个 ping
              │
t=30~60s ←── 缓冲窗口：上一个 ping 的 pong 仍未回复，继续等待
              │
t=60s   判定断连 → 触发重连
              │
t=90s   服务端连接空闲超时（比客户端判定早 30s 触发，合理）
```

> 服务端空闲超时是独立维度：服务端 90s 未收到**任何消息**（含 ping 在内的所有上行消息）时断开连接。客户端最坏情况下 60s 判定断连，早于服务端 90s，因此客户端总是主动发起重连方，时序合理。
