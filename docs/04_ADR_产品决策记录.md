## ADR-001：产品核心定位

| 字段 | 内容 |
|------|------|
| **决策编号** | ADR-001 |
| **决策日期** | 2026-07-08 |
| **状态** | ✅ Approved |
| **决策者** | 产品负责人 |

**内容：**

产品定位为 **AI 人生档案顾问**。

融合三大核心模块：

1. **A：AI 陪伴体验** —— 以 AI 角色「知命」为载体，提供长期有温度的陪伴
2. **B：专业命理研究能力** —— 基于四柱八字体系提供严谨分析
3. **C：长期人生档案系统** —— 持续记录用户人生事件

**理由：** 三合一模式区别于传统算命的一次性和普通 AI 聊天的无深度，建立长期用户价值。

```

    ↓
```

## ADR-012：数据三层体系

| 字段 | 内容 |
|------|------|
| **决策编号** | ADR-012 |
| **决策日期** | 2026-07-08 |
| **状态** | ✅ Approved |
| **决策者** | 产品负责人 |

**内容：**

数据管理体系分为三个层级：

| 层级 | 说明 | 优先级 |
|------|------|--------|
| **事实（Fact）** | 用户确认的人生事件、出生信息等 | 最高优先级 |
| **推测（Inference）** | AI 基于命理规则的分析推论 | 中间优先级 |
| **待验证（Pending）** | AI 提出的假设，等待用户反馈确认 | 最低优先级 |

**核心原则：**
- AI 推测 ≠ 事实
- 用户确认的事实优先于任何 AI 推测
- 每个数据点标注其层级归属

**理由：** 建立明确的数据可信度体系，防止 AI 推测被误认为事实，同时也为后续用户验证提供追踪能力。

## ADR-013：用户资料版本历史

| 字段 | 内容 |
|------|------|
| **决策编号** | ADR-013 |
| **决策日期** | 2026-07-08 |
| **状态** | ✅ Accepted Concept（概念已确认，技术方案待设计） |
| **决策者** | 产品负责人 |

**内容：**

产品层面已经确认：

- 用户资料不能简单覆盖旧数据
- 用户修改信息时，需要保留历史版本
- AI 需要知道资料变化过程

以下内容尚未完成技术设计：

- 数据库存储方案
- 快照还是增量存储
- 历史数据查询方式
- 用户端历史查看界面
- 性能优化方案

这些内容将在后续 **V1 技术架构设计阶段** 进一步确定。

**理由：** 用户人生档案的核心价值在于可追溯。覆盖历史会丢失档案的完整性，版本机制保证用户可以回看任何时间点的档案状态。技术方案需在架构设计阶段细化。

## ADR-014：混合架构——本地优先 + 云端扩展

| 字段 | 内容 |
|------|------|
| **决策编号** | ADR-014 |
| **决策日期** | 2026-07-08 |
| **状态** | ✅ Draft |
| **决策者** | 产品负责人 |

**Context（背景）：**

知命 AI 涉及用户敏感数据：出生信息、人生事件、命理分析结果。用户对数据隐私有较高要求。同时，AI 命理分析需要较大算力（LLM 推理），纯粹本地或纯粹云端方案都存在明显缺陷：

- 纯本地：无法使用大型 AI 模型，分析质量受限
- 纯云端：用户隐私数据全部上云，信任门槛高，离线不可用

**Decision（决策）：**

V1 采用 **"本地优先 + 云端扩展"** 混合架构：

- **本地优先：** 核心八字排盘、用户档案数据、基础分析逻辑优先在本地处理
- **云端扩展：** AI 深度分析需要 LLM 算力时，按需调用云端 API
- **数据最小化：** 上传到云端的数据仅限必要的分析上下文，原始档案数据留本地
- **渐进增强：** 离网可用基础功能，联网自动启用增强分析

**Consequences（选择理由）：**

- ✅ 用户数据隐私保护更好，降低信任门槛
- ✅ 离线状态下基础功能仍可用（排盘、档案浏览）
- ✅ 按需调用云 API，降低成本
- ⚠️ 本地存储增加客户端复杂度
- ⚠️ 数据同步机制需要额外设计
- ⚠️ 云端 AI 分析仍需网络，非完全离线

**后续影响：**
- 影响数据库选型（需要支持本地 + 远程同步）
- 需要设计本地与云端的数据同步 / 冲突解决机制
- 影响 AI 分析流程的接口设计（本地预处理 → 云端推理 → 本地缓存）

## ADR-015：Windows APP 优先

| 字段 | 内容 |
|------|------|
| **决策编号** | ADR-015 |
| **决策日期** | 2026-07-08 |
| **状态** | ✅ Draft |
| **决策者** | 产品负责人 |

**Context（背景）：**

开发资源有限，需要聚焦。目标用户群体（命理爱好者）中 Windows 桌面用户占比较高。移动端和 Web 端虽然覆盖面广，但开发维护成本高，V1 阶段应集中资源做出优质体验。

**Decision（决策）：**

V1 阶段优先开发 **Windows APP**，未来根据市场反馈逐步扩展：

| 阶段 | 平台 | 优先级 |
|------|------|--------|
| V1 | Windows APP | 🥇 首发 |
| V2 | Web 端 | 🥈 扩展 |
| V3 | 移动端（iOS / Android） | 🥉 扩展 |

**Consequences（选择理由）：**

- ✅ 开发成本最低，资源聚焦，快速迭代
- ✅ Windows 桌面端用户体验可控
- ✅ 便于本地优先架构落地（本地文件存储、离线计算）
- ⚠️ 窗口期错失移动端用户
- ⚠️ 未来跨平台需考虑 UI 适配
- ⚠️ Windows 原生开发 vs 跨平台框架需技术选型

**后续影响：**
- 技术选型偏向 Windows 原生（C# / WinUI3）或跨平台（Flutter / Tauri / Electron）
- UI 设计需考虑后续向其他平台迁移的可扩展性
- 产品发布渠道（Microsoft Store / 官网下载）

## ADR-016：单核心知命 AI Agent 架构

| 字段 | 内容 |
|------|------|
| **决策编号** | ADR-016 |
| **决策日期** | 2026-07-08 |
| **状态** | ✅ Draft |
| **决策者** | 产品负责人 |

**Context（背景）：**

项目初期核心目标是验证产品价值和用户体验。多 Agent 架构（如规划 Agent、推理 Agent、记忆 Agent 分离）虽然扩展性强，但在 V1 阶段会增加不必要的架构复杂度和开发成本。

**Decision（决策）：**

V1 采用 **单核心知命 AI Agent 架构**：

- 一个统一的 AI Agent 处理所有用户交互
- Agent 内部包含对话管理、命理分析、档案管理三大能力
- 通过模块化提示词（Prompt）区分不同任务类型
- 预留多 Agent 扩展接口，为未来拆分做准备

```
V1 架构：
┌─────────────┐
│  知命 AI     │  ← 单个 Agent，统一处理
│  Agent      │     对话 / 分析 / 档案
└─────────────┘

未来演进：
┌──────┐ ┌──────┐ ┌──────┐
│对话  │ │分析  │ │档案  │  ← 多 Agent 分工
│Agent │ │Agent │ │Agent │
└──────┘ └──────┘ └──────┘
```

**Consequences（选择理由）：**

- ✅ 架构简单，开发效率高，适合快速验证
- ✅ 问题定位容易，调试成本低
- ✅ Prompt 层面可以灵活调整行为
- ⚠️ 单一 Agent 职责边界不清晰，长期可能膨胀
- ⚠️ 未来拆分为多 Agent 时可能涉及重构
- ⚠️ 单一 Agent 的上下文窗口压力更大

**后续影响：**
- Prompt 工程时需按模块设计，方便未来拆分
- Agent 调用接口需预留扩展点（如 plugin / tool 机制）
- 监控指标需关注单一 Agent 的 Token 消耗

## ADR-017：核心业务服务与客户端解耦原则

| 字段 | 内容 |
|------|------|
| **决策编号** | ADR-017 |
| **决策日期** | 2026-07-08 |
| **状态** | ✅ Draft |
| **决策者** | 产品负责人 |

**Context（背景）：**

知命 AI 定位为长期发展的平台化产品（ADR-001），未来将逐步扩展到 Web 端和移动端（ADR-015 的 V2/V3 规划）。当前 V1 架构将表示层、业务逻辑层和数据层统一部署在 Windows APP 内部（见 `08_系统架构设计.md` 8.1 架构图），虽然简化了初期开发，但存在以下耦合风险：

1. **客户端锁定风险：** 如果核心业务逻辑（排盘、AI 分析、档案管理）与 Windows APP 深度绑定，未来拓展到 Web 或移动端时需要大量重构
2. **AI 模型僵化：** 如果 AI 分析逻辑直接嵌入 APP 代码，更换或升级 AI 模型（如从云端 LLM 切换到本地模型）将涉及 APP 全量更新
3. **能力复用困难：** 排盘引擎、分析流程等核心能力无法被其他客户端或自动化脚本独立调用
4. **测试复杂度：** 业务逻辑与 UI 耦合导致自动化测试成本高

同时，考虑到 V1 本地优先架构（ADR-014）和 Windows APP 首发策略（ADR-015），需要找到一个既不增加过多初期复杂度、又为未来解耦预留路径的折中方案。

**Decision（决策）：**

V1 采用 **Windows APP（表示层）+ Python 核心服务（本地）** 分层架构：

```
┌─────────────────────────────────────┐
│        Windows APP（表示层）          │
│  ┌──────────┐  ┌──────────────────┐ │
│  │ UI 组件   │  │ 客户端服务适配层   │ │
│  │ (对话/    │  │ (API 调用/数据     │ │
│  │  排盘/    │  │  序列化/本地IPC)   │ │
│  │  档案)    │  └────────┬─────────┘ │
│  └──────────┘           │           │
└──────────────────────────┼──────────┘
                           │
                    ┌──────┴──────┐
                    │  本地 IPC    │
                    │ (gRPC/HTTP) │
                    └──────┬──────┘
                           │
┌──────────────────────────┼──────────┐
│        Python 核心服务（本地进程）      │
│  ┌──────────┐  ┌──────────┐  ┌────┐ │
│  │ 排盘引擎   │  │ AI 分析   │  │档案 │ │
│  │ (本地)    │  │ 引擎     │  │管理 │ │
│  └──────────┘  │ (LLM调用) │  └────┘ │
│                └──────────┘         │
│  ┌──────────┐  ┌──────────┐        │
│  │ 规则推理   │  │ 数据层级  │        │
│  │ 引擎      │  │ 管理器    │        │
│  └──────────┘  └──────────┘        │
└─────────────────────────────────────┘
```

核心要点：

1. **Windows APP** 负责：UI 渲染、用户交互、本地 IPC 调用、离线状态管理
2. **Python 核心服务** 负责：排盘计算、AI 分析编排、档案数据管理、规则推理
3. **通信方式：** 通过本地 IPC（建议 gRPC 或本地 HTTP）进行进程间通信
4. **部署方式：** Python 核心服务作为 Windows APP 的子进程/伴随进程启动，用户无感知

**Consequences（影响）：**

**优点：**

- ✅ **未来 Web 扩展：** Python 核心服务天然可作为 Web 后端 API 复用，Web 端直接调用同一套服务
- ✅ **移动端扩展：** 移动端通过网络 API 调用部署在云端的 Python 服务副本即可
- ✅ **AI 模型可替换：** AI 分析逻辑在 Python 服务内部，更换模型（如从 GPT 切换到本地模型）不影响客户端
- ✅ **核心能力独立演进：** 排盘引擎、分析流程可独立更新，无需 APP 全量发布
- ✅ **可测试性：** 核心逻辑作为独立服务，支持自动化测试和 CI/CD

**代价：**

- ⚠️ **初期架构复杂度增加：** 需要设计 IPC 通信协议、进程生命周期管理、错误处理机制
- ⚠️ **部署复杂度：** 需要打包 Python 运行环境或使用 PyInstaller 等工具打包为独立二进制
- ⚠️ **进程通信开销：** 相比进程内调用，IPC 有性能损耗（但对命理分析场景影响可接受）
- ⚠️ **状态同步：** 客户端与核心服务之间需要维护会话状态一致性

**后续影响：**

1. **技术选型影响：** 客户端框架需能方便嵌入和管理子进程（如 Tauri 的 sidecar 能力、Electron 的 child_process）；Python 服务端框架需支持轻量级 API 服务（如 FastAPI）
2. **IPC 协议设计：** 需要定义清晰的 API 契约文档（建议 OpenAPI 或 protobuf 定义）
3. **安装包设计：** V1 安装包需同时包含 APP 二进制和 Python 运行时/打包产物
4. **版本管理：** APP 与核心服务可能各自独立版本号，需设计版本兼容性矩阵
5. **未来云部署：** 核心服务的 API 设计需考虑从本地 IPC 平滑过渡到远程网络 API

## ADR-018：V1 Windows APP 技术架构选择

| 字段 | 内容 |
|------|------|
| **决策编号** | ADR-018 |
| **决策日期** | 2026-07-08 |
| **状态** | ✅ Draft |
| **决策者** | 产品负责人 |

**Context（背景）：**

知命 AI 产品定位为长期 AI 人生档案平台（ADR-001）。Windows APP 是 V1 阶段的第一用户入口（ADR-015），但不是唯一入口。产品未来需要逐步扩展到 Web 端和移动端（ADR-015 V2/V3 规划）。

因此，V1 技术架构需要同时满足两个核心要求：

1. **V1 开发效率：** 聚焦 Windows 桌面端，快速推出 MVP 验证产品价值
2. **未来 Web 扩展：** 不因 V1 技术选型造成未来向 Web 端迁移的巨大重构成本

**Decision（决策）：**

V1 采用 **PySide6 + Vue 渐进式混合架构**：

| 层级 | 技术 | 职责 |
|------|------|------|
| **Windows 桌面入口** | PySide6 | 用户入口、界面容器、系统交互（通知/任务栏/进程管理）、本地应用管理 |
| **富交互 UI 模块** | Vue（渐进引入） | 复杂交互页面、AI 聊天界面、高级报告展示、数据可视化 |
| **核心业务服务** | Python | 排盘计算、AI Agent 编排、用户档案管理、知识库与规则推理 |

架构示意：

```
Windows APP（PySide6）
    │
    ▼
Vue UI 模块（逐步引入）
    │
    ▼
Python 核心服务
    │
    ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
    │ 排盘  │ │AI    │ │用户   │ │知识库 │
    │ 引擎  │ │Agent │ │档案   │ │      │
    └──────┘ └──────┘ └──────┘ └──────┘
```

**Consequences（影响）：**

**优点：**

- ✅ 保持 Python 核心技术路线，与 ADR-017（核心服务与客户端解耦）天然兼容
- ✅ 支持未来 Web 扩展——Python 核心服务可直接复用为 Web 后端
- ✅ Vue 作为渐进式 UI 能力，可提升 AI 产品的交互体验（聊天界面、可视化报告等）
- ✅ 避免未来从纯原生向 Web 混合架构的完全重构

**代价：**

- ⚠️ 技术体系增加：团队需要同时掌握 PySide6 和 Vue 两个前端技术栈
- ⚠️ 需要同时维护桌面层和 Web UI 层的共存逻辑
- ⚠️ 初期设计复杂度提高：需要预留 PySide6 与 Vue 的集成通道
- ⚠️ PySide6 与 Vue 之间的通信/集成方式待后续专项确定

**后续影响：**

1. **V1 初期不强制引入 Vue** — 以 PySide6 为起步界面框架，Vue 仅在需要复杂交互功能时逐步引入
2. **按产品需求驱动引入节奏** — Vue 的引入时机由产品功能复杂度决定，非技术驱动
3. **后续需要确定 Web 通信方式** — PySide6 嵌入 Vue（QWebEngineView）还是独立窗口/进程运行，需专项决策
4. **共享 Python 核心服务** — 无论前端使用 PySide6 还是 Vue，都调用同一套 Python 核心服务（ADR-017）

## ADR-019：客户端与核心服务通信协议设计

| 字段 | 内容 |
|------|------|
| **决策编号** | ADR-019 |
| **决策日期** | 2026-07-08 |
| **状态** | ✅ Draft |
| **决策者** | 产品负责人 |

**Context（背景）：**

ADR-017 已确定 Windows APP 与 Python 核心服务分离，客户端（表示层）与核心服务（业务逻辑层）通过本地 IPC 通信。ADR-018 已确定 PySide6 作为 V1 Windows 客户端技术路线，并支持未来向 Web 端扩展。

当前缺少客户端与核心服务之间的通信标准。需要选择一种既适合 V1 本地运行、又方便未来 Web 扩展的通信方式。

**候选方案分析：**

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| 1. 直接函数调用 | 客户端直接导入并调用核心服务模块 | 实现简单，零通信开销 | 强耦合，客户端与核心服务必须在同一进程，不利于未来 Web 扩展 |
| 2. localhost HTTP API | 核心服务启动本地 HTTP 服务，客户端通过 localhost 调用 RESTful API | 简单易调试，与未来 Web API 天然兼容，协议通用性强 | 有少量通信开销（序列化/反序列化），对 AI 流式输出需额外机制 |
| 3. gRPC | 使用 Protocol Buffers 定义服务接口，通过 HTTP/2 进行双向通信 | 高性能，强类型协议约束，支持双向流 | 开发复杂度较高，调试工具链不如 HTTP 成熟，V1 阶段过度设计 |
| 4. WebSocket | 在客户端与服务间建立持久连接，支持全双工通信 | 天然支持实时通信与推送，适合 AI 流式输出场景 | 作为全部业务接口不合适，请求/响应模式不如 HTTP 直观 |

**Decision（决策）：**

V1 采用 **HTTP API + WebSocket 双通道通信架构**。

| 通道 | 协议 | 技术实现 | 负责场景 |
|------|------|----------|----------|
| **通道一：HTTP API** | HTTP/REST | FastAPI（Python） | 用户资料管理、八字排盘请求、人生档案查询、报告生成、配置管理 |
| **通道二：WebSocket** | WebSocket | FastAPI WebSocket | AI 实时对话、Token 流式输出、长任务状态反馈、分析过程进度通知 |

**双通道通信架构示意：**

```
┌─────────────────────────────────────────────────────┐
│                  PySide6 Client                      │
│  ┌──────────────────────────────────────────────┐   │
│  │              API Client Layer                 │   │
│  │  ┌──────────────────┐  ┌──────────────────┐  │   │
│  │  │   HTTP Client    │  │  WebSocket Client │  │   │
│  │  │  (REST 请求)     │  │  (实时通信/流式)   │  │   │
│  │  └────────┬─────────┘  └────────┬─────────┘  │   │
│  └───────────┼─────────────────────┼─────────────┘   │
└──────────────┼─────────────────────┼─────────────────┘
               │                     │
               │   localhost HTTP    │   localhost WebSocket
               │   (127.0.0.1:Port)  │   (ws://127.0.0.1:Port)
               │                     │
┌──────────────┼─────────────────────┼─────────────────┐
│              ▼                     ▼                 │
│        FastAPI API Server (Python Core Service)      │
│  ┌──────────────────────────────────────────────┐   │
│  │  ┌──────────────┐  ┌──────────────────────┐  │   │
│  │  │ HTTP Router   │  │  WebSocket Router     │  │   │
│  │  │ (REST 接口)   │  │ (实时消息通道)       │  │   │
│  │  └──────┬───────┘  └──────────┬───────────┘  │   │
│  │         │                     │               │   │
│  │         └─────────┬───────────┘               │   │
│  │                   ▼                           │   │
│  │         Python 核心业务服务                    │   │
│  │    ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐   │   │
│  │    │ 排盘  │  │ AI   │  │ 档案  │  │ 知识库│   │   │
│  │    │ 引擎  │  │ Agent│  │ 管理  │  │      │   │   │
│  │    └──────┘  └──────┘  └──────┘  └──────┘   │   │
│  └──────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

**Consequences（影响）：**

**优点：**

- ✅ **未来 Web 扩展兼容：** HTTP API 天然可作为 Web 后端 API 复用，Web 前端直接调用同一套接口
- ✅ **开发简单，易调试：** HTTP 协议工具链成熟（Postman、curl、浏览器 DevTools），WebSocket 也有丰富调试支持
- ✅ **支持 AI 实时体验：** WebSocket 通道为 AI 对话流式输出提供了天然支持，提升用户体验
- ✅ **服务边界清晰：** 双通道各司其职，HTTP 负责请求/响应式业务，WebSocket 负责实时通信，职责明确

**代价：**

- ⚠️ **增加 API 设计工作量：** 需要定义完整的 API 接口契约（路由、请求/响应格式、状态码）
- ⚠️ **需要版本管理：** API 随着产品演进需要版本化（如 `/api/v1/`），避免客户端不兼容
- ⚠️ **需要处理服务启动失败：** PySide6 客户端需要检测核心服务是否已启动，处理连接失败和重试逻辑
- ⚠️ **本地通信安全需考虑：** 虽然仅监听 localhost，仍需防止其他本地进程的未授权访问

**后续影响：**

1. **需要设计 API 接口契约** —— 定义所有 HTTP endpoint 和 WebSocket 消息格式
2. **需要定义请求/响应数据结构** —— 统一的 JSON 序列化格式、分页、排序、过滤规范
3. **需要设计错误码体系** —— 业务错误码、HTTP 状态码映射、错误信息国际化
4. **需要设计核心服务健康检查机制** —— 客户端检测服务状态、自动重连、心跳保活

## ADR-020：V1 PySide6 与 Vue 集成方案

| 字段 | 内容 |
|------|------|
| **决策编号** | ADR-020 |
| **决策日期** | 2026-07-08 |
| **状态** | ✅ Draft |
| **决策者** | 产品负责人 |
| **关联风险** | docs/18 ARC-15-P0-001 |

**Context（背景）：**

ADR-018 已确定 V1 客户端技术架构为 **PySide6 + Vue 渐进式混合架构**，ADR-017 已确定 Windows APP（表示层）与 Python 核心服务（业务逻辑层）解耦。但 ADR-018 的「后续影响」第 3 条明确指出：*PySide6 嵌入 Vue（QWebEngineView）还是独立窗口/进程运行，需专项决策*。

若不在编码前确定集成机制，将面临三种差异巨大的方案：

| 方案 | 打包体积 | 内存 | 通信延迟 | 风险 |
|------|---------|------|---------|------|
| QWebEngineView 嵌入式 | Chromium 已捆绑（~100MB+，ADR-018 已接受） | 单进程，~200MB 差异可控 | 进程内，低 | 选错则需大规模重构 |
| 独立窗口（PySide6 窗口 + 独立 Vue 窗口） | 同左 | 双进程 | 跨窗口 IPC | 破坏单窗口体验、窗口管理复杂 |
| 纯进程间通信（Vue 独立进程） | 同左 | 双进程 | IPC 开销 | 架构复杂度最高 |

**Decision（决策）：**

V1 采用 **QWebEngineView 嵌入式集成方案**：

1. **渲染方式** — Vue 应用作为 PySide6 `QWebEngineView` 的内嵌网页渲染。Vue 构建产物以**本地静态资源**或 **本地 HTTP 服务（localhost）** 形式提供，由 PySide6 主窗口的 `QWebEngineView` 加载。
2. **宿主集成通道** — PySide6 与内嵌 Vue 之间通过 **Qt WebChannel** 进行类型安全的双向调用（Python ↔ JS），用于窗口级事件（通知、任务栏、进程生命周期）。
3. **业务通信通道** — Vue 内业务交互仍遵循 ADR-019，通过 **HTTP API + WebSocket（`/ws/v1/session`）** 调用本地 Python 核心服务。形成**双层通信**：WebChannel = 宿主集成，HTTP/WS = 业务通信，两者职责清晰。
4. **渐进引入** — 遵循 ADR-018 后续影响第 1 条：V1 初期以 PySide6 原生控件起步，Vue 仅在需要复杂交互（AI 聊天界面、高级报告、数据可视化）时按需引入，经 `QWebEngineView` 嵌入。

```
V1（桌面首发）
  Windows APP（PySide6 主窗口）
    ├── 原生控件区（通知/任务栏/进程管理）
    └── QWebEngineView（内嵌 Vue 模块，按需）
          │  Qt WebChannel（宿主集成：窗口事件/生命周期）
          │
          └── Vue 业务页（AI 聊天/报告/可视化）
                │  HTTP API + WebSocket（业务通信，ADR-019）
                ▼
          Python 核心服务（本地）
```

**Consequences（影响）：**

**优点：**

- ✅ **单进程、单窗口** — 用户体验统一，无独立窗口管理复杂度
- ✅ **原生集成** — QWebEngineView 在 PySide6 中原生支持，WebChannel 提供类型安全双向调用
- ✅ **Chromium 代价已接受** — 打包体积与内存差异在 ADR-018 决策时已纳入考量
- ✅ **Vue 复用性** — 未来 Web 扩展（ADR-018 8.3.4）时 Vue 代码直接复用，仅剥离 PySide6 外壳
- ✅ **职责分离** — 宿主集成（WebChannel）与业务通信（HTTP/WS）互不耦合

**代价：**

- ⚠️ **Chromium 体积** — 已在 ADR-018 接受，但需优化首屏加载（预编译、懒加载 Vue 模块）
- ⚠️ **WebChannel 线程模型** — Python 主线程与 JS 引擎的调用需处理线程安全
- ⚠️ **调试复杂度** — 需同时调试 Python 侧与内嵌 Web 侧，建议统一 DevTools 接入

**后续影响：**

1. **V1 初期不强制引入 Vue** — 以 PySide6 原生控件起步，降低初期复杂度（呼应 ADR-018 后续影响第 1 条）
2. **集成机制明确** — `docs/08_系统架构设计.md` 8.3 节补充 8.3.5 集成机制小节，引用本 ADR
3. **未来 Web 扩展路径不变** — Vue 模块经 `QWebEngineView` 嵌入仅影响 V1 桌面壳，V2 Web 端直接由 Web 前端复用 Vue 代码（ADR-018 8.3.4）
4. **不在本 ADR 范围** — 整体 Windows APP 框架选型（Tauri/Electron/Flutter，08_架构 8.1.4【待确认】）属 P1-001，由后续任务处理
