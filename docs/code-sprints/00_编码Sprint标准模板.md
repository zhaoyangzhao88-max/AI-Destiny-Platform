# Phase 3 编码 Sprint — 标准提示词模板

> 本文件是 External Research Center 与 Claude Code 协作开发知命项目代码阶段的
> 固定协议，2026-07-17 经产品负责人逐条确认后定稿。每次编码 Sprint 的提示词
> 都基于本模板填充，不再重复解释规则。

---

## 已确认的九条规则

| # | 规则 | 决定 |
|---|------|------|
| 1 | 测试证据 | 强制跑真实测试框架，终端**原始输出**整段贴出，不允许用"测试通过"代替 |
| 1.1 | **测试证据的载体（2026-07-19 补充，血的教训）** | **任何涉及 self_check/测试运行结果的 Sprint，原始输出必须写入报告文件并上传，不允许只在对话/聊天窗口里用文字复述结果——哪怕任务本身很简单（如单纯的 git commit 检查点）。原因：Sprint 42 曾发生一次聊天窗口复述的"self_check 输出"与代码实际运行结果完全不符（内容疑似虚构），因为当时的提示词允许"不生成文件、直接对话回报"，这个口子必须永久关闭。** |
| 2 | 改动粒度 | 小颗粒——一个 Sprint 只做一个函数/一个模块 |
| 3 | git commit 节奏 | 沿用治理阶段模式，攒几个 Sprint 一起提交（产品负责人已知晓小颗粒+攒批的潜在风险，接受该权衡） |
| 4 | secret 扫描 | 只在任务涉及 AI 调用/配置相关的 Sprint 才检查，不是每次都做 |
| 5 | 静态检查（lint/类型） | 现阶段不强制，优先功能正确性，风格检查留到后面集中处理 |
| 6 | self_check() 自检 | 只有核心领域逻辑（排盘/用神/AI 调用）强制要求，工具类代码不强求 |
| 7 | 测试/自检失败处理 | 允许 Claude Code 自行重试 1-2 次，但**每次失败**都必须在报告里如实记录原始输出，不能只交最终成功版本 |
| 8 | 报告文件位置 | 新目录 `docs/code-sprints/`，与治理文档的 `docs/sprints/` 分开 |
| 9 | 上下文附带 | 默认只贴改动部分（diff），只有产品负责人明确要求时才贴完整现有代码/文档上下文 |

## 每个编码 Sprint 提示词的固定结构

```markdown
# Sprint <N> — <任务名称>

## 项目路径
E:\VSCODE\AI-Destiny-Platform

## 本 Sprint 改动范围（小颗粒，明确边界）
[只描述这一个函数/模块，不要在一个 Sprint 里塞多个不相关改动]

## 执行模式建议
[直接执行 / 建议 Plan Mode] — 一句话原因
判断依据：任务范围单一、步骤已具体到函数/文件级别 → 直接执行；
任务本身有开放性（需设计/决策空间）、涉及多文件依赖需先梳理顺序、
或执行中可能需要 Claude Code 自行判断如何调整方案 → 建议 Plan Mode。
由产品负责人在生成提示词时判断标注，Claude Code 不自行决定。

## 铁律（每次都带）
1. 先读实际内容再改，不凭记忆/摘要操作
2. 每处改动记录 diff
3. 如果方案/文档描述与实际代码现状不符，如实报告，不擅自扩大范围
4. 不 git commit（除非提示词明确说明这是提交检查点）
5. [如任务涉及 AI 调用/配置] 本 Sprint 涉及密钥/配置相关代码，
   完成后必须跑一次 secret 扫描：
   grep -rn "sk-\|api_key\s*=\s*['\"]" <改动文件> 
   确认无硬编码密钥后才能算完成

## 具体任务
[任务细节]

## 测试要求
必须跑 <pytest/对应框架> 并把终端原始输出整段贴进报告，
禁止用"测试通过"这类叙述代替。
如果测试失败：允许重试 1-2 次，但每次失败都要把失败时的原始输出
完整贴出，最终报告要能看到"失败→重试→成功"或"失败→仍失败"的
完整过程，不能只交最终版本。

## 文件系统证据（新增，2026-07-17 补充）
凡涉及新建/移动文件的 Sprint，收尾时必须贴一次
`find <本次改动的目录> -type f` 的原始输出，作为独立于"已创建"这类
叙述之外的文件系统证据。不能只靠"测试跑通了所以路径应该没问题"
去反推文件位置——测试跑通只能验证被直接 import 的文件路径正确，
不能验证报告里列出的其他文件（如空的 __init__.py、辅助文件）
是否精确落在预期位置。

## self_check 要求（如果本 Sprint 涉及核心领域逻辑：排盘/用神/AI调用）
必须实现/更新 self_check()，用已知边界案例验证，输出结果贴进报告。
（工具类/非核心代码可跳过此项）

## 输出要求
生成 `docs/code-sprints/sprint<N>-<主题>.md`，结构：
- 任务描述
- 改动 diff（新建文件可用"新建文件"标注+完整内容代替传统 diff 格式）
- 测试原始输出（含失败重试记录，如有）
- 文件系统证据（`find <改动目录> -type f` 原始输出，凡涉及新建/移动文件必须有）
- self_check 输出（如适用）
- secret 扫描结果（如适用）
- 审核摘要（意外情况/需要产品负责人判断的地方/新发现问题）

完成后上传，不需要额外上传上下文文件，除非我在提示词里特别要求
附带完整现有代码/文档内容。

**没有例外情况**：即使本 Sprint 只是一次 git commit 检查点、或任务
看起来极其简单，只要涉及运行 self_check/测试，就必须生成报告文件、
把原始输出写入文件后上传，不允许只在对话里用文字复述运行结果
（2026-07-19 教训：这个口子曾被打开过一次，导致一份内容疑似虚构的
"self_check 输出"未经文件化就被当作提交依据）。
```

## 使用说明

1. 产品负责人（Chief Research Architect / External Research Center 视角）
   基于此模板，针对每个具体任务生成填充后的提示词
2. Claude Code 执行后生成的报告统一存放 `docs/code-sprints/`
3. 产品负责人审核报告后，决定：批准进入下一 Sprint / 要求修正 / 
   标记为提交检查点（每 3-5 个 Sprint 建议做一次 git commit，
   避免小颗粒改动长期堆积无法追溯）
4. 如遇到方案与实际代码不符、测试持续失败、或发现方案外的新问题，
   审核方（产品负责人 + External Research Center）先给出结论，
   再决定下一个 Sprint 怎么走，不允许 Claude Code 自行扩大任务范围
