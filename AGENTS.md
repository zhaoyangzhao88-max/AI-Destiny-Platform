## 项目概述

**知命 AI（ZhiMing AI）**——基于东方智慧 × 人工智能的人生档案顾问，核心建立在**四柱八字**中国命理学之上。区别于一次性算命，它是一个长期陪伴的 AI 角色，按严格的信任层级排盘、分析并归档用户人生。

- **状态：** 纯文档仓库。Phase 2（设计与文档建设）已收口；**Phase 3（编码）于 2026-07-10 刚获批启动，尚无任何业务代码。** 不要假设任何实现已存在——文档是唯一事实来源，代码不是。
- **语言：** 所有文档、ADR 与 UX 均为**简体中文**；命理术语为中文。代码注释与变量名用英文；面向用户的回复用中文。
- **Git：** 仅 2 次提交（初始文档 + Phase 2→3 过渡）。无实现历史可用来推断编码约定。

## 分层与边界规则

4 层本地优先架构（`docs/08`）：表现层（Windows APP）→ 业务逻辑层 → 数据层（本地）→ 云扩展层（按需）。

- **严格规则：** 上层只能调用直接下层（如 Service 不得跳过 Domain 直接调用 Repository）。同层横向调用允许。所有依赖经构造函数注入。
- **客户端不得：** 直接访问数据库；包含八字分析逻辑；直接调用 LLM——全部委托给核心服务。
- **HTTP = REST CRUD / 同步；WebSocket = 实时流式 / 推送。** 二者职责分离——不可混用。
- **能本地做的，就不上云。** 排盘、档案存储、基础分析、对话渲染均在本地。云端仅用于 AI 推理（LLM）与真太阳时校准；上传最小化、标记为临时、云端不持久化。

## API 与 WebSocket 约定（关键承载项）

这些是最易被违反的契约细节——全部已冻结。

**HTTP**（`docs/13`）：base `http://localhost:{port}/api/v1`；`Content-Type: application/json`；字段命名 `snake_case`；时间戳 ISO-8601 UTC；ID 为前缀+随机串（`u_xxx`、`chart_xxx`、`rpt_xxx`、`task_xxx`）。
- 标准响应体：`{ "code": 0, "message": "...", "data": {}, "request_id": "req_..." }`。**按 `code` 分支判断，绝不按 `message` 字符串。**
- 错误 `code` 区间：`0` 成功；`10000-19999` 系统；`20000-29999` 鉴权；`30000-39999` 校验；`40000-49999` 资源；`50000-59999` 业务；`60000-69999` 限流。
- 鉴权：`Authorization: Bearer <jwt>`（HS256）；仅 3 个免鉴权端点（health、register、login）。
- 异步长任务：POST → `202` + `task_id`；轮询 `GET /api/v1/tasks/{task_id}` 或经 WS 订阅。
- 版本：URL 路径版本（`/api/v1/`）；向后兼容的新增不升版本；破坏性变更须 `/api/v2/` + ≥6 个月弃用期。

**WebSocket**（`docs/14`）：**单连接**——仅一条路径：`ws://127.0.0.1:{port}/ws/v1/session?token=<jwt>`。其他路径（`/ws/v1/chat`、`/ws/v1/task/{id}`）属未来扩展，**V1 不得使用**。
- 信封：`{ "type", "message_id": "msg_...", "sequence"?, "timestamp", "data" }`。共 17 种消息类型。
- 心跳：客户端每 30s 发 `heartbeat.ping`；服务端 90s 空闲后断开；重连用指数退避 1s→30s、最多 10 次、±500ms 抖动；经 `task.resume` 携 `last_sequence` 恢复。
- 安全：每用户 ≤3 活跃连接；上行 ≤10 条/秒；单消息 ≤1MB；日志中对出生信息/日志内容/token 做脱敏。

**标准化字段名（已解决的 P0 漂移——不得回退）：**
- Task 对象使用字段 **`task_type`**（不是 `type`）。消息路由字段仍为 `type`。
- `POST /reports` 返回 `status: "pending"` + `report_id: null` + `task_id`（不是 `completed`）。

## 数据信任层级（ADR-012）

**事实 > 推测 > 待验证。** 用户确认的事实优先于任何 AI 推测；AI 推测不得作为事实呈现；每个数据点须按层级标注。

6 步八字分析流程（`docs/07`，ADR-009）为此服务：出生信息采集 → 排盘（确定性）→ AI 初步分析（标注"推测"）→ AI 动态追问（1–2 个问题）→ 用户反馈验证（"是"则推测→事实）→ 最终报告（8 节 + 可信度标注）。

**记忆架构**（`docs/19`，3 层）：工作记忆（最近约 20 轮，瞬时）→ 摘要记忆（压缩后持久化）→ 核心记忆（档案 + 已确认事实，版本化，绝不覆盖）。上下文注入顺序：系统提示 → 核心 → 摘要 → 工作 → 当前消息（在 70% 窗口上限内）。**关键：** 未确认的 AI 摘要绝不可写入核心记忆的事实层。70% 水位线与 20 轮保留数为可调参数，非固定规格。

## Agent 治理红线

源自 `docs/12` 与 `docs/17`。最易被违反的：

- **先读后写 / 不假设 / 最小修改 / 文档即契约**——任何任务前先读相关文档；绝不自行填补【待确认】内容（等待产品负责人）；关键信息不得仅存于对话中。
- **绝不自动提交**——必须先展示 diff 并等待确认。提交格式：`<type>: <desc>` + 修改文件/关联 ADR/修改理由 + Co-Authored-By。
- **绝不在 AI 沙箱内调用外部 LLM API**（SiliconFlow/DeepSeek 等须在沙箱外运行）。不触外部 API 的单元测试可在沙箱内运行；涉及外部服务的集成测试须在沙箱外运行。
- **Audit Agent 与 Developer Agent 在同一任务中不得为同一 AI**（禁止自审）。
- **每个任务须向 `docs/11_AI协作日志.md` 追加记录**——Agent ID 格式 `Agent-YYYYMMDD-NN`；最少字段：任务信息、修改清单（文件/操作/行区间/原因）、关键决策（标注 待确认/已确认/建议）、问题、引用 ADR、遗留。紧急问题在日志顶部加 **【紧急】** 前缀。
- **冲突优先级（高→低）：** Approved ADR > 架构文档 > API 契约 > 实现建议 > AI 普通建议。
- **按角色的写权限：** 仅 Architect 可写 `docs/08`/`13`/`14`；仅 Developer 可写 `src/` 与 `tests/`；仅 Audit 可写审计报告；**Approved ADR 对所有 AI 角色只读。**

## 平台与角色设定

- **V1 优先 Windows 桌面端**（ADR-015）：V1 桌面 → V2 Web → V3 移动。技术栈：**PySide6 外壳 + Vue（渐进式，经 QWebEngineView + Qt WebChannel）+ 本地 Python FastAPI 核心服务 + SQLite（V1 → PostgreSQL V2）**。V1 不要提出 Web/移动原生方案。
- **知命角色设定**（`docs/05`）：中等正式、儒雅但不古奥；可引经典如"《滴天髓》有云……"再以平实语言解释。**慎用绝对判断词**（"一定/绝对/必然"）；多用"倾向/可能/参考"。适当留白；绝不为用户做重大人生决定，也不断言具体未来事件。
- **领域范围：** 核心 = 八字排盘与分析（四柱/五行/十神/大运/流年）。**范围外：紫微斗数、西方占星、塔罗。**
